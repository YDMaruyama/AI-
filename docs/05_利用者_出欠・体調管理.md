# 05. 利用者の出欠・体調管理

## 1. 概要

### 1.1 この機能が解決する課題

就労継続支援事業所（A型/B型）において、利用者の出欠管理は事業運営の根幹をなす業務である。現状では以下の課題が存在する：

- **記録の散在**: 出欠は紙の出席簿やExcelで管理されており、リアルタイムに状況を把握できない。社長が「今日は何人来ているか」を知るには現場に確認するしかない
- **体調変化の埋もれ**: スタッフが利用者の体調の変化に気づいても、口頭で共有されるのみで記録に残らない。後から振り返った時に「いつ頃から調子が悪かったか」を追跡できない
- **欠席パターンの見逃し**: 利用者が徐々に出席率を落としていても、日々の変化は小さいため気づきにくい。気づいた時には長期欠席に至っていることがある
- **行政報告の手間**: 毎月の実績報告書には出席日数・欠席日数の集計が必要であり、手作業で集計するとミスが発生しやすい
- **担当者不在時の情報断絶**: 担当スタッフが休みの日に利用者の状況を把握する手段がない

### 1.2 提供する価値

- LINEで「佐藤さん欠席」と送るだけで出欠記録が完了する。紙の出席簿への記入が不要になる
- 「佐藤さん午後から調子悪い」の一言で体調変化が記録され、担当管理者に自動通知される
- AIが出席率の推移を自動分析し、低下傾向にある利用者を早期に検知して管理者にアラートを出す
- 社長が「今日の出欠状況」と送るだけで、全利用者の出席状況を即座に把握できる
- 月末に出席率サマリーが自動生成され、行政報告用の集計が瞬時に完了する
- 体調記録・出席率・日報の情報を横断的に統合し、「佐藤さん最近どう？」の一言で利用者の全体像を把握できる

### 1.3 対象となる「利用者」の定義

本ドキュメントにおける「利用者」とは、事業所の就労継続支援サービスを利用する障がい者を指す。事業所のスタッフ（支援員・管理者・社長）とは明確に区別される。利用者はLINEアカウントを持たない（AI秘書を直接操作しない）前提であり、スタッフがLINEを通じて利用者の出欠・体調を記録する。

### 1.4 行政報告との関係

就労継続支援事業所は毎月、市区町村に対して利用者ごとの出席実績を報告する義務がある。この報告書には以下のデータが含まれる：

| 報告項目 | 本機能が提供するデータ |
|---------|---------------------|
| 利用者ごとの出席日数 | attendance_records から自動集計 |
| 利用者ごとの欠席日数 | attendance_records から自動集計 |
| 月間営業日数 | business_calendar から取得 |
| 出席率 | 出席日数 / 営業日数 で自動計算 |
| 欠席理由の内訳 | absence_reason の分類別集計 |

これらのデータを正確かつ自動的に集計できることは、事務作業の大幅な削減につながる。

## 2. ユーザーストーリー

### 2.1 スタッフ：朝の出欠確認

```
スタッフとして、
朝の受付時に「今日の出欠」とLINEで送り、
本日利用予定の利用者一覧と、既に欠席連絡が入っている人を確認して、
受付業務をスムーズに進めたい。
```

**受入条件:**
- 当日の利用予定者の一覧が表示される（A型/B型の区分付き）
- 既に欠席連絡を受けた利用者には「欠席」マークが付いている
- 未連絡の利用者が何名いるかが分かる
- 開所時刻を過ぎて未連絡の利用者がいれば注意喚起される

### 2.2 スタッフ：欠席登録

```
スタッフとして、
利用者から欠席の電話連絡を受けた時に、
「鈴木さん体調不良で欠席」とLINEで送るだけで欠席が記録され、
理由も自動的に分類してほしい。
```

**受入条件:**
- 「鈴木さん欠席」だけでも記録できる（理由は「未記入」として保存）
- 「鈴木さん体調不良で欠席」なら理由が「体調不良」に自動分類される
- 記録完了の確認メッセージが返る
- 同じ利用者の当日分が既に登録済みの場合はエラーではなく上書き確認が出る

### 2.3 スタッフ：体調変化の記録

```
スタッフとして、
利用者の体調の変化に気づいた時に、
「佐藤さん午後から調子悪そう」とLINEで送れば体調メモが記録され、
担当管理者にも自動で共有されてほしい。
```

**受入条件:**
- 体調に関するキーワード（「調子悪い」「しんどそう」「元気ない」等）を検知して体調記録として処理する
- 当日の出欠レコードに health_note として追記される
- 担当管理者への通知を提案する（自動通知 or 確認後通知を設定で選択可能）
- 過去に同様の体調メモがある場合はその旨を表示する

### 2.4 管理者：担当利用者の出席率・体調傾向の把握

```
管理者として、
自分が担当する利用者の出席率と体調の傾向を定期的に確認し、
支援計画の見直しが必要な利用者を早期に把握したい。
```

**受入条件:**
- 「担当の出席率」で担当利用者全員の今月の出席率一覧が見られる
- 出席率が低下傾向にある利用者にはアラートマークが付く
- 体調メモが直近1週間に2回以上記録された利用者が強調表示される
- 各利用者の名前をタップすると詳細が見られる（将来のリッチメニュー対応時）

### 2.5 社長：全体の出欠状況確認

```
社長として、
「今日の出欠状況」と送るだけで、
本日の全利用者の出欠状況と出席率を把握し、
気になる利用者がいれば即座に詳細を確認したい。
```

**受入条件:**
- 本日の出席人数/予定人数/出席率が即座に表示される
- A型/B型の区分別の出席状況が表示される
- 欠席者の一覧と欠席理由が表示される
- 体調不良での欠席が多い場合はAIが注意喚起する

### 2.6 社長：個人の傾向確認

```
社長として、
「佐藤さん最近どう？」と送ったら、
出席率の推移・体調記録・日報での言及を統合した情報で、
その利用者の状態を総合的に把握したい。
```

**受入条件:**
- 直近1ヶ月の出席率と推移（先月比）が表示される
- 体調メモの履歴が新しい順に表示される
- 日報内でその利用者に言及されている内容が抽出される
- 支援計画の見直し時期が近い場合はリマインドが表示される
- AIによる総合所見（「先月より出席率が5%低下。体調メモが3件あり注視が必要」等）が付く

### 2.7 社長：月次出席率レポート

```
社長として、
月末に「今月の出席率レポート」と送ったら、
全利用者の出席率・欠席理由内訳・前月比較が一覧でき、
行政報告用のデータとして活用したい。
```

**受入条件:**
- 全利用者の月間出席率が一覧表示される
- A型/B型別の平均出席率が表示される
- 欠席理由別の内訳（体調不良/通院/家庭の事情/その他）が集計される
- 前月との比較が表示される
- 目標出席率（事業所全体・個人別）との乖離が分かる

## 4. 機能要件

### 4.1 必須機能（MVP）

| # | 機能名 | 説明 | 優先度 |
|---|--------|------|--------|
| F-01 | 出欠記録 | 自然言語で利用者の出欠（出席/欠席/遅刻/早退/半日）を記録する | P0 |
| F-02 | 欠席理由分類 | 欠席理由を自動分類する（体調不良/通院/家庭の事情/その他/未記入） | P0 |
| F-03 | 体調メモ記録 | 利用者の体調変化をテキストで記録し、出欠レコードに紐づける | P0 |
| F-04 | 出席率計算 | 月間出席率を自動計算する（半日=0.5カウント） | P0 |
| F-05 | 本日の出欠一覧 | 当日の予定利用者と現在の出欠状況を一覧表示する | P0 |
| F-06 | 利用者名マッチング | 自然言語入力から利用者名を特定する（ふりがな・部分一致・曖昧検索対応） | P0 |
| F-07 | 月間出席率一覧 | 全利用者の月間出席率を一覧表示する（A型/B型別） | P0 |
| F-08 | 個人傾向表示 | 特定利用者の出席率推移・体調メモ・日報言及を統合表示する | P1 |
| F-09 | 出欠記録の修正 | 登録済みの出欠を変更する（変更履歴を保存） | P1 |
| F-10 | 営業日カレンダー | 祝日・事業所休業日を管理し、出席率計算に反映する | P1 |

```sql
    target_attendance_rate DECIMAL(5,2) DEFAULT 80.00, -- 目標出席率（%）
```

### 5.6 ビュー: 月間出席率サマリー

```sql
CREATE OR REPLACE VIEW v_monthly_attendance_summary AS
SELECT
    c.id AS client_id,
    c.name AS client_name,
    c.client_type,
    c.assigned_manager_id,
    c.target_attendance_rate,
    DATE_TRUNC('month', ar.date) AS month,

    -- 営業日数（その利用者の予定曜日で絞り込み）
    (
        SELECT COUNT(*)
        FROM business_calendar bc
        WHERE bc.day_type IN ('business', 'special')
          AND DATE_TRUNC('month', bc.date) = DATE_TRUNC('month', ar.date)
          AND (
              -- 予定曜日のチェック（0=日, 1=月, ..., 5=金, 6=土）
              (EXTRACT(DOW FROM bc.date) = 1 AND c.scheduled_days & 1 > 0) OR   -- 月曜
              (EXTRACT(DOW FROM bc.date) = 2 AND c.scheduled_days & 2 > 0) OR   -- 火曜
              (EXTRACT(DOW FROM bc.date) = 3 AND c.scheduled_days & 4 > 0) OR   -- 水曜
              (EXTRACT(DOW FROM bc.date) = 4 AND c.scheduled_days & 8 > 0) OR   -- 木曜
              (EXTRACT(DOW FROM bc.date) = 5 AND c.scheduled_days & 16 > 0)     -- 金曜
          )
          -- 利用開始日〜終了日の範囲内に限定
          AND bc.date >= c.start_date
          AND (c.end_date IS NULL OR bc.date <= c.end_date)
    ) AS business_days,

    -- 出席換算合計
    COALESCE(SUM(ar.attendance_value), 0) AS total_attendance_value,

    -- 出席率
    CASE
        WHEN (
            SELECT COUNT(*)
            FROM business_calendar bc
            WHERE bc.day_type IN ('business', 'special')
              AND DATE_TRUNC('month', bc.date) = DATE_TRUNC('month', ar.date)
              AND (
                  (EXTRACT(DOW FROM bc.date) = 1 AND c.scheduled_days & 1 > 0) OR
                  (EXTRACT(DOW FROM bc.date) = 2 AND c.scheduled_days & 2 > 0) OR
                  (EXTRACT(DOW FROM bc.date) = 3 AND c.scheduled_days & 4 > 0) OR
                  (EXTRACT(DOW FROM bc.date) = 4 AND c.scheduled_days & 8 > 0) OR
                  (EXTRACT(DOW FROM bc.date) = 5 AND c.scheduled_days & 16 > 0)
              )
              AND bc.date >= c.start_date
              AND (c.end_date IS NULL OR bc.date <= c.end_date)
        ) = 0 THEN 0
        ELSE ROUND(
            COALESCE(SUM(ar.attendance_value), 0) * 100.0 /
            (
                SELECT COUNT(*)
                FROM business_calendar bc
                WHERE bc.day_type IN ('business', 'special')
                  AND DATE_TRUNC('month', bc.date) = DATE_TRUNC('month', ar.date)
                  AND (
                      (EXTRACT(DOW FROM bc.date) = 1 AND c.scheduled_days & 1 > 0) OR
                      (EXTRACT(DOW FROM bc.date) = 2 AND c.scheduled_days & 2 > 0) OR
                      (EXTRACT(DOW FROM bc.date) = 3 AND c.scheduled_days & 4 > 0) OR
                      (EXTRACT(DOW FROM bc.date) = 4 AND c.scheduled_days & 8 > 0) OR
                      (EXTRACT(DOW FROM bc.date) = 5 AND c.scheduled_days & 16 > 0)
                  )
                  AND bc.date >= c.start_date
                  AND (c.end_date IS NULL OR bc.date <= c.end_date)
            ),
            1
        )
    END AS attendance_rate,

    -- 欠席理由別集計
    COUNT(*) FILTER (WHERE ar.absence_reason = 'health') AS absences_health,
    COUNT(*) FILTER (WHERE ar.absence_reason = 'hospital') AS absences_hospital,
    COUNT(*) FILTER (WHERE ar.absence_reason = 'family') AS absences_family,
    COUNT(*) FILTER (WHERE ar.absence_reason = 'mental') AS absences_mental,
    COUNT(*) FILTER (WHERE ar.absence_reason = 'transport') AS absences_transport,
    COUNT(*) FILTER (WHERE ar.absence_reason = 'no_contact') AS absences_no_contact,
    COUNT(*) FILTER (WHERE ar.absence_reason = 'other') AS absences_other,

    -- 体調メモ件数
    COUNT(*) FILTER (WHERE ar.health_note IS NOT NULL) AS health_note_count

FROM clients c
LEFT JOIN attendance_records ar ON ar.client_id = c.id
WHERE c.status = 'active'
GROUP BY c.id, c.name, c.client_type, c.assigned_manager_id,
         c.target_attendance_rate, c.scheduled_days, c.start_date, c.end_date,
         DATE_TRUNC('month', ar.date);

COMMENT ON VIEW v_monthly_attendance_summary IS '月間出席率サマリービュー。利用者ごとの月間出席率・欠席理由内訳を集計する。';
```

### 6.1 朝の出欠確認処理

```
┌──────────────────────────────────────────────────────────┐
│ リクエスト: 「今日の出欠」                                  │
└───────────────────────┬──────────────────────────────────┘
                        │
                        ▼
┌──────────────────────────────────────────────────────────┐
│ 1. 当日が営業日かチェック                                   │

│    WHERE date = CURRENT_DATE                              │

└───────────────────────┬──────────────────────────────────┘
                        │ 営業日
                        ▼
┌──────────────────────────────────────────────────────────┐

│    SELECT * FROM clients                                  │

                        │
                        ▼
```

**当日の曜日ビット変換ロジック:**

```sql
-- PostgreSQLのEXTRACT(DOW)は日=0, 月=1, ..., 金=5, 土=6
-- ビットフラグは 月=1, 火=2, 水=4, 木=8, 金=16
-- 変換: DOW 1→bit 1, DOW 2→bit 2, DOW 3→bit 4, DOW 4→bit 8, DOW 5→bit 16

SELECT POWER(2, EXTRACT(DOW FROM CURRENT_DATE) - 1)::INTEGER AS day_bit;
-- 月曜(DOW=1) → 2^0 = 1
-- 火曜(DOW=2) → 2^1 = 2
-- 水曜(DOW=3) → 2^2 = 4
-- 木曜(DOW=4) → 2^3 = 8
-- 金曜(DOW=5) → 2^4 = 16
```

### 6.3 出席率計算ロジック

```
■ 基本計算式

  月間出席率(%) = (出席換算値の合計 / 当月の対象営業日数) * 100

■ 出席換算値

  | ステータス     | attendance_value |
  |---------------|-----------------|
  | present（出席）| 1.00            |
  | absent（欠席） | 0.00            |
  | late（遅刻）   | 0.50            |
  | early_leave   | 0.50            |
  | half_day      | 0.50            |

■ 対象営業日数の算出条件
  1. business_calendar で day_type が 'business' または 'special' の日
  2. かつ、その利用者の scheduled_days（予定曜日）に該当する日
  3. かつ、利用者の start_date 以降の日
  4. かつ、利用者の end_date 以前の日（end_date が NULL なら制限なし）
  5. 月途中の入所・退所にも正しく対応する

■ 計算例

  利用者：佐藤次郎（B型、月〜金利用）
  対象月：2025年3月
  営業日数：23日（3/1〜3/31の平日、祝日除く）
  出席：18日（present × 18 = 18.00）
  遅刻：1日（late × 1 = 0.50）
  欠席：4日（absent × 4 = 0.00）

  出席換算合計 = 18.00 + 0.50 + 0.00 = 18.50
  出席率 = 18.50 / 23 * 100 = 80.4%
```

**出席率計算のSQL:**

```sql
-- 特定利用者の特定月の出席率を計算
WITH target_month AS (
    SELECT
        DATE_TRUNC('month', :target_date::DATE) AS month_start,
        (DATE_TRUNC('month', :target_date::DATE) + INTERVAL '1 month' - INTERVAL '1 day')::DATE AS month_end
),
business_days AS (
    SELECT COUNT(*) AS total_days
    FROM business_calendar bc, target_month tm
    WHERE bc.date BETWEEN tm.month_start AND tm.month_end
      AND bc.day_type IN ('business', 'special')
      AND (
          (EXTRACT(DOW FROM bc.date) = 1 AND (SELECT scheduled_days FROM clients WHERE id = :client_id) & 1 > 0) OR
          (EXTRACT(DOW FROM bc.date) = 2 AND (SELECT scheduled_days FROM clients WHERE id = :client_id) & 2 > 0) OR
          (EXTRACT(DOW FROM bc.date) = 3 AND (SELECT scheduled_days FROM clients WHERE id = :client_id) & 4 > 0) OR
          (EXTRACT(DOW FROM bc.date) = 4 AND (SELECT scheduled_days FROM clients WHERE id = :client_id) & 8 > 0) OR
          (EXTRACT(DOW FROM bc.date) = 5 AND (SELECT scheduled_days FROM clients WHERE id = :client_id) & 16 > 0)
      )
      AND bc.date >= (SELECT start_date FROM clients WHERE id = :client_id)
      AND (
          (SELECT end_date FROM clients WHERE id = :client_id) IS NULL
          OR bc.date <= (SELECT end_date FROM clients WHERE id = :client_id)
      )
),
attendance_sum AS (
    SELECT COALESCE(SUM(attendance_value), 0) AS total_value
    FROM attendance_records ar, target_month tm
    WHERE ar.client_id = :client_id
      AND ar.date BETWEEN tm.month_start AND tm.month_end
)
SELECT
    bd.total_days AS business_days,
    ats.total_value AS attendance_value,
    CASE
        WHEN bd.total_days = 0 THEN 0
        ELSE ROUND(ats.total_value * 100.0 / bd.total_days, 1)
    END AS attendance_rate
FROM business_days bd, attendance_sum ats;
```

### 6.4 連続欠席日数の計算

```sql
-- 特定利用者の直近の連続欠席日数を計算
WITH ordered_attendance AS (
    SELECT
        ar.date,
        ar.status,
        ROW_NUMBER() OVER (ORDER BY ar.date DESC) AS rn
    FROM attendance_records ar
    JOIN business_calendar bc ON bc.date = ar.date
    WHERE ar.client_id = :client_id
      AND bc.day_type IN ('business', 'special')
    ORDER BY ar.date DESC
)
SELECT COUNT(*) AS consecutive_absent_days
FROM ordered_attendance
WHERE status = 'absent'
  AND rn <= (
      SELECT MIN(rn) - 1
      FROM ordered_attendance
      WHERE status != 'absent'
      UNION ALL
      SELECT MAX(rn) FROM ordered_attendance
      LIMIT 1
  );
```

## 8. エッジケース・エラー処理

### 8.1 利用者名が似ている場合の名前解決

**問題:** 「佐藤さん」に対して佐藤次郎・佐藤美咲の2名が該当する場合

**解決方針:**
1. 完全一致が1名のみ → そのまま処理
2. 完全一致が複数 → 選択肢を提示（番号選択）
3. 部分一致のみ → 候補を提示（最大5名）
4. 該当なし → トリグラム類似度で候補を提示 → それでもなければエラー

**名前解決の優先度:**

```
1. name の完全一致（「佐藤次郎」→ 佐藤次郎）
2. name の前方一致（「佐藤」→ 佐藤次郎, 佐藤美咲）
3. furigana の部分一致（「さとう」→ 佐藤次郎, 佐藤美咲）
4. name のトリグラム類似度 > 0.3（「佐等」→ 佐藤次郎 [類似度0.5]）
5. 最近の記録者との関連で推定（田中スタッフが最後に記録した佐藤さんは佐藤次郎）
```

**コンテキストによる推定（将来実装）:**
- 同じスタッフが直前に操作した利用者を優先する
- 担当管理者が入力した場合、担当利用者を優先する

### 8.2 途中入所・退所した利用者の出席率計算

**問題:** 月途中（例：3/15）に入所した利用者の3月の出席率をどう計算するか

**解決方針:**
- 分母（営業日数）は `start_date` 以降の営業日のみをカウントする
- 分子（出席日数）は実際の出席レコードの合計
- 3/15入所の場合：3/15〜3/31の営業日数が分母になる

**SQLの対応箇所:**

```sql
-- business_calendar のクエリに start_date / end_date の条件を含める
AND bc.date >= c.start_date
AND (c.end_date IS NULL OR bc.date <= c.end_date)
```

**表示時の注意:**
- 月途中入所の利用者には「※3/15入所（対象: 12日間）」のような注記を付ける
- 退所後は出席率のレポート対象から除外するが、過去データは参照可能

### 8.3 祝日・事業所休業日の扱い

**営業日カレンダーの初期設定:**
1. 年度初めに、土日・祝日を自動登録する（祝日マスタまたは外部API利用）
2. 事業所独自の休業日（お盆・年末年始・創立記念日等）を管理者が手動追加する
3. 特別営業日（土曜開所イベント等）を管理者が手動追加する

**カレンダー登録の会話フロー:**

```
管理者「8/13〜8/15を休業日にして」

AI「営業日カレンダーを更新しました。

  ■ 休業日に設定：
  ・8/13（火）→ 事業所休業日
  ・8/14（水）→ 事業所休業日
  ・8/15（木）→ 事業所休業日

  ※ 8月の営業日数は 20日 → 17日 に変更されます。
  ※ 出席率の計算に自動反映されます。」
```

### 8.5 出欠の締め処理（日次バッチ）

毎日の営業終了後（例：17:00）に以下の日次バッチ処理を実行する。

```
1. 当日の予定利用者のうち、出欠レコードが存在しない利用者を検出
2. 検出された利用者について：
   a. 当日出席していた可能性が高い場合（開所時間中に他の記録がある等）
      → 「出席」として自動登録し、スタッフに確認通知を送信
   b. 完全に記録がない場合
      → 「無断欠席（no_contact）」として仮登録し、
        担当管理者 + 社長に通知（N-07）
3. 翌営業日の予定利用者一覧を生成し、翌朝の出欠確認に備える
```

### 8.6 利用者のステータス変更時の扱い

| 変更 | 出欠記録への影響 |
|------|----------------|
| active → inactive | 翌日以降の予定利用者一覧から除外。過去の出欠記録は保持 |
| inactive → active | 翌日から予定利用者一覧に復帰。復帰月の出席率は復帰日以降で計算 |
| active → graduated | end_date を設定。過去データは参照可能。月末レポートからは除外 |
| active → terminated | graduated と同じ処理 |

## 10. 将来の拡張案

### 10.1 QRコード打刻連携

利用者が事業所に到着した際にQRコードをスキャンして自動的に出席を記録する仕組み。

**構想:**
- 事業所の入口にQRコードリーダー（タブレット）を設置
- 利用者ごとのQRコード付きカード（またはスマートフォン画面）を発行
- スキャン時に attendance_records を自動で `present` として登録
- 開所時刻から一定時間遅れた場合は自動的に `late` として記録
- スタッフのLINE操作なしで出欠が完了する

**技術的検討:**
- QRコードにはクライアントIDをエンコード
- タブレットアプリ → Supabase API → attendance_records に直接書き込み
- スキャン音やメッセージで利用者に到着確認のフィードバック
- オフライン時はローカルに保存し、接続回復後に同期

### 10.2 保護者への自動連絡

利用者の出欠状況や体調変化を保護者（家族・グループホーム等）に自動で共有する仕組み。

**構想:**
- clients テーブルの metadata に保護者の連絡先（LINE/メール/電話）を登録
- 以下のイベントで保護者に自動連絡：
  - 利用者が体調不良で早退する場合
  - 緊急の体調変化が報告された場合
  - 月末の出席率サマリー（保護者向けの簡易版）
- 保護者向けのメッセージはAIが平易な表現に変換する
- 送信前に担当管理者の承認を経る（設定で自動送信も可能に）

### 10.3 体調パターンのAI分析

蓄積された出欠・体調データをAIが分析し、利用者の変調を早期に検知する。

**分析内容:**
- **周期的パターン**: 毎週月曜の欠席率が高い、月初に体調を崩しやすい等
- **低下トレンド**: 直近3ヶ月の出席率が右肩下がりの利用者を検出
- **複合指標**: 出席率低下 + 体調メモ増加 + 日報での作業効率低下 の組み合わせで総合スコアを算出
- **予測モデル**: 過去のデータから「来月の出席率が80%を下回る可能性が高い」利用者を予測

**レポート例:**

```
📊 AI分析レポート（月次）

■ 注視推奨の利用者

1. 佐藤次郎さん（B型）- 総合スコア: 要注意
   ・出席率: 3ヶ月連続低下（90% → 87% → 82%）
   ・体調メモ: 月3件（先月の2倍）
   ・日報言及: 作業効率低下の報告あり
   ・パターン: 水曜・木曜の欠席が多い
   → 支援計画の見直しを推奨

2. 山田花子さん（B型）- 総合スコア: 注意
   ・出席率: 先月比-5%
   ・パターン: 月曜の欠席率が60%（他の曜日は90%以上）
   → 週末の過ごし方に関するヒアリングを推奨
```

### 10.4 季節性の欠席傾向分析

年間を通じた欠席パターンを分析し、事前対策を提案する。

**分析内容:**
- **季節変動**: 梅雨時期・猛暑日・冬季に欠席率が上がる傾向の検出
- **年度切替**: 4月の環境変化による不安定さの検出
- **イベント影響**: GW・お盆・年末年始前後の欠席パターン
- **気象データ連携**: 気温・気圧・天候と出席率の相関分析（将来）

**活用例:**
- 「昨年の6月は精神的不調による欠席が30%増加しています。今年も注意が必要です」
- 「気圧が急低下する日は、○○さんの体調不良欠席率が通常の2倍になる傾向があります」
- 年間カレンダーに「欠席率が上がりやすい時期」をマッピングし、事前にスタッフ体制を厚くする

### 10.5 行政報告書の自動生成

月次の出席実績データを行政報告のフォーマットに自動変換し、提出書類の作成を効率化する。

**構想:**
- 市区町村ごとに異なる報告フォーマットをテンプレートとして登録
- 出席率データからテンプレートに値を自動埋め込み
- CSV/Excel/PDF形式での出力に対応
- 提出前の確認画面で管理者がデータを最終チェック

### 10.6 利用者向けの出席フィードバック

利用者本人（または保護者を通じて）に出席状況のフィードバックを提供し、モチベーション向上につなげる。

**構想:**
- 月間の出席率を分かりやすいビジュアル（カレンダー形式、スタンプカード形式）で提供
- 「今月は○日出席しました！先月より○日多いです」のようなポジティブなメッセージ
- 皆勤賞・出席率向上賞などの仕組みとの連携
- 利用者の自己肯定感の向上に配慮した表現設計
