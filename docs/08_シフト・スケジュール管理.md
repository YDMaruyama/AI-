# 08. シフト・スケジュール管理

## 1. 概要

### 1.1 この機能が解決する課題

就労継続支援事業所（A型/B型）は少人数体制（スタッフ5〜8名程度）で運営されるため、一人が休むだけで現場の人員配置に大きな影響が出る。現状の課題は以下の通り。

- **シフトの全体把握が困難** -- 社長がスタッフ全員の出勤予定を即座に把握する手段がなく、LINEや口頭での確認に頼っている
- **休み申請のフローが曖昧** -- 「来週休みたい」と口頭で伝えたが記録に残らず、当日になって「聞いてない」が発生する
- **人手不足の事前検知ができない** -- 翌日の出勤スタッフ数が利用者数に対して不足していても、前日夜や当日朝まで気づけない
- **急な欠勤時の対応が属人的** -- 体調不良などの急な欠勤時、社長が手動で代替スタッフを探すが、誰が対応可能かすぐにわからない
- **半日出勤や祝日対応が複雑** -- 午前だけ出勤、祝日だが事業所は開所など、イレギュラーなシフトパターンの管理が手作業になっている

### 1.2 提供する価値

- LINEで「来週のシフト」と送るだけで、曜日別のスタッフ配置が一覧で見える
- スタッフがLINEで「来週の月曜休みたい」と送るだけで、AIが影響範囲を提示し、社長に正式な申請が届く
- 社長は申請に対して「承認」「却下」をワンタップで対応でき、結果がスタッフに自動通知される
- 翌日の出勤人数が利用者数に対して不足している場合、前日夕方に社長へ自動警告が届く
- 急な欠勤の連絡を受けたAIが即座に社長に通知し、出勤可能なスタッフの候補リストを提示する
- 毎週日曜に翌週のシフト一覧が全員に自動配信され、認識のズレが起きない

### 1.3 対象ユーザーとロール別の利用範囲

| ロール | できること |
|--------|-----------|
| 社長 | シフト全体の閲覧・編集、休み申請の承認/却下、人員警告の受信、シフト調整指示 |
| 管理者 | 自分のシフト確認、担当利用者のスケジュールとの突合、休み申請の提出 |
| スタッフ | 自分のシフト確認、休み申請の提出、承認結果の確認 |

### 1.4 前提条件

- 事業所の基本営業日は月〜金（土日祝は原則休所だが、事業所カレンダーで個別設定可能）
- 利用者の出席予定は「05_出欠管理」モジュールから取得する
- スタッフ情報は「02_ユーザー管理」モジュールの users テーブルから取得する
- 1日のシフトパターンは「全日」「午前のみ」「午後のみ」「休み」「祝日休所」の5種類
- 人員配置の基準比率は事業所設定で管理（デフォルト：スタッフ1人あたり利用者3名）

## 2. ユーザーストーリー

### 2.1 スタッフ：休み申請

```
スタッフとして、
「来週の月曜休みたい」とLINEで送ったら、
AIがその日の残りスタッフ数と利用者数を算出してくれて、
影響があるかないかを教えてくれた上で、
社長に正式な休み申請が送信されて、
結果（承認/却下）がLINEで通知されたい。
```

**受入条件：**
- 「来週の月曜休みたい」「4/7有給で休みます」「水曜と木曜休み希望」など、自然言語で日付を指定できる
- AIが対象日の出勤予定スタッフ数・利用者予定数を表示する
- 人員不足の場合は警告マーク付きで表示する
- 申請送信後、ステータスが「申請中」になる
- 社長が承認/却下した時点でスタッフに結果が通知される

### 2.2 管理者：自分のシフトと利用者スケジュールの突合確認

```
管理者として、
「来週のスケジュール」と送ったら、
自分のシフトと担当利用者の出席予定が
一緒に表示されて、
どの日に誰が来るのかすぐわかりたい。
```

**受入条件：**
- 管理者の出勤シフトと、担当利用者の出席予定が日別で並んで表示される
- 自分が休みの日に担当利用者が来所予定の場合、代替対応が必要な旨を表示する
- 「来週」「今週」「4月第1週」など柔軟な期間指定ができる

### 2.3 社長：シフト一覧確認

```
社長として、
「来週のシフト」と送ったら、
曜日×スタッフのマトリクス形式でシフト一覧が表示され、
人手不足の日が一目でわかりたい。
```

**受入条件：**
- 曜日（月〜金）×スタッフのマトリクスで表示される
- 各セルにシフト区分（全日/午前/午後/休み）が表示される
- 人員不足の日は警告アイコンと不足数が表示される
- 休み申請中（未承認）のセルは区別表示される

### 2.4 社長：休み申請の承認・却下

```
社長として、
スタッフからの休み申請通知を受け取ったら、
影響範囲を確認した上で、
「承認」「却下」をワンタップで選択して、
結果がスタッフに自動通知されたい。
```

**受入条件：**
- 申請通知に申請者名・日付・理由・影響（残りスタッフ数/利用者数）が含まれる
- 承認/却下ボタン（LINEリッチメニューまたはクイックリプライ）で操作できる
- 却下時は任意でコメントを追加できる
- 承認/却下後にスタッフへ自動通知される

### 2.5 社長：シフト調整

```
社長として、
「来週月曜は田中さんも出勤にして」と送ったら、
田中さんのシフトが更新されて、
田中さんに通知が届いてほしい。
```

**受入条件：**
- 自然言語でスタッフ名・日付・シフト区分を指定できる
- 変更前後の内容を確認表示し、社長の確定後に反映する
- 対象スタッフに変更通知が届く
- シフト変更の履歴が残る

### 2.6 スタッフ：急な欠勤連絡

```
スタッフとして、
「今日体調悪いので休みます」と送ったら、
即座に社長に通知が届いて、
代わりに出勤できるスタッフの候補が社長に提示されてほしい。
```

**受入条件：**
- 当日の欠勤連絡は即座に社長へ通知される
- 通知には欠勤者名・当日の残りスタッフ数・利用者数が含まれる
- 当日休みのスタッフのうち、出勤切替可能な候補がリスト表示される
- 社長が候補スタッフに出勤依頼を送信できる

### 2.7 社長：人手不足の自動警告受信

```
社長として、
翌日の出勤予定スタッフ数が利用者数に対して不足している場合、
前日の夕方に自動で警告通知を受け取り、
事前に対策を打ちたい。
```

**受入条件：**
- 毎日17:00に翌営業日の人員チェックが自動実行される
- スタッフ数/利用者数の比率が基準（1:3）以下の場合に警告通知が送信される
- 警告には不足数・出勤予定スタッフ名・休みのスタッフ名が含まれる
- 警告内から休みスタッフへの出勤依頼を送信できる

## 3. 会話フロー（詳細）

### 3.1 正常系：休み申請（スタッフ）

```
スタッフ「来週の月曜休みたいです」

AI「来週月曜（4/7）のお休み申請ですね。
  確認しますね...

  ------
  4/7（月）の状況：
  出勤予定：佐藤さん、鈴木さん、高橋さん（3名）
  あなたが休むと：2名
  利用者予定：8名
  スタッフ/利用者比：1:4.0
  ------
  [!] 基準（1:3）を下回ります。

  それでも申請しますか？」

スタッフ「はい、お願いします」

AI「休み申請を送信しました。
  ------
  申請内容：
  日付：4/7（月）
  種別：有給休暇
  状況：申請中（社長の承認待ち）
  ------
  承認されたらお知らせします！」

--- （社長への通知） ---

AI→社長「休み申請が届きました。
  ------
  申請者：山田さん
  日付：4/7（月）
  種別：有給休暇

  [!] 人員影響あり
  承認後の出勤：2名
  利用者予定：8名（比率 1:4.0）
  ------
  【承認】 【却下】」

社長「承認」

AI→社長「山田さんの4/7（月）の休みを承認しました。
  山田さんに通知します。」

AI→スタッフ「休み申請が承認されました！
  ------
  4/7（月）：お休み（有給）
  承認者：社長
  ------
  ゆっくり休んでくださいね。」
```

```
  │高橋  │全│全│全│休│全│
  │山田  │休│全│全│全│全│
  │田中  │全│全│全│全│午後│
  │伊藤  │全│休│全│全│全│
  ├──────┼──┼──┼──┼──┼──┤

  └──────┴──┴──┴──┴──┴──┘

  凡例：全=全日, 午前/午後=半日, 休=休み
  ※半日出勤は0.5人としてカウント
```

```
社長「OK」
```

### 3.4 正常系：急な欠勤対応

```

AI→鈴木「お大事にしてください。
  社長に連絡します。

  本日の欠勤を社長に通知しました。
  お休みに変更しておきますね。」

AI→社長「[緊急] 本日の欠勤連絡
  ------
  欠勤者：鈴木さん
  日付：本日（4/7 月）
  理由：体調不良

  本日の状況（変更後）：
  出勤：佐藤さん、高橋さん、田中さん（3名）
  利用者予定：8名
  スタッフ/利用者比：1:2.7
  ------
  基準はクリアしていますが、余裕は少ない状況です。

  本日休みのスタッフ：
  ・山田さん（有給）
  ・伊藤さん（公休）

  出勤依頼を送りますか？
  【山田さんに依頼】【伊藤さんに依頼】【対応不要】」

社長「対応不要」

AI「了解しました。本日は3名体制で対応します。」
```

### 3.5 異常系：全員が同じ日に休み申請

```
（既に4/7に佐藤・鈴木・高橋が休み申請済みの状態で）

スタッフ（山田）「来週月曜休みたい」

AI「来週月曜（4/7）のお休み申請ですね。
  確認しますね...

  ------
  [!!!] 重大な人員不足が発生します

  4/7（月）の状況：
  既に休み申請中：佐藤さん、鈴木さん、高橋さん
  あなたが休むと出勤可能：田中さん、伊藤さん（2名）
  利用者予定：8名
  スタッフ/利用者比：1:4.0
  ------

  この日は既に3名が休み申請しており、
  運営に支障が出る可能性が高いです。

  それとも、この内容で社長に申請しますか？」

山田「別の日にします。火曜はどうですか？」

AI「4/8（火）の状況：
  出勤予定：5名（全員出勤）
  利用者予定：7名
  スタッフ/利用者比：1:1.4

  問題ありません！4/8（火）で申請しますか？」
```

## 4. 機能要件

### 4.1 必須機能（MVP）

#### F-SHIFT-001：シフト表示

| 項目 | 内容 |
|------|------|
| 説明 | 指定期間のスタッフシフトを表示する |
| トリガー | 「来週のシフト」「今週の出勤」「4/7のシフト」等の自然言語入力 |
| ロール | 全ロール（表示範囲はロールに応じて制限） |
| 表示形式 | 社長：マトリクス（全スタッフ×曜日）、管理者/スタッフ：自分のシフトのみ |
| 含む情報 | シフト区分、出勤人数、利用者予定数、スタッフ/利用者比率 |
| 優先度 | P0 |

#### F-SHIFT-002：休み申請

| 項目 | 内容 |
|------|------|
| 説明 | スタッフが休みを申請し、社長が承認/却下する |
| トリガー | 「来週月曜休みたい」「4/7有給で休みます」等 |
| ロール | スタッフ・管理者が申請、社長が承認/却下 |
| フロー | 申請→影響表示→確認→社長通知→承認/却下→スタッフ通知 |
| 休み種別 | 有給、無給（欠勤）、病欠、その他 |
| 優先度 | P0 |

#### F-SHIFT-003：人員チェック

| 項目 | 内容 |
|------|------|
| 説明 | 指定日の出勤スタッフ数と利用者予定数を比較し、人員不足を検知する |
| 基準比率 | スタッフ1人あたり利用者3名（事業所設定で変更可能） |
| 検知タイミング | 休み申請時、シフト変更時、毎日17:00の定時チェック |
| 警告レベル | 注意（基準ギリギリ）、警告（基準以下）、緊急（スタッフ2名以下） |
| 優先度 | P0 |

#### F-SHIFT-004：急な欠勤対応

| 項目 | 内容 |
|------|------|
| 説明 | 当日の欠勤連絡を処理し、社長に即時通知と代替候補を提示する |
| トリガー | 「今日休みます」「体調不良で出られません」等 |
| 処理 | シフト即時変更→社長通知→代替候補リスト表示→出勤依頼送信 |
| 優先度 | P0 |

#### F-SHIFT-005：休み申請の承認・却下

| 項目 | 内容 |
|------|------|
| 説明 | 社長が休み申請を承認または却下する |
| 操作方法 | LINEクイックリプライまたはリッチメニューのボタン |
| 却下時 | コメントを任意入力可能 |
| 後処理 | シフトテーブル更新、申請者への結果通知 |
| 優先度 | P0 |

## 5. データモデル

### 5.1 ER図（概要）

```
users (02_ユーザー管理)
  │
  ├──< shifts（シフト）
  │     日ごとのシフト情報
  │
  ├──< leave_requests（休み申請）
  │     休み申請と承認/却下の記録
  │
  └──< shift_change_logs（シフト変更履歴）
        シフト変更の監査ログ

business_calendar（事業所カレンダー）
  事業所の開所日・休所日・祝日の管理

staffing_config（人員配置設定）
  スタッフ/利用者比率の基準値等
```

### 5.2 テーブル定義

#### shifts テーブル

```sql
CREATE TABLE shifts (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    date DATE NOT NULL,
    shift_type VARCHAR(20) NOT NULL DEFAULT 'full'
        CHECK (shift_type IN ('full', 'morning', 'afternoon', 'off', 'holiday')),
    status VARCHAR(20) NOT NULL DEFAULT 'scheduled'
        CHECK (status IN ('scheduled', 'confirmed', 'changed', 'cancelled')),
    note TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    -- 同一ユーザー・同一日付のシフトはユニーク
    UNIQUE (user_id, date)
);

-- インデックス
CREATE INDEX idx_shifts_user_date ON shifts(user_id, date);
CREATE INDEX idx_shifts_date ON shifts(date);
CREATE INDEX idx_shifts_date_type ON shifts(date, shift_type);

-- shift_type の補足
-- full      : 全日出勤（例: 9:00〜17:00）
-- morning   : 午前出勤（例: 9:00〜12:00）
-- afternoon : 午後出勤（例: 13:00〜17:00）
```

```sql
WITH staff_count AS (
    SELECT
        s.date,
        SUM(CASE
            WHEN s.shift_type = 'full' THEN 1.0
            WHEN s.shift_type IN ('morning', 'afternoon') THEN 0.5
            ELSE 0
        END) AS effective_staff
    FROM shifts s
    JOIN users u ON s.user_id = u.id
    WHERE s.date = '2026-04-07'
        AND s.shift_type NOT IN ('off', 'holiday')
        AND s.status != 'cancelled'
        AND u.is_active = true
    GROUP BY s.date
),
user_count AS (
    -- 05_出欠管理の attendance_records から取得
    SELECT
        date,
        COUNT(*) AS expected_users
    FROM attendance_records
    WHERE date = '2026-04-07'
        AND status = 'expected'
    GROUP BY date
)
SELECT
    sc.date,
    sc.effective_staff,
    uc.expected_users,
    ROUND(uc.expected_users::numeric / NULLIF(sc.effective_staff, 0), 1) AS ratio,
    CASE
        WHEN sc.effective_staff <= 2 THEN 'critical'
        WHEN uc.expected_users::numeric / NULLIF(sc.effective_staff, 0) > 3 THEN 'warning'
        WHEN uc.expected_users::numeric / NULLIF(sc.effective_staff, 0) > 2.5 THEN 'caution'
        ELSE 'ok'
    END AS alert_level
FROM staff_count sc
LEFT JOIN user_count uc ON sc.date = uc.date;
```

```javascript
async function handleShiftView(userId, message) {
    // 1. ユーザー情報取得
    const user = await getUser(userId);

    // 2. 期間パース
    const period = parsePeriod(message); // { start: '2026-04-07', end: '2026-04-11' }

    // 3. データ取得
    const shifts = await getShifts(period.start, period.end);
    const pendingLeaves = await getPendingLeaves(period.start, period.end);
    const businessDays = await getBusinessDays(period.start, period.end);
    const expectedUsers = await getExpectedUsers(period.start, period.end);

    // 4. 人員チェック
    const staffingCheck = businessDays.map(day => {
        const dayShifts = shifts.filter(s => s.date === day.date);
        const effectiveStaff = dayShifts.reduce((sum, s) => {
            if (s.shift_type === 'full') return sum + 1;
            if (s.shift_type === 'morning' || s.shift_type === 'afternoon') return sum + 0.5;
            return sum;
        }, 0);
        const userCount = expectedUsers.find(u => u.date === day.date)?.count || 0;
        const ratio = userCount / (effectiveStaff || 1);

        return {
            date: day.date,
            effectiveStaff,
            userCount,
            ratio,
            alertLevel: getAlertLevel(effectiveStaff, ratio)
        };
    });

    // 5. ロール別レスポンス整形
    if (user.role === 'owner') {
        return formatMatrixView(shifts, pendingLeaves, staffingCheck);
    } else if (user.role === 'admin') {
        return formatAdminView(user, shifts, expectedUsers);
    } else {
        return formatStaffView(user, shifts);
    }
}

function getAlertLevel(effectiveStaff, ratio) {
    if (effectiveStaff <= 2) return 'critical';
    const threshold = getConfig('staff_user_ratio'); // default: 3
    if (ratio > threshold) return 'warning';
    if (ratio > threshold * 0.85) return 'caution';
    return 'ok';
}
```

### 6.3 人員チェックロジック

```
[人員チェック実行]
  │
  ├─ 対象日の出勤予定スタッフを取得
  │   └─ shifts WHERE date = 対象日 AND shift_type NOT IN ('off', 'holiday')
  │
  ├─ 実効スタッフ数を算出
  │   ├─ full = 1.0
  │   ├─ morning = 0.5
  │   └─ afternoon = 0.5
  │
  ├─ 対象日の利用者予定数を取得
  │   └─ attendance_records（05_出欠管理）WHERE date = 対象日 AND status = 'expected'
  │
  ├─ 比率を算出
  │   └─ ratio = 利用者予定数 / 実効スタッフ数
  │
  ├─ 警告レベル判定
  │   ├─ critical : 実効スタッフ数 <= min_staff_count（デフォルト2）
  │   ├─ warning  : ratio > staff_user_ratio（デフォルト3）
  │   ├─ caution  : ratio > staff_user_ratio * 0.85
  │   └─ ok       : 上記以外
  │
  └─ 結果返却
      ├─ effective_staff : 実効スタッフ数
      ├─ expected_users  : 利用者予定数
      ├─ ratio           : スタッフ/利用者比率
      ├─ alert_level     : 警告レベル
      ├─ staff_on_duty   : 出勤スタッフ名リスト
      └─ staff_off_duty  : 休みスタッフ名リスト
```

```javascript
async function checkStaffingImpact(date, excludeUserId = null) {
    // 出勤予定スタッフ取得
    let shifts = await db.query(`
        SELECT s.*, u.display_name
        FROM shifts s
        JOIN users u ON s.user_id = u.id
        WHERE s.date = $1
            AND s.shift_type NOT IN ('off', 'holiday')
            AND s.status != 'cancelled'
            AND u.is_active = true
    `, [date]);

    // 除外ユーザー（申請者自身）を除く
    if (excludeUserId) {
        shifts = shifts.filter(s => s.user_id !== excludeUserId);
    }

    // 実効スタッフ数
    const effectiveStaff = shifts.reduce((sum, s) => {
        if (s.shift_type === 'full') return sum + 1.0;
        if (s.shift_type === 'morning' || s.shift_type === 'afternoon') return sum + 0.5;
        return sum;
    }, 0);

    // 利用者予定数（05_出欠管理から）
    const userResult = await db.query(`
        SELECT COUNT(*) as count
        FROM attendance_records
        WHERE date = $1 AND status = 'expected'
    `, [date]);
    const expectedUsers = parseInt(userResult.rows[0].count);

    // 比率・警告レベル
    const ratio = expectedUsers / (effectiveStaff || 1);
    const minStaff = parseInt(await getConfig('min_staff_count'));
    const maxRatio = parseFloat(await getConfig('staff_user_ratio'));

    let alertLevel = 'ok';
    if (effectiveStaff <= minStaff) alertLevel = 'critical';
    else if (ratio > maxRatio) alertLevel = 'warning';
    else if (ratio > maxRatio * 0.85) alertLevel = 'caution';

    // 休みスタッフ取得
    const offStaff = await db.query(`
        SELECT u.display_name
        FROM shifts s
        JOIN users u ON s.user_id = u.id
        WHERE s.date = $1
            AND s.shift_type = 'off'
            AND u.is_active = true
    `, [date]);

    return {
        effectiveStaff,
        expectedUsers,
        ratio: Math.round(ratio * 10) / 10,
        alertLevel,
        staffOnDuty: shifts.map(s => s.display_name),
        staffOffDuty: offStaff.rows.map(s => s.display_name)
    };
}
```

### 8.1 祝日の扱い（事業所カレンダー）

```
ケース2：祝日だが事業所は開所（特別開所日）
→ business_calendar に 'special_open' を登録。
  シフトは通常通り管理。
  「4/29は祝日ですが、事業所は開所です。シフトを確認してください。」

ケース3：平日だが事業所は休所（特別休所日）
→ business_calendar に 'special_close' を登録。
  シフトは自動的に 'holiday' に設定。
  「12/29は特別休所日です。」

処理ロジック：
```

```javascript
async function isBusinessDay(date) {
    const dayOfWeek = new Date(date).getDay();

    // 事業所カレンダーを優先チェック
    const calendarEntry = await db.query(
        'SELECT day_type FROM business_calendar WHERE date = $1', [date]
    );

    if (calendarEntry.rows.length > 0) {
        const dayType = calendarEntry.rows[0].day_type;
        if (dayType === 'special_open') return true;
        if (dayType === 'special_close') return false;
        if (dayType === 'holiday') return false;
    }

    // 土日は休所
    if (dayOfWeek === 0 || dayOfWeek === 6) return false;

    // 国民の祝日チェック（外部ライブラリまたは祝日テーブル）
    const isNationalHoliday = await checkNationalHoliday(date);
    if (isNationalHoliday) return false;

    return true;
}
```

### 8.3 シフト変更の履歴管理

すべてのシフト変更は shift_change_logs テーブルに記録される。

```
記録される操作：
- シフトの新規作成（change_type: 'created'）
- シフト区分の変更（change_type: 'updated'）-- full→off, morning→full 等
- シフトのキャンセル（change_type: 'cancelled'）

- 変更前のshift_type、変更後のshift_type
- 変更前のstatus、変更後のstatus
- 変更者（changed_by）
- 変更理由（reason）
- タイムスタンプ
```

### 8.4 利用者数の変動（見学者等）

```
ケース1：見学者が来所予定

ケース2：利用者が急にキャンセル
```
