## 1. 概要

### 1.1 背景と課題

就労継続支援事業所（A型/B型）は、障害者総合支援法および関連する省令・通知に基づき、行政機関（都道府県・市区町村・国保連等）に対して多数の書類を定期的に提出する義務がある。これらの書類は提出期限が厳格に定められており、期限を超過すると以下のような深刻な影響が生じる。

- **行政指導・監査指摘**: 実地指導や監査において「書類提出の遅延」として指摘され、改善勧告の対象となる
- **報酬請求の遅延**: 実績報告書や請求書の提出遅延は、当月の報酬（訓練等給付費）の支払い遅延に直結する
- **加算の取消し**: 処遇改善加算等の各種加算について、実績報告の未提出は加算の取消し事由となる
- **指定取消しリスク**: 事業報告書の未提出が継続する場合、最悪のケースでは指定取消し処分の対象となりうる
- **信用の毀損**: 行政との信頼関係の悪化により、新規事業の申請・変更届出等で不利になる可能性がある

### 1.2 主な行政提出書類

就労継続支援事業所が提出すべき主要な書類は以下のとおりである。

| 書類名 | 提出頻度 | 期限 | 提出先 | 概要 |
|--------|---------|------|--------|------|
| 実績報告書（サービス提供実績記録票） | 毎月 | 翌月10日 | 国保連 | 利用者ごとのサービス提供実績を報告 |
| 工賃支払い報告 | 毎月 | 翌月15日 | 市区町村 | B型：利用者への工賃支払い実績を報告 |
| 利用者負担額の請求 | 毎月 | 翌月20日 | 市区町村 | 利用者負担額の代理請求 |
| 処遇改善加算の実績報告 | 年1回 | 7月末日 | 都道府県 | 処遇改善加算の前年度実績を報告 |
| 事業報告書 | 年1回 | 6月末日 | 都道府県 | 前年度の事業活動全体を報告 |
| 自己評価・第三者評価 | 年1回 | 年度末（3月末） | 都道府県 | サービスの質の評価結果を提出 |
| 処遇改善計画書 | 年1回 | 2月末日 | 都道府県 | 次年度の処遇改善計画を提出 |
| 変更届 | 随時 | 変更後10日以内 | 都道府県 | 事業所の重要事項変更時 |
| 事故報告書 | 随時 | 発生後速やかに | 市区町村 | 利用者の事故発生時 |

### 1.3 現状の運用と問題点

現在、多くの小規模事業所では以下のような方法で書類期限を管理している。

- **手帳・壁掛けカレンダー**: 社長や管理者が手書きで期限を記入。見落としやすく、担当者変更時に引き継ぎが困難
- **Excelの年間スケジュール表**: 年度初めに作成するが、日常の業務に追われて確認を怠りがち
- **記憶に頼った管理**: 「毎月10日が実績報告の期限」と覚えているが、繁忙期に失念する
- **行政からの督促で気付く**: 期限を過ぎてから行政の催促で初めて気付くケース

これらの問題を解決するため、AI秘書システムに「行政書類リマインド機能」を実装し、書類提出の期限管理を自動化・確実化する。

### 1.4 本機能の目的

- **提出漏れゼロ**: 全ての行政書類の期限を自動追跡し、段階的なリマインドで確実に提出を促す
- **業務負荷の軽減**: 社長・管理者が「次に何を出すべきか」を常に把握できる状態を作る
- **データ連携による準備促進**: 書類作成に必要なデータ（日報、出欠、工賃等）の入力状況を事前にチェックし、直前の慌ただしい作業を防ぐ
- **組織的な管理体制**: 担当者の明確化と進捗の可視化により、属人化を排除する

## 2. ユーザーストーリー

### 2.1 社長のユーザーストーリー

#### US-12-01: 月初の行政タスク一覧確認

```
役割:   社長（事業所の最終責任者）
目的:   今月提出すべき行政書類を月初に一覧で確認したい
理由:   月全体の行政対応スケジュールを把握し、漏れなく準備を進めるため
受入条件:
  - 毎月1日の朝（9:00）にAIから自動的に今月のタスク一覧が届く
  - 各書類の名前、期限日、担当者、現在のステータスが表示される
  - 前月に未提出の書類がある場合は警告表示される
  - 「了解」「詳細を見せて」等で対話を続けられる
```

**利用シーン**: 毎月1日の朝、社長がLINEを開くとAI秘書から通知が届いている。「4月の行政関連タスク」として、今月提出すべき書類が一覧表示されており、それぞれの期限日と担当者が明記されている。社長は一覧を見て「今月は処遇改善の実績報告もあるんだった」と確認し、管理者に声をかける。

#### US-12-02: 特定書類の期限確認

```
役割:   社長
目的:   特定の書類の期限をすぐに確認したい
理由:   行政や関係者から問い合わせがあった時に即座に回答するため
受入条件:
  - 「実績報告書いつまで？」のような自然言語で質問できる
  - 期限日、残り日数、対象期間が即座に回答される
  - 関連する未完了タスク（データ入力等）があれば合わせて表示
```

**利用シーン**: 行政から電話があり「今月の実績報告書はいつ届きますか？」と聞かれた社長が、すぐにAI秘書に「実績報告いつまでだっけ？」と聞く。「毎月10日が期限です。4月分の提出期限は5月10日（土）ですが、休日のため5月9日（金）が実質期限です。あと8日です」と即座に回答がある。

#### US-12-03: AIによる段階的自動リマインド

```
役割:   社長
目的:   期限が近づくにつれて段階的にリマインドを受けたい
理由:   日常業務に追われていても期限を忘れないため
受入条件:
  - 期限7日前: 通常のリマインド（担当者へ）
  - 期限3日前: 少し強めのリマインド（担当者＋社長へ）
  - 期限当日: 緊急リマインド（社長へ）
  - 期限超過: 毎日警告（社長へ、提出完了まで）
```

**利用シーン**: 期限7日前に担当の管理者にリマインドが届く。管理者が対応を始めるが、3日前になっても未提出のままなので社長にもリマインドが届く。社長が管理者に確認すると「データが揃っていなくて...」とのこと。社長はAI秘書に「何のデータが足りない？」と聞き、不足データを特定して対応する。

#### US-12-04: 完了報告と記録

```
役割:   社長
目的:   書類提出の完了を記録したい
理由:   提出済みの書類を管理し、リマインドを止めるため
受入条件:
  - 「実績報告書出した」のような自然言語で完了報告できる
  - ステータスが「提出済み」に更新される
  - 提出日時と提出者が自動記録される
  - 以降のリマインドが停止する
```

#### US-12-05: 年間スケジュールの確認

```
役割:   社長
目的:   年間を通じた主要な行政書類の提出スケジュールを確認したい
理由:   年度の見通しを立て、繁忙期を事前に把握するため
受入条件:
  - 「今年度の行政スケジュール」で年間カレンダーが表示される
  - 月ごとの書類一覧と期限が時系列で表示される
  - 特に重要な書類（年次報告等）は強調表示される
```

### 2.2 管理者のユーザーストーリー

#### US-12-06: 担当書類のリマインド受信

```
役割:   管理者（サービス管理責任者等）
目的:   自分が担当する書類のリマインドを適切なタイミングで受け取りたい
理由:   担当書類の準備を計画的に進めるため
受入条件:
  - 自分が担当者として割り当てられた書類のリマインドのみ受け取る
  - リマインドには書類名、期限、必要な準備事項が含まれる
  - 「対応中」「完了」等のステータス更新ができる
```

**利用シーン**: 管理者が朝の業務開始時にAI秘書から「担当書類のリマインド：実績報告書の提出期限が7日後です。3月分の日報データは全て入力済みです。サービス提供実績記録票の作成を開始してください」という通知を受け取る。

#### US-12-07: 必要データの入力状況確認

```
役割:   管理者
目的:   書類作成に必要なデータの入力状況を確認したい
理由:   データ不足で書類作成が遅れるのを防ぐため
受入条件:
  - 「実績報告に必要なデータ揃ってる？」で確認できる
  - 日報、出欠、工賃等の入力状況が表示される
  - 未入力がある場合は対象日と対象者が特定される
```

## 5. データモデル

### 5.1 ER図（テキスト表現）

```
admin_documents (書類マスタ)
├── 1:N → admin_document_records (提出記録)
│         ├── N:1 → users (提出者)
│         └── 参照 → daily_reports (日報データ)
│         └── 参照 → attendance_records (出欠データ)
│         └── 参照 → wage_records (工賃データ)
└── N:1 → users (担当者: responsible_role経由)
```

### 5.2 テーブル定義

#### admin_documents テーブル（書類マスタ）

```sql
-- =============================================
-- 行政書類マスタテーブル
-- 就労継続支援事業所が行政に提出する書類の
-- マスタ情報を管理する
-- =============================================
CREATE TABLE admin_documents (
    -- 主キー
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    -- 書類基本情報
    name VARCHAR(200) NOT NULL,
    description TEXT,

    -- 提出頻度
    -- monthly:    毎月提出
    -- quarterly:  四半期ごと（3ヶ月ごと）
    -- yearly:     年1回提出
    -- one_time:   随時（変更届、事故報告等）
    frequency VARCHAR(20) NOT NULL
        CHECK (frequency IN ('monthly', 'quarterly', 'yearly', 'one_time')),

    -- 期限日の設定
    -- monthly の場合: due_day_of_month に日を設定（例: 10 → 毎月10日）
    -- quarterly の場合: due_day_of_month + due_months（JSON配列 例: [4,7,10,1]）
    -- yearly の場合: due_month + due_day_of_month（例: 7月31日）
    -- one_time の場合: NULL（都度設定）
    due_day_of_month INTEGER
        CHECK (due_day_of_month IS NULL OR (due_day_of_month >= 1 AND due_day_of_month <= 31)),
    due_month INTEGER
        CHECK (due_month IS NULL OR (due_month >= 1 AND due_month <= 12)),

    -- 提出先情報
    submission_target VARCHAR(200),
    -- 例: '国保連合会', '○○市障害福祉課', '○○県障害福祉課'

    -- 担当ロール
    -- admin: 管理者
    -- owner: 社長
    -- staff: スタッフ
    responsible_role VARCHAR(50) NOT NULL DEFAULT 'admin',

    -- 担当者（特定のユーザーを指定する場合）
    responsible_user_id UUID REFERENCES users(id),

    -- 書類作成に必要なデータソース（JSONB）
    -- 例: {"daily_reports": true, "attendance": true, "wages": false}
    required_data JSONB DEFAULT '{}',

    -- 書類に関する補足情報
    -- 作成手順、注意事項、参考URLなどを記載
    notes TEXT,

    -- 有効/無効フラグ
    -- 制度変更等で不要になった書類は無効化する
    is_active BOOLEAN NOT NULL DEFAULT TRUE,

    -- カテゴリ
    -- regular:  定期提出書類
    -- ad_hoc:   随時提出書類
    -- special:  特別な書類（加算関連等）
    category VARCHAR(50) DEFAULT 'regular'
        CHECK (category IN ('regular', 'ad_hoc', 'special')),

    -- 重要度
    -- critical: 未提出で報酬に直接影響（実績報告書等）
    -- high:     未提出で加算取消し等の影響（処遇改善関連等）
    -- medium:   未提出で行政指導の対象
    -- low:      提出が望ましいが即座の影響は少ない
    importance VARCHAR(20) NOT NULL DEFAULT 'medium'
        CHECK (importance IN ('critical', 'high', 'medium', 'low')),

    -- タイムスタンプ
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- インデックス
CREATE INDEX idx_admin_documents_frequency ON admin_documents(frequency);
CREATE INDEX idx_admin_documents_is_active ON admin_documents(is_active);
CREATE INDEX idx_admin_documents_importance ON admin_documents(importance);
CREATE INDEX idx_admin_documents_responsible_user ON admin_documents(responsible_user_id);

-- 書類名のユニーク制約（有効な書類に限る）
CREATE UNIQUE INDEX idx_admin_documents_name_unique
    ON admin_documents(name)
    WHERE is_active = TRUE;

-- updated_at 自動更新トリガー
CREATE OR REPLACE FUNCTION update_admin_documents_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_admin_documents_updated_at
    BEFORE UPDATE ON admin_documents
    FOR EACH ROW
    EXECUTE FUNCTION update_admin_documents_updated_at();

-- テーブルコメント
COMMENT ON TABLE admin_documents IS '行政書類マスタ: 就労継続支援事業所が行政機関に提出する書類の定義';
COMMENT ON COLUMN admin_documents.frequency IS '提出頻度: monthly(毎月), quarterly(四半期), yearly(年1回), one_time(随時)';
COMMENT ON COLUMN admin_documents.due_day_of_month IS '期限日（月内の日）: 毎月10日なら10';
COMMENT ON COLUMN admin_documents.due_month IS '期限月（年次書類の場合）: 7月なら7';
COMMENT ON COLUMN admin_documents.required_data IS '必要データソース: {"daily_reports": true, "attendance": true, "wages": true}';
COMMENT ON COLUMN admin_documents.importance IS '重要度: critical(報酬影響), high(加算影響), medium(指導対象), low(推奨)';
```

#### admin_document_records テーブル（提出記録）

```sql
-- =============================================
-- 行政書類提出記録テーブル
-- 各期間における書類の提出状態を管理する
-- =============================================
CREATE TABLE admin_document_records (
    -- 主キー
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    -- 対象書類（外部キー）
    document_id UUID NOT NULL REFERENCES admin_documents(id) ON DELETE RESTRICT,

    -- 対象期間
    -- monthly: '2026-03'（年月）
    -- quarterly: '2026-Q1'（年度＋四半期）
    -- yearly: '2025'（年度）
    -- one_time: '2026-03-15'（発生日）
    target_period VARCHAR(20) NOT NULL,

    -- 提出期限日（実際の日付）
    -- 祝日・休日の場合は前営業日に調整済みの日付
    due_date DATE NOT NULL,

    -- 元の提出期限日（調整前）
    -- 祝日調整があった場合に元の期限を保持
    original_due_date DATE,

    -- ステータス
    -- pending:     未着手（レコード作成直後）
    -- in_progress: 対応中（担当者が作業開始）
    -- submitted:   提出済み
    -- overdue:     期限超過（未提出のまま期限を過ぎた）
    -- cancelled:   取消し（制度変更等で提出不要になった）
    status VARCHAR(20) NOT NULL DEFAULT 'pending'
        CHECK (status IN ('pending', 'in_progress', 'submitted', 'overdue', 'cancelled')),

    -- 提出情報
    submitted_at TIMESTAMPTZ,
    submitted_by UUID REFERENCES users(id),

    -- 担当者（書類マスタのデフォルトを上書きする場合）
    assigned_to UUID REFERENCES users(id),

    -- 備考
    note TEXT,

    -- 通知履歴（JSONB）
    -- どの段階の通知が送信済みかを記録
    -- 例: {"7_days": "2026-04-03T08:00:00Z", "3_days": "2026-04-07T08:00:00Z"}
    notification_history JSONB DEFAULT '{}',

    -- タイムスタンプ
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- インデックス
CREATE INDEX idx_admin_doc_records_document_id ON admin_document_records(document_id);
CREATE INDEX idx_admin_doc_records_due_date ON admin_document_records(due_date);
CREATE INDEX idx_admin_doc_records_status ON admin_document_records(status);
CREATE INDEX idx_admin_doc_records_target_period ON admin_document_records(target_period);
CREATE INDEX idx_admin_doc_records_assigned_to ON admin_document_records(assigned_to);

-- 同一書類・同一期間のレコードはユニーク
CREATE UNIQUE INDEX idx_admin_doc_records_unique_period
    ON admin_document_records(document_id, target_period);

-- 期限超過の書類を素早く検索するための部分インデックス
CREATE INDEX idx_admin_doc_records_overdue
    ON admin_document_records(due_date)
    WHERE status IN ('pending', 'in_progress', 'overdue');

-- updated_at 自動更新トリガー
CREATE OR REPLACE FUNCTION update_admin_doc_records_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_admin_doc_records_updated_at
    BEFORE UPDATE ON admin_document_records
    FOR EACH ROW
    EXECUTE FUNCTION update_admin_doc_records_updated_at();

-- ステータス変更時の自動処理トリガー
-- submitted になったら submitted_at を自動設定
CREATE OR REPLACE FUNCTION handle_admin_doc_record_status_change()
RETURNS TRIGGER AS $$
BEGIN
    -- 提出済みに変更された場合
    IF NEW.status = 'submitted' AND OLD.status != 'submitted' THEN
        IF NEW.submitted_at IS NULL THEN
            NEW.submitted_at = NOW();
        END IF;
    END IF;

    -- 期限超過に変更された場合（Cronジョブから呼ばれる想定）
    IF NEW.status = 'overdue' AND OLD.status != 'overdue' THEN
        -- notification_history に overdue 通知の予約を追加
        NEW.notification_history = COALESCE(NEW.notification_history, '{}')::jsonb
            || jsonb_build_object('overdue_detected', NOW()::text);
    END IF;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_admin_doc_record_status_change
    BEFORE UPDATE OF status ON admin_document_records
    FOR EACH ROW
    EXECUTE FUNCTION handle_admin_doc_record_status_change();

-- テーブルコメント
COMMENT ON TABLE admin_document_records IS '行政書類提出記録: 各期間における書類の提出状態と履歴';
COMMENT ON COLUMN admin_document_records.target_period IS '対象期間: monthly→2026-03, quarterly→2026-Q1, yearly→2025';
COMMENT ON COLUMN admin_document_records.due_date IS '提出期限日（祝日調整済み）';
COMMENT ON COLUMN admin_document_records.original_due_date IS '元の提出期限日（調整前の日付）';
COMMENT ON COLUMN admin_document_records.status IS 'ステータス: pending/in_progress/submitted/overdue/cancelled';
COMMENT ON COLUMN admin_document_records.notification_history IS '通知送信履歴: {"7_days": "2026-04-03T08:00:00Z", ...}';
```

```sql
-- =============================================
CREATE OR REPLACE VIEW v_annual_submission_summary AS
SELECT
    d.name AS document_name,
    d.frequency,
    d.importance,
    COUNT(*) AS total_records,
    COUNT(*) FILTER (WHERE r.status = 'submitted') AS submitted_count,
    COUNT(*) FILTER (WHERE r.status = 'overdue') AS overdue_count,
    COUNT(*) FILTER (WHERE r.status IN ('pending', 'in_progress')) AS pending_count,
    ROUND(
        COUNT(*) FILTER (WHERE r.status = 'submitted')::NUMERIC
        / NULLIF(COUNT(*) FILTER (WHERE r.status != 'cancelled'), 0) * 100,
        1
    ) AS submission_rate_pct,
    -- 平均提出タイミング（期限の何日前に提出したか）

        AVG(r.due_date - r.submitted_at::DATE) FILTER (WHERE r.status = 'submitted'),
        1
    ) AS avg_days_before_due
FROM admin_documents d
LEFT JOIN admin_document_records r ON d.id = r.document_id
WHERE d.is_active = TRUE
GROUP BY d.id, d.name, d.frequency, d.importance
ORDER BY d.importance, d.name;
```

```typescript
// cron/generateMonthlyReminders.ts

async function generateMonthlyReminders() {
  const now = new Date();
  const currentYear = now.getFullYear();
  const currentMonth = now.getMonth() + 1; // 1-12

  // 対象期間（前月分）
  const targetPeriodDate = new Date(currentYear, currentMonth - 2, 1);
  const targetPeriod = `${targetPeriodDate.getFullYear()}-${String(targetPeriodDate.getMonth() + 1).padStart(2, '0')}`;

  // ① 月次書類を取得
  const monthlyDocs = await supabase
    .from('admin_documents')
    .select('*')
    .eq('is_active', true)
    .eq('frequency', 'monthly');

  // ② 年次書類で当月が期限月のものを取得
  const yearlyDocs = await supabase
    .from('admin_documents')
    .select('*')
    .eq('is_active', true)
    .eq('frequency', 'yearly')
    .eq('due_month', currentMonth);

  const allDocs = [...monthlyDocs.data, ...yearlyDocs.data];
  const createdRecords = [];

  for (const doc of allDocs) {
    // ③ 既存レコードの確認
    const existing = await supabase
      .from('admin_document_records')
      .select('id')
      .eq('document_id', doc.id)
      .eq('target_period', doc.frequency === 'yearly'
        ? String(currentYear - 1) // 年次は前年度分
        : targetPeriod
      )
      .single();

    if (existing.data) continue; // 既に存在する場合はスキップ

    // ④ 期限日の計算
    let dueDate = new Date(currentYear, currentMonth - 1, doc.due_day_of_month);
    const originalDueDate = new Date(dueDate);

    // 祝日・休日チェックと前営業日調整
    dueDate = await adjustToBusinessDay(dueDate);

    // ⑤ レコード作成
    const record = await supabase
      .from('admin_document_records')
      .insert({
        document_id: doc.id,
        target_period: doc.frequency === 'yearly'
          ? String(currentYear - 1)
          : targetPeriod,
        due_date: dueDate.toISOString().split('T')[0],
        original_due_date: originalDueDate.toISOString().split('T')[0],
        status: 'pending',
      })
      .select()
      .single();

    createdRecords.push({ doc, record: record.data });
  }

  // ⑥ 前月の未提出レコードを overdue に更新
  await supabase
    .from('admin_document_records')
    .update({ status: 'overdue' })
    .lt('due_date', `${currentYear}-${String(currentMonth).padStart(2, '0')}-01`)
    .in('status', ['pending', 'in_progress']);

  // ⑦⑧ 通知メッセージの生成と送信
  const message = buildMonthlyTaskMessage(createdRecords, currentMonth);
  await sendLineNotification('owner', message);
}

// 営業日調整関数
async function adjustToBusinessDay(date: Date): Promise<Date> {
  let adjusted = new Date(date);

  while (true) {
    const dayOfWeek = adjusted.getDay();
    // 土曜日(6)または日曜日(0)の場合
    if (dayOfWeek === 0 || dayOfWeek === 6) {
      adjusted.setDate(adjusted.getDate() - 1);
      continue;
    }

    // 祝日チェック
    const dateStr = adjusted.toISOString().split('T')[0];
    const holiday = await supabase
      .from('holidays')
      .select('id')
      .eq('holiday_date', dateStr)
      .single();

    if (holiday.data) {
      adjusted.setDate(adjusted.getDate() - 1);
      continue;
    }

    break; // 営業日に到達
  }

  return adjusted;
}
```

```typescript
// cron/dailyDeadlineCheck.ts

interface NotificationTarget {
  userId: string;
  role: string;
  message: string;
  urgency: 'normal' | 'warning' | 'urgent' | 'critical';
}

async function dailyDeadlineCheck() {
  const today = new Date();
  const todayStr = today.toISOString().split('T')[0];

  // ① 未提出レコードを取得
  const { data: pendingRecords } = await supabase
    .from('admin_document_records')
    .select(`
      *,
      document:admin_documents(*)
    `)
    .in('status', ['pending', 'in_progress', 'overdue']);

  const notifications: NotificationTarget[] = [];

  for (const record of pendingRecords) {
    const dueDate = new Date(record.due_date);
    const daysRemaining = Math.ceil(
      (dueDate.getTime() - today.getTime()) / (1000 * 60 * 60 * 24)
    );
    const notifHistory = record.notification_history || {};

    // ③ 段階的通知の判定
    if (daysRemaining === 7 && !notifHistory['7_days']) {
      // 7日前: 担当者へ通知
      const dataStatus = await checkRequiredData(record);
      notifications.push({
        userId: record.assigned_to || record.document.responsible_user_id,
        role: record.document.responsible_role,
        message: build7DayReminder(record, dataStatus),
        urgency: 'normal',
      });
      await updateNotificationHistory(record.id, '7_days');

    } else if (daysRemaining === 3 && !notifHistory['3_days']) {
      // 3日前: 担当者＋社長へ通知
      notifications.push({
        userId: record.assigned_to || record.document.responsible_user_id,
        role: 'admin',
        message: build3DayReminder(record),
        urgency: 'warning',
      });
      notifications.push({
        userId: null,
        role: 'owner',
        message: build3DayReminderForOwner(record),
        urgency: 'warning',
      });
      await updateNotificationHistory(record.id, '3_days');

    } else if (daysRemaining === 0 && !notifHistory['due_day']) {
      // 当日: 社長へ緊急通知
      notifications.push({
        userId: null,
        role: 'owner',
        message: buildDueDayReminder(record),
        urgency: 'urgent',
      });
      await updateNotificationHistory(record.id, 'due_day');

    } else if (daysRemaining < 0) {
      // 超過: 毎日社長へ通知
      if (record.status !== 'overdue') {
        await supabase
          .from('admin_document_records')
          .update({ status: 'overdue' })
          .eq('id', record.id);
      }
      notifications.push({
        userId: null,
        role: 'owner',
        message: buildOverdueReminder(record, Math.abs(daysRemaining)),
        urgency: 'critical',
      });
      await updateNotificationHistory(
        record.id,
        `overdue_${todayStr}`
      );
    }
  }

  // 通知の送信
  for (const notif of notifications) {
    await sendLineNotification(notif.role, notif.message, notif.urgency);
  }
}
```

```typescript
// api/admin-documents/complete.ts

interface CompleteDocumentRequest {
  recordId?: string;       // レコードIDで直接指定
  documentName?: string;   // 書類名で指定（自然言語対応）
  targetPeriod?: string;   // 対象期間
  note?: string;           // 備考
}

async function completeDocument(
  req: CompleteDocumentRequest,
  userId: string
) {
  let record;

  if (req.recordId) {
    // レコードIDで直接取得
    record = await supabase
      .from('admin_document_records')
      .select('*, document:admin_documents(*)')
      .eq('id', req.recordId)
      .single();

  } else if (req.documentName) {
    // 書類名で検索（あいまい検索対応）
    const { data: docs } = await supabase
      .from('admin_documents')
      .select('id, name')
      .eq('is_active', true)
      .ilike('name', `%${req.documentName}%`);

    if (!docs || docs.length === 0) {
      throw new Error(`「${req.documentName}」に該当する書類が見つかりません`);
    }

    if (docs.length > 1) {
      // 複数候補がある場合は確認を求める
      return {
        status: 'ambiguous',
        candidates: docs.map(d => d.name),
        message: '複数の書類が見つかりました。どちらですか？',
      };
    }

    // 最新の未提出レコードを取得
    record = await supabase
      .from('admin_document_records')
      .select('*, document:admin_documents(*)')
      .eq('document_id', docs[0].id)
      .in('status', ['pending', 'in_progress', 'overdue'])
      .order('due_date', { ascending: false })
      .limit(1)
      .single();
  }

  if (!record?.data) {
    throw new Error('対象の書類レコードが見つかりません');
  }

  // ステータス更新
  const { data: updated } = await supabase
    .from('admin_document_records')
    .update({
      status: 'submitted',
      submitted_at: new Date().toISOString(),
      submitted_by: userId,
      note: req.note || null,
    })
    .eq('id', record.data.id)
    .select()
    .single();

  // 残りの未提出タスクを取得
  const { data: remaining } = await supabase
    .from('admin_document_records')
    .select('*, document:admin_documents(name)')
    .in('status', ['pending', 'in_progress', 'overdue'])
    .gte('due_date', new Date().toISOString().split('T')[0])
    .order('due_date', { ascending: true });

  return {
    status: 'completed',
    record: updated,
    daysBeforeDue: Math.ceil(
      (new Date(updated.due_date).getTime() - new Date(updated.submitted_at).getTime())
      / (1000 * 60 * 60 * 24)
    ),
    remainingTasks: remaining,
  };
}
```

### 6.4 自然言語クエリ処理

```typescript
// ai/adminDocumentIntent.ts

interface AdminDocumentIntent {
  action: 'check_deadline' | 'complete' | 'list_monthly'
        | 'list_annual' | 'check_status' | 'change_assignee';
  documentName?: string;
  period?: string;
  newAssignee?: string;
}

function parseAdminDocumentQuery(userMessage: string): AdminDocumentIntent | null {
  // 期限確認パターン
  const deadlinePatterns = [
    /(.+?)(?:の|は)(?:期限|締め切り|いつまで|いつ)/,
    /(.+?)(?:いつ|期限)/,
  ];

  // 完了報告パターン
  const completePatterns = [
    /(.+?)(?:出した|提出した|完了|出しました|終わった|送った)/,
    /(.+?)(?:を|は)(?:提出|完了)/,
  ];

  // 月次一覧パターン
  const monthlyPatterns = [
    /今月の(?:行政|書類|タスク)/,
    /(?:行政|書類)(?:の|)(?:タスク|一覧|リスト)/,
  ];

  // 年間スケジュールパターン
  const annualPatterns = [
    /(?:今年度|年間|1年)の(?:.*?)(?:スケジュール|期限|予定)/,
    /主要(?:な|)(?:期限|スケジュール)/,
  ];

  for (const pattern of deadlinePatterns) {
    const match = userMessage.match(pattern);
    if (match) {
      return {
        action: 'check_deadline',
        documentName: match[1].trim(),
      };
    }
  }

  for (const pattern of completePatterns) {
    const match = userMessage.match(pattern);
    if (match) {
      return {
        action: 'complete',
        documentName: match[1].trim(),
      };
    }
  }

  for (const pattern of monthlyPatterns) {
    if (pattern.test(userMessage)) {
      return { action: 'list_monthly' };
    }
  }

  for (const pattern of annualPatterns) {
    if (pattern.test(userMessage)) {
      return { action: 'list_annual' };
    }
  }

  return null;
}
```

### 7.3 通知抑制ルール

- **提出済み書類**: ステータスが `submitted` の書類には一切の通知を送らない
- **取消済み書類**: ステータスが `cancelled` の書類には一切の通知を送らない
- **同一段階の重複防止**: `notification_history` に記録済みの段階は再送しない（ただし `overdue` は例外で毎日送信）
- **夜間・早朝の抑制**: 通知は08:00〜20:00の間のみ送信。時間外に生成された通知は翌朝08:00に送信
- **通知頻度の上限**: 同一ユーザーへの1日あたりの通知は最大10件まで。超過分は翌日に繰り越し

### 7.4 通知チャネル

|--------|---------|--------------|

| 緊急 | LINE + プッシュ通知 | SMS（設定がある場合） |
| 最重要 | LINE + プッシュ通知 | SMS + メール |

## 8. エッジケース・エラー処理

### 8.1 祝日・休日が期限日の場合

**ルール**: 期限日が土日祝日に該当する場合は、直前の営業日（平日かつ祝日でない日）を実質期限とする。

```
例: 4月20日（日）が期限の場合
    → 4/19（土）もスキップ
    → 4/18（金）を実質期限とする

例: 5月5日（月・こどもの日）が期限の場合
    → 5/4（日）もスキップ
    → 5/3（土・憲法記念日）もスキップ
    → 5/2（金）を実質期限とする
```

**処理の詳細**:
- `admin_document_records.due_date` には調整後の日付を格納
- `admin_document_records.original_due_date` には調整前の日付を格納
- 通知メッセージには「※ {元の期限日}は{祝日名/休日}のため{調整後期限日}に繰り上げ」と表示
- 祝日マスタ（`holidays` テーブル）は毎年12月に翌年分を投入するCronジョブで更新

### 8.2 行政の制度変更で期限が変わった場合

**ルール**: 書類マスタの更新が必要。管理者権限で変更可能。

```
対応フロー:
1. 社長/管理者が「○○の期限が△日に変わった」と報告
2. AI秘書が確認: 「{書類名}の期限を毎月{新しい日}日に変更しますか？」
3. 確認後、admin_documents の due_day_of_month を更新
4. 既に作成済みの未提出レコード（admin_document_records）の
   due_date も再計算して更新
5. 変更履歴を note に記録
6. 関係者に通知: 「{書類名}の期限が変更されました（旧: 毎月{旧日}日 → 新: 毎月{新日}日）」
```

**会話例**:
```
社長: 「来月から実績報告書の期限が15日に変わるらしい」

AI:  「実績報告書の提出期限を変更します。
      変更前: 毎月10日
      変更後: 毎月15日
      適用:   2026年5月提出分（4月分）から

      この変更でよろしいですか？」

社長: 「はい」

AI:  「変更を保存しました。
      ・書類マスタを更新しました
      ・5月分の期限を5/15に再設定しました
      ・田中管理者にも変更を通知しました」
```

### 8.3 書類の担当者が変更になった場合

**ルール**: 担当者変更は即座に反映。変更月の書類は新旧どちらの担当者にも通知。

```
対応フロー:
1. 社長が「○○の担当を△△さんに変更」と指示
2. admin_documents の responsible_user_id を更新
3. 進行中の admin_document_records の assigned_to も更新（任意）
4. 新担当者に引き継ぎ通知を送信
5. 旧担当者にも変更通知を送信
```

**考慮事項**:
- 変更月は旧担当者がすでに着手している可能性がある。その場合は「今月分は{旧担当者}が継続、来月分から{新担当者}に変更」とすることもできる
- 担当者が退職した場合は、書類マスタの担当をデフォルト（社長）に戻し、新担当者のアサインを促す通知を送る

### 8.4 年度をまたぐ書類の扱い

**ルール**: 年度は4月始まり3月終わり。年次書類の対象年度と提出時期をを正しく管理する。

```
例: 事業報告書
  対象年度: 2025年度（2025年4月〜2026年3月）
  提出期限: 2026年6月30日
  target_period: '2025'（対象年度）

例: 3月分の実績報告書
  対象月: 2026年3月（2025年度の最終月）
  提出期限: 2026年4月10日（2026年度の最初の月）
  target_period: '2026-03'

年度切り替え時の注意:
- 4月1日の月次リマインド生成時に、3月分（前年度最終月）の
  書類も忘れずに含める
- 年次書類（事業報告書等）の対象年度は「前年度」であることに注意
- 年度初め（4月）は前年度の後処理と新年度の書類が重なるため
  通知が増える→社長に注意喚起のメッセージを追加
```

### 8.5 Cronジョブの失敗

```
対応フロー:
1. Cronジョブ実行時にエラーが発生
2. エラーログを記録（Supabase Edge Function のログ）
3. 自動リトライ（最大3回、5分間隔）
4. 3回失敗した場合:
   - システム管理者にメール通知
   - 次回のCronジョブ実行時に未処理分を含めて再実行
5. 手動リカバリ: 管理画面から「リマインド再生成」を実行可能
```

### 8.6 LINE通知の送信失敗

```
対応フロー:
1. LINE API のレスポンスでエラーを検出
2. リトライ（最大3回、指数バックオフ: 1分, 5分, 30分）
3. 全リトライ失敗の場合:
   - notification_history にエラーを記録
   - 次回のCronジョブ実行時に再送対象として検出
   - 別チャネル（メール等）でのフォールバック送信
```

### 8.7 同一書類の複数期間レコードが存在する場合

```
例: 3月分の実績報告書が未提出のまま4月を迎えた
  3月分: status = 'overdue', due_date = 4/10
  4月分: status = 'pending', due_date = 5/10（新規生成）

対応:
- 両方のレコードが通知対象となる
- overdue の書類は通知メッセージの冒頭に「⚠ 前月分の未提出書類があります」と表示
- 月初の一覧通知では overdue を先頭に表示
```

### 9.4 重要度の評価

行政書類の期限管理は、事業所の存続に直結する極めて重要な機能である。

- **報酬への影響**: 実績報告書の未提出は訓練等給付費の支払い遅延を意味し、事業所のキャッシュフローに直接影響する
- **加算への影響**: 処遇改善加算の実績報告未提出は加算の取消しとなり、年間数百万円の減収につながる可能性がある
- **指定への影響**: 継続的な書類未提出は指定取消し処分の対象となり、事業所の閉鎖を意味する

このため、**開発優先度は「最高」**とし、システム基盤・ユーザー管理の直後に着手することを推奨する。
