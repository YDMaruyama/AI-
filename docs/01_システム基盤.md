# 01. システム基盤

## 1. 概要

### 1.1 この機能が解決する課題
AI秘書システムの全機能が動作するための技術基盤を構築する。LINE公式アカウントからのメッセージを受信し、AIで処理し、データベースに保存し、返答を返す一連のパイプラインを確立する。

### 1.2 提供する価値
- すべての機能が共通で利用するインフラストラクチャ
- メッセージの受信→AI処理→DB保存→返答のパイプライン
- 安定した応答速度（3秒以内の初回応答目標）
- スケーラブルな設計（利用者数増加に対応）

---

## 2. ユーザーストーリー

### 2.1 初期セットアップ（開発者）
```
開発者として、
LINE Bot → Vercel → Claude API → Supabase の一連の接続を構築し、
「こんにちは」とLINEで送ったらAIが返答する状態を作りたい。
```

### 2.2 基本的な対話（全ユーザー）
```
スタッフとして、
LINEでメッセージを送ったら、
AI秘書が適切な返答を3秒以内に返してくれることを期待する。
```

## 3. 技術構成の詳細

### 3.1 LINE Messaging API

#### 必要なアカウント設定
| 項目 | 設定値 |
|------|--------|
| アカウントタイプ | LINE公式アカウント（認証済み推奨） |
| プラン | コミュニケーションプラン以上 |
| Messaging API | 有効化 |
| Webhook URL | `https://<project>.vercel.app/api/webhook` |
| チャネルアクセストークン | 長期トークンを使用 |
| チャネルシークレット | Webhook署名検証に使用 |

#### メッセージ形式
- **テキストメッセージ**: 基本の対話
- **リッチメニュー**: 「日報を書く」「タスク確認」等のショートカット
- **クイックリプライ**: 選択肢の提示（「はい/いいえ」等）
- **Flex Message**: 日報やタスク一覧の整形表示

#### LINE APIの制約
| 制約 | 値 | 対処法 |
|------|-----|--------|
| Reply token有効期限 | 30秒 | 即座にreplyし、長い処理はpush messageで送信 |
| メッセージサイズ上限 | 5,000文字 | 長い返答は分割送信 |
| 月間無料メッセージ | 200通（プランによる） | pushメッセージを最適化 |
| リクエスト頻度制限 | 制限あり | キューイング対応 |

### 3.3 Claude API (AI処理)

#### モデル選定
| 用途 | モデル | 理由 |
|------|--------|------|
| 日報アシスト | claude-sonnet-4-6 | コスト効率と品質のバランス |
| タスク抽出 | claude-haiku-4-5 | 高速・低コスト。構造化出力に適する |
| 社長の質問応答 | claude-sonnet-4-6 | 複雑な質問への対応力 |

#### システムプロンプト設計方針
- **ロールに応じた切り替え**: 社長/管理者/スタッフで異なるプロンプト
- **コンテキスト注入**: ユーザー情報・会話履歴・関連データをプロンプトに含める
- **出力形式の指定**: JSON形式での構造化出力を求める（タスク抽出等）
- **日本語特化**: 就労支援業界の用語を理解するよう指示

#### 会話管理
```
会話状態の管理フロー：

1. ユーザーがメッセージ送信
2. Supabaseから会話履歴（直近10件）を取得
3. ユーザーのロール・現在の状態を取得
4. コンテキストを組み立ててClaude APIに送信
5. AIの返答を保存・送信
6. 必要に応じて状態を更新（例：日報作成中→日報完成）
```

#### 会話状態（conversation_state）
| 状態 | 説明 |
|------|------|
| idle | 通常状態。何でも受け付ける |
| writing_report | 日報作成中。AIが質問を続ける |
| confirming_report | 日報確認中。送信するかの確認 |
| writing_incident | 事故報告作成中 |
| confirming_task | タスク確認中 |

### 3.4 Supabase (データベース)

#### 接続方式
- **サーバーサイド**: Service Role Key（RLSバイパス）- Vercel Edge Functionsから使用
- **注意**: クライアントサイドアクセスは行わない（LINE Bot経由のため）

#### リアルタイム機能（将来）
- Supabase Realtimeを使って、データ変更時にWebhookを発火させることが可能
- 例：日報が保存されたら自動で社長に通知をトリガー

## 5. データモデル

### 5.1 会話履歴テーブル（conversation_messages）

```sql
CREATE TABLE conversation_messages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id),
    role TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system')),
    content TEXT NOT NULL,
    metadata JSONB DEFAULT '{}',
    -- metadata例: { "intent": "daily_report", "state": "writing_report" }
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_conv_messages_user_id ON conversation_messages(user_id);
CREATE INDEX idx_conv_messages_created_at ON conversation_messages(created_at DESC);
```

### 5.2 会話状態テーブル（conversation_states）

```sql
CREATE TABLE conversation_states (
    user_id UUID PRIMARY KEY REFERENCES users(id),
    state TEXT NOT NULL DEFAULT 'idle',
    context JSONB DEFAULT '{}',
    -- context例: { "report_date": "2026-03-30", "collected_data": {...} }
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## 6. API/処理フロー

### 6.1 メッセージ受信→応答の全体フロー

```
LINE App
  │
  │ POST /api/webhook
  │ (署名付きHTTPリクエスト)
  ▼
┌─────────────────────────────────┐
│ api/webhook.ts                  │
│                                 │
│ 1. 署名検証 (LINE Channel Secret) │
│ 2. イベント解析                   │
│ 3. ユーザー識別 (LINE User ID)    │
│    └→ users テーブルを検索        │
│ 4. 会話状態を取得                 │
│    └→ conversation_states を検索  │
│ 5. ルーティング                   │
│    ├→ 日報関連 → 日報ハンドラー    │
│    ├→ タスク関連 → タスクハンドラー │
│    ├→ 質問 → 質問応答ハンドラー    │
│    └→ その他 → 汎用AIハンドラー   │
│ 6. Claude API で応答生成          │
│ 7. 会話履歴を保存                 │
│ 8. LINE Reply API で返答送信     │
└─────────────────────────────────┘
```

### 6.2 Reply Token の時間制約対応

```
LINE Webhookを受信
    │
    ├─ 処理が3秒以内に完了する場合
    │   └→ Reply Token で直接返答
    │
    └─ 処理が3秒以上かかる場合
        ├→ Reply Token で「処理中です...」を即座に返答
        └→ 処理完了後に Push Message で本文を送信
```

### 6.3 エラーハンドリングフロー

```
Claude API呼び出し
    │
    ├─ 成功 → 通常応答
    │
    ├─ タイムアウト (30秒)
    │   └→ 「申し訳ありません。処理に時間がかかっています。
    │        もう一度お試しください。」
    │
    ├─ レート制限 (429)
    │   └→ 「現在アクセスが集中しています。
    │        1分後にもう一度お試しください。」
    │
    └─ その他のエラー (500等)
        └→ 「システムエラーが発生しました。
             管理者に連絡してください。」
        └→ エラーログをSupabaseに保存
```

## 8. エッジケース・エラー処理

### 8.1 想定されるエッジケース

| ケース | 対処 |
|--------|------|
| 未登録ユーザーからのメッセージ | 「登録されていません。管理者に連絡してください」と返答 |
| 同時に複数メッセージが送られた | キューイングで順次処理 |
| Claude APIがダウン | フォールバックメッセージを返す |
| Supabaseがダウン | メッセージをメモリに一時保持、復旧後に保存 |
| 非テキストメッセージ（スタンプ等） | 「テキストメッセージでお願いします」と返答 |
| 極端に長いメッセージ（5000文字超） | 先頭2000文字を処理し「長すぎるため一部のみ処理しました」と通知 |
| Webhook重複送信 | イベントIDで重複排除 |

### 8.2 セキュリティ対策

| 脅威 | 対策 |
|------|------|
| 不正なWebhookリクエスト | LINE署名検証（X-Line-Signature ヘッダー） |
| SQLインジェクション | Supabase SDKのパラメータバインディング使用 |
| APIキー漏洩 | 環境変数管理、GitHubにはコミットしない |
| 不正アクセス | ユーザー登録制（LINE User IDベース） |

## 9. 開発優先度・依存関係

### 9.1 このドキュメントの実装順序

```
Step 1: プロジェクト初期化
  └→ npm init, TypeScript設定, Vercelプロジェクト作成

Step 2: LINE Bot接続
  └→ Webhook受信 → 署名検証 → Echo応答（オウム返し）

Step 3: Claude API接続
  └→ メッセージ → Claude API → AI応答 → LINE返答

Step 4: Supabase接続
  └→ テーブル作成 → ユーザー登録 → 会話履歴保存

Step 5: ルーティング
  └→ ユーザーロール判定 → 意図判定 → 適切なハンドラーへ
```

### 9.2 他機能への依存関係
- **全機能**がこの基盤に依存する
- 02_ユーザー管理は基盤の直後に実装必須
- 基盤が安定しないと他機能の開発に進めない
