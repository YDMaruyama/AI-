# 02. ユーザー管理

## 1. 概要

### 1.1 この機能が解決する課題
AI秘書は社長・管理者・スタッフの3つのロールで異なる体験を提供する必要がある。社長には全情報へのアクセスと経営視点の応答を、管理者には担当利用者の管理機能を、スタッフには簡易な記録機能を提供する。ロール判定なしでは、全員に同じ応答をすることになり、セキュリティリスクも生じる。

### 1.2 提供する価値
- LINEで初回メッセージを送るだけで自動的にユーザー登録される
- ロールに応じた適切な応答・機能・データアクセスの制御
- スタッフが他のスタッフの日報や利用者情報に不正アクセスできない
- 社長が一人で全スタッフを管理できる

---

## 2. ユーザーストーリー

### 2.1 社長：スタッフの登録
```
社長として、
新しいスタッフがLINEで友だち追加した時に通知を受け、
「管理者として登録」「スタッフとして登録」を選択して
ロールを割り当てたい。
```

### 2.2 スタッフ：初回利用
```
スタッフとして、
LINE公式アカウントを友だち追加して最初のメッセージを送ったら、
「まだ登録されていません。管理者に連絡してください」と案内され、
社長が登録してくれた後は普通に使えるようになりたい。
```

### 2.3 社長：スタッフ一覧確認
```
社長として、
「スタッフ一覧」と送ったら、
全スタッフの名前・ロール・最終ログイン日時が見たい。
```

### 2.4 社長：ロール変更
```
社長として、
スタッフAを管理者に昇格させたい時に、
「○○さんを管理者にして」と送るだけで変更したい。
```

## 3. 会話フロー（詳細）

### 3.1 新規ユーザーの初回メッセージ（正常系）

```
[未登録ユーザーがメッセージ送信]

AI「はじめまして！さくら映像研のAI秘書です。
  あなたはまだ登録されていません。
  管理者（社長）に登録を依頼してください。

  ※ あなたのLINE表示名：田中太郎」

    ↓ 同時に社長へ通知 ↓

AI → 社長
「新しいユーザーがメッセージを送りました：
  LINE表示名：田中太郎

  どのロールで登録しますか？
  ① 管理者として登録
  ② スタッフとして登録
  ③ 登録しない（拒否）」

社長「②」

AI → 社長
「田中太郎さんをスタッフとして登録しました。
  表示名を設定してください（日報等に使用します）」

社長「田中」

AI → 社長
「田中太郎さんを「田中」として登録完了しました。」

AI → 田中さん
「登録が完了しました！
  スタッフとしてご利用いただけます。

  できること：
  ・「日報」→ 日報を書く
  ・「タスク」→ タスクを確認する

  何かあればメッセージを送ってください！」
```

### 3.2 新規ユーザーの初回メッセージ（社長が拒否した場合）

```
社長「③」（登録しない）

AI → 社長
「田中太郎さんの登録を見送りました。」

AI → 田中太郎さん
（何も送信しない。次にメッセージが来た場合も同じ「未登録」メッセージを返す）
```

### 3.3 スタッフ一覧の確認

```
社長「スタッフ一覧」

AI「現在の登録メンバー：

  👑 社長
  ・丸山（最終利用：今日 10:30）

  📋 管理者（2名）
  ・佐藤（最終利用：今日 9:15）
  ・鈴木（最終利用：昨日 17:00）

  👤 スタッフ（3名）
  ・田中（最終利用：今日 8:50）
  ・山田（最終利用：3日前）⚠
  ・高橋（最終利用：今日 12:00）

  ⚠ 山田さんは3日間利用がありません。」
```

### 3.4 ロール変更

```
社長「田中を管理者にして」

AI「田中さんのロールを変更します：
  スタッフ → 管理者

  管理者になると以下が可能になります：
  ・担当利用者の支援記録の閲覧・編集
  ・他スタッフの日報の閲覧
  ・利用者の出欠管理

  変更してよろしいですか？」

社長「はい」

AI「田中さんを管理者に変更しました。
  田中さんに通知しますか？」

社長「通知して」

AI → 田中さん
「お知らせ：あなたのロールが「管理者」に変更されました。
  新しくできること：
  ・担当利用者の支援記録の閲覧・編集
  ・他スタッフの日報の閲覧
  ・利用者の出欠管理」
```

### 3.5 異常系：存在しないユーザー名でのロール変更

```
社長「佐々木を管理者にして」

AI「「佐々木」という名前のスタッフが見つかりません。
  登録メンバー一覧を表示しますか？」
```

## 4. 機能要件

### 4.1 必須機能

| # | 機能 | 詳細 |
|---|------|------|
| U-01 | ユーザー自動検知 | 未登録ユーザーからのメッセージを検知し、社長に通知する |
| U-02 | ロール割り当て | 社長が新規ユーザーにロール（管理者/スタッフ）を割り当てる |
| U-03 | 表示名設定 | ロール割り当て時に日報等で使用する表示名を設定する |
| U-04 | ロール変更 | 社長がスタッフのロールを変更できる |
| U-05 | スタッフ一覧 | 社長が全スタッフの一覧を確認できる |
| U-06 | アクセス制御 | ロールに応じたデータアクセスの制限 |
| U-07 | 最終利用日時記録 | 各ユーザーの最終メッセージ日時を記録する |

### 4.2 任意機能

| # | 機能 | 詳細 |
|---|------|------|
| U-08 | ユーザー無効化 | 退職したスタッフのアカウントを無効化（削除はしない） |
| U-09 | 担当利用者の紐付け | 管理者と担当利用者の関係を設定 |
| U-10 | プロフィール編集 | スタッフ自身が表示名を変更できる |
| U-11 | 利用統計 | 各スタッフの利用頻度・日報提出率等の統計 |

## 5. データモデル

### 5.1 ユーザーテーブル（users）

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    line_user_id TEXT UNIQUE NOT NULL,
    -- LINEのユーザーID (U + 32桁の16進数)
    line_display_name TEXT,
    -- LINEの表示名（プロフィールから取得）
    display_name TEXT NOT NULL,
    -- システム内での表示名（社長が設定）
    role TEXT NOT NULL DEFAULT 'pending'
        CHECK (role IN ('owner', 'manager', 'staff', 'pending', 'rejected')),
    -- owner: 社長, manager: 管理者, staff: スタッフ
    -- pending: 承認待ち, rejected: 拒否済み
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    -- FALSE = 退職等で無効化
    last_message_at TIMESTAMPTZ,
    -- 最後にメッセージを送った日時
    metadata JSONB DEFAULT '{}',
    -- 将来の拡張用（担当利用者一覧等）
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- インデックス
CREATE UNIQUE INDEX idx_users_line_user_id ON users(line_user_id);
CREATE INDEX idx_users_role ON users(role);
CREATE INDEX idx_users_is_active ON users(is_active);
```

### 5.2 ロール権限マトリクス

| 操作 | owner | manager | staff | pending |
|------|-------|---------|-------|---------|
| 自分の日報作成 | ○ | ○ | ○ | × |
| 自分の日報閲覧 | ○ | ○ | ○ | × |
| 全員の日報閲覧 | ○ | ○ | × | × |
| 日報の日付指定検索 | ○ | ○ | × | × |
| タスク一覧（全員） | ○ | × | × | × |
| タスク一覧（自分） | ○ | ○ | ○ | × |
| 利用者情報閲覧 | ○ | ○（担当のみ） | × | × |
| 売上・工賃情報 | ○ | × | × | × |
| スタッフ管理 | ○ | × | × | × |
| 設定変更 | ○ | × | × | × |

## 6. API/処理フロー

### 6.1 ユーザー識別フロー

```
メッセージ受信
    │
    ▼
LINE User ID を取得
    │
    ▼
users テーブルを検索
    │
    ├─ 見つかった場合
    │   ├─ role = 'pending' → 「登録待ちです」と返答
    │   ├─ role = 'rejected' → 何も返さない（無視）
    │   ├─ is_active = FALSE → 「アカウントが無効です」と返答
    │   └─ 有効なロール → last_message_at を更新、通常処理へ
    │
    └─ 見つからない場合
        ├→ 仮レコードを作成（role = 'pending'）
        ├→ 社長に承認リクエストを送信
        └→ ユーザーに「登録待ち」メッセージを送信
```

### 6.2 社長の初回セットアップフロー

```
[最初のユーザー（社長）がメッセージ送信]

システム検知：usersテーブルが空 → 最初のユーザーを社長と判定

AI「はじめまして！AI秘書のセットアップを開始します。
  あなたを社長（管理者）として登録します。

  表示名を教えてください（例：丸山）」

社長「丸山」

AI「丸山さん、登録完了です！
  あなたは社長として全機能をご利用いただけます。

  次のステップ：
  スタッフにこのLINEアカウントを友だち追加してもらってください。
  メッセージが届いたら、ロールの割り当てをお願いします。」
```

## 7. 通知ルール

| トリガー | 通知先 | 内容 |
|---------|--------|------|
| 未登録ユーザーがメッセージ送信 | 社長 | 新規ユーザーの承認リクエスト |
| ロールが割り当てられた | 該当ユーザー | ロール割り当て通知 + 利用ガイド |
| ロールが変更された | 該当ユーザー | ロール変更通知（社長が「通知する」を選んだ場合） |
| スタッフが3日以上利用なし | 社長 | 非アクティブ警告（週次） |

## 8. エッジケース・エラー処理

| ケース | 対処 |
|--------|------|
| 社長が自分のロールを変更しようとした | 「社長のロールは変更できません」と返答 |
| 最後の管理者をスタッフに降格しようとした | 「管理者が0名になるため変更できません」と警告 |
| 同じ表示名のスタッフが既に存在する | 「同じ名前のスタッフがいます。別の名前を設定してください（例：田中A）」 |
| LINEの表示名が変更された | line_display_nameは毎回更新するが、display_nameは変更しない |
| 無効化されたユーザーが再度メッセージを送った | 社長に「無効化されたユーザーからメッセージがあります。再有効化しますか？」 |
| 社長のLINEアカウントが変わった | 初期セットアップが再度走る。既存データは保持 |
| 複数の社長が存在する状況 | 防止策：ownerロールは1名のみに制限 |

## 9. 開発優先度・依存関係

### 9.1 依存関係
- **依存元**: 01_システム基盤（LINE Bot + Supabase接続が必要）
- **依存先**: 全機能（ユーザー識別は全機能の前提）

### 9.2 実装順序
```
1. usersテーブルの作成
2. LINE User IDからのユーザー検索
3. 初回セットアップ（社長登録）
4. 新規ユーザー検知→社長への通知
5. ロール割り当てフロー
6. ロールに応じたアクセス制御
7. スタッフ一覧・ロール変更
```
