# 10. 問い合わせ・見学対応

## 1. 概要

### 1.1 この機能が解決する課題

就労継続支援事業所（A型/B型）では、利用者の獲得が事業運営の根幹に関わる。見学希望や利用相談の問い合わせは、電話・相談支援事業所からの紹介・Webフォームなど複数経路から入ってくるが、以下の課題が発生しやすい。

- **問い合わせの取りこぼし**: スタッフが電話で受けた問い合わせを口頭で社長に伝えたが、忙しくて忘れてしまう
- **フォロー漏れ**: 見学後に「前向きに検討します」と言っていた方へのフォロー連絡が抜ける
- **パイプラインの不可視**: 現在何件の問い合わせがあり、そのうち何件が見学済みで、何件が利用開始に至ったかが把握できない
- **定員管理の混乱**: A型・B型それぞれの定員に対して空きが何名あるか、リアルタイムで把握できない
- **紹介元への報告不備**: 相談支援事業所から紹介を受けた利用者の進捗を報告し忘れ、信頼関係を損ねる

### 1.2 提供する価値

- LINEでスタッフが「見学の問い合わせがあった」と一言送るだけで、AIが必要情報をヒアリングし構造化して記録する
- 問い合わせから利用開始までの全プロセスをパイプラインとして可視化し、どの段階に何件あるかを一目で確認できる
- 見学前日のリマインド、フォロー期限の通知、定員到達アラートなど、適切なタイミングで自動通知する
- 紹介元別・月別の統計を自動集計し、営業活動の効果測定を支援する
- 定員管理と連動し、空きがない場合はウェイティングリストへ自動振り分けする

### 1.3 パイプライン全体像

```
問い合わせ受付 → 見学日程調整 → 見学実施 → 検討期間 → 利用開始 or 辞退
  (inquiry)     (visit_scheduled)  (visited)  (considering)  (started / declined)
                                                              (waitlist: 定員満時)
```

### 1.4 問い合わせ経路

| 経路 | referral_type | 具体例 |
|------|--------------|--------|
| 相談支援事業所からの紹介 | `agency` | ○○相談支援センター、△△基幹相談支援センター |
| 本人・家族からの直接連絡 | `self` | 電話、来所 |
| Web経由 | `web` | ホームページ問い合わせフォーム、SNS経由 |
| その他 | `other` | ハローワーク、病院、他事業所からの紹介 |

## 2. ユーザーストーリー

### 2.1 スタッフ：問い合わせの記録

```
スタッフとして、
「見学の問い合わせがあった」とLINEで送ったら、
AIが必要情報（名前、連絡先、紹介元、希望日）を順番に聞いてくれて、
記録してくれたうえで社長に自動通知してほしい。
```

**背景**: 電話対応中にメモを取り忘れたり、伝言を忘れるリスクを排除する。対話形式で情報を収集するため、スタッフは最低限の情報だけ分かっていれば記録できる。

### 2.2 スタッフ：見学当日の準備確認

```
スタッフとして、
見学予定日の前日に「明日14時に木村さんの見学があります。
準備をお願いします」とリマインドを受けたい。
```

**背景**: 見学対応は第一印象に直結する。資料の準備、事業所内の清掃、対応スタッフの確認など、事前準備が必要。

### 2.3 管理者：見学実施後の記録

```
管理者として、
見学が終わった後に「木村さんの見学が終わった。前向きに検討中」と送ったら、
ステータスが自動更新され、フォロー日が設定されてほしい。
```

**背景**: 見学後の温度感を記録しておくことで、適切なタイミングでフォロー連絡ができる。

### 2.4 社長：問い合わせ一覧の確認

```
社長として、
「最近の問い合わせは？」と聞いたら、
直近1ヶ月の問い合わせ件数とステータス別内訳が見たい。
「見学予定の人は？」と聞けばリストが出てほしい。
```

**背景**: 経営者として、営業パイプラインの状況をリアルタイムで把握し、定員到達の見通しを立てたい。

### 2.5 社長：定員状況の確認

```
社長として、
「定員まであと何名？」と聞いたら、
A型・B型それぞれの現在利用者数、定員、空き枠が
一目で分かるように表示してほしい。
```

**背景**: 定員に近づいた場合、新規受入の判断や相談支援事業所への空き状況連絡が必要になる。

### 2.6 社長：利用開始の確定と受入準備

```
社長として、
「木村さん来月から利用開始」と送ったら、
利用者マスタへの登録と受入準備タスクの自動生成をしてほしい。
```

**背景**: 利用開始までに契約書類の準備、個別支援計画の作成、送迎ルートの調整など、多数のタスクが発生する。漏れなく対応するために自動化したい。

### 2.7 社長：紹介元への報告管理

```
社長として、
相談支援事業所から紹介された方が利用開始に至ったとき、
「○○相談支援センターに利用開始の報告を」とリマインドしてほしい。
```

**背景**: 紹介元への報告は、継続的な紹介獲得のために重要な営業活動の一環。

## 4. 機能要件

### 4.1 必須機能（MVP）

| ID | 機能 | 説明 | 優先度 |
|----|------|------|--------|
| INQ-001 | 問い合わせ記録 | 対話形式で名前・連絡先・紹介元・希望日を収集し構造化して保存 | P0 |
| INQ-002 | 見学予約管理 | 見学日時の確定、カレンダー登録、リマインド設定 | P0 |
| INQ-003 | ステータス管理 | inquiry → visit_scheduled → visited → considering → started/declined のパイプライン管理 | P0 |
| INQ-004 | 定員チェック | A型/B型の現在利用者数と定員を比較し、空き状況を表示 | P0 |
| INQ-005 | 社長への即時通知 | 新規問い合わせ時に社長のLINEへ即時通知 | P0 |
| INQ-006 | 見学前日リマインド | 見学予定日の前日に担当者へリマインド | P0 |
| INQ-007 | フォロー管理 | follow_up_dateに基づくリマインド通知 | P0 |
| INQ-008 | 一覧表示 | ステータス別の問い合わせ一覧表示 | P0 |
| INQ-009 | 利用開始処理 | ステータス変更→clientsテーブル登録→受入準備タスク生成 | P0 |
| INQ-010 | 重複検知 | 名前・ふりがなで既存レコードを検索し重複を防止 | P0 |

```sql
-- ※ アプリケーション層で実装するか、以下のトリガーで対応
CREATE OR REPLACE FUNCTION log_inquiry_status_change()
RETURNS TRIGGER AS $$
BEGIN
    IF OLD.status IS DISTINCT FROM NEW.status THEN
        NEW.notes = NEW.notes || jsonb_build_array(jsonb_build_object(
            'date', to_char(NOW(), 'YYYY-MM-DD'),
            'type', 'status_change',
            'from', OLD.status,
            'to', NEW.status,
            'timestamp', NOW()
        ));
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_log_inquiry_status_change
    BEFORE UPDATE ON inquiries
    FOR EACH ROW
```

```python
        if referral_type == 'agency':
            state.current_step = 'referral_name'

        else:
            state.current_step = 'visit_date'
            return "見学の希望日はありますか？"

            # 状態クリア
            await clear_conversation_state(user_id)

            return "最初からやり直しますか？修正したい項目を教えてください。"
```

#### 6.2.2 ステータス更新処理

```python
VALID_TRANSITIONS = {
    'inquiry':         ['visit_scheduled', 'cancelled', 'declined', 'waitlist'],
    'visit_scheduled': ['visited', 'cancelled', 'inquiry'],  # inquiry: 日程未定に戻す
    'visited':         ['considering', 'started', 'declined'],
    'considering':     ['started', 'declined'],
    'waitlist':        ['visit_scheduled', 'cancelled', 'declined'],
    'started':         [],  # 終了状態
    'declined':        ['inquiry'],  # 再問い合わせで復活可
    'cancelled':       ['inquiry', 'visit_scheduled'],  # 再調整可
}

async def update_inquiry_status(inquiry_id, new_status, user_id, note=None):
    """ステータスを更新し、関連処理を実行する"""

    inquiry = await get_inquiry(inquiry_id)
    old_status = inquiry.status

    # 遷移の妥当性チェック
    if new_status not in VALID_TRANSITIONS.get(old_status, []):
        raise InvalidTransitionError(
            f"'{old_status}' から '{new_status}' への遷移は許可されていません"
        )

    # ステータス更新
    await db.execute("""
        UPDATE inquiries
        SET status = $1, updated_at = NOW()
        WHERE id = $2
    """, new_status, inquiry_id)

    # 履歴記録
    await db.execute("""
        INSERT INTO inquiry_status_history
            (inquiry_id, old_status, new_status, changed_by, note)
        VALUES ($1, $2, $3, $4, $5)
    """, inquiry_id, old_status, new_status, user_id, note)

    # ステータス別の後続処理
    if new_status == 'visit_scheduled':
        await schedule_visit_reminders(inquiry)

    elif new_status == 'visited':
        await db.execute("""
            UPDATE inquiries
            SET visit_completed_date = NOW()
            WHERE id = $1
        """, inquiry_id)

    elif new_status == 'considering':
        # デフォルトのフォロー日を1週間後に設定
        if not inquiry.follow_up_date:
            await db.execute("""
                UPDATE inquiries
                SET follow_up_date = CURRENT_DATE + INTERVAL '7 days'
                WHERE id = $1
            """, inquiry_id)

    elif new_status == 'started':
        await handle_start_process(inquiry)

    elif new_status == 'declined':
        await handle_decline_process(inquiry)
```

### 6.3 定員チェックロジック

```python
async def check_capacity(service_type: str) -> dict:
    """定員状況をチェックし、空き情報を返す"""

    result = await db.fetchrow("""
        SELECT
            service_type,
            max_capacity,
            current_count,
            waitlist_count,
            max_capacity - current_count AS available
        FROM capacity_settings
        WHERE service_type = $1
    """, service_type)

    return {
        'service_type': result['service_type'],
        'max_capacity': result['max_capacity'],
        'current_count': result['current_count'],
        'available': result['available'],
        'waitlist_count': result['waitlist_count'],
        'is_full': result['available'] <= 0,
        'is_near_full': result['available'] <= 2,  # 残り2名以下で警告
    }

async def handle_start_process(inquiry):
    """利用開始時の処理"""

    service_type = inquiry.desired_service_type
    capacity = await check_capacity(service_type)

    if capacity['is_full']:
        raise CapacityFullError(
            f"{service_type}型は定員に達しています（{capacity['current_count']}/{capacity['max_capacity']}）"
        )

    # 1. clients テーブルに利用者を登録
    client_id = await create_client_from_inquiry(inquiry)

    # 2. 定員カウントを更新
    await db.execute("""
        UPDATE capacity_settings
        SET current_count = current_count + 1,
            updated_at = NOW()
        WHERE service_type = $1
    """, service_type)

    # 3. inquiries のステータスを更新
    await db.execute("""
        UPDATE inquiries
        SET status = 'started',
            start_date = $1
        WHERE id = $2
    """, inquiry.start_date, inquiry.id)

    # 4. 受入準備タスクを生成（04_タスク自動生成と連携）
    await generate_onboarding_tasks(inquiry, client_id)

    # 5. 定員アラート（残り2名以下）
    updated_capacity = await check_capacity(service_type)
    if updated_capacity['is_near_full']:
        await notify_owner_capacity_warning(updated_capacity)

    # 6. 紹介元が相談支援事業所の場合、報告リマインドを設定
    if inquiry.referral_type == 'agency':
        await schedule_referral_report_reminder(inquiry)
```

### 8.2 定員超過した場合のウェイティングリスト

**シナリオ**: B型が定員20名に達した状態で新規問い合わせが入った場合。

**処理方針**:
1. 問い合わせ記録時に `desired_service_type` と `capacity_settings` を照合
2. 定員に達している場合、スタッフにその旨を通知
3. ウェイティングリスト扱いで記録するか、社長に判断を仰ぐか選択させる
4. ウェイティングリストのステータスは `waitlist`
5. 空きが発生した場合（退所・利用終了時）、ウェイティングリストの最古の待機者に通知

### 8.4 紹介元の表記ゆれ

**シナリオ**: 同じ相談支援事業所が「さくら相談支援センター」「さくら相談支援事業所」「さくら相談」など異なる名称で入力される。

**処理方針**:
1. 入力された紹介元名を `referral_sources` テーブルの `aliases` 配列と照合
2. 類似度が高い既存レコードがあれば提示して確認
3. 確認が得られたら正式名称に置換して保存
4. 新しい表記の場合は `aliases` に追加

```python
async def check_referral_name_match(input_name: str) -> dict:
    """紹介元名の表記ゆれをチェック"""

    # 1. 完全一致チェック
    exact = await db.fetchrow("""
        SELECT id, name FROM referral_sources
        WHERE name = $1 OR $1 = ANY(aliases)
    """, input_name)

    if exact:
        return exact

    # 2. 部分一致チェック（前方一致 + 類似度）
    partial = await db.fetch("""
        SELECT id, name, aliases,
               similarity(name, $1) AS sim
        FROM referral_sources
        WHERE similarity(name, $1) > 0.3
           OR name LIKE '%' || $1 || '%'
           OR $1 LIKE '%' || name || '%'
        ORDER BY sim DESC
        LIMIT 3
    """, input_name)

    if partial:
        return partial  # 候補リストを返し、ユーザーに選択させる

    return None
```

### 10.3 見学→利用開始の平均期間分析

問い合わせから利用開始までの各段階の平均所要日数を分析し、営業プロセスの改善に活用する。

```sql
-- 各段階の平均所要日数を算出
SELECT
    -- 問い合わせ → 見学までの平均日数
    AVG(
        EXTRACT(DAY FROM visit_completed_date - inquiry_date::timestamp)
    ) FILTER (WHERE visit_completed_date IS NOT NULL)
        AS avg_days_inquiry_to_visit,

    -- 見学 → 利用開始までの平均日数
    AVG(
        start_date - DATE(visit_completed_date)
    ) FILTER (WHERE start_date IS NOT NULL AND visit_completed_date IS NOT NULL)
        AS avg_days_visit_to_start,

    -- 問い合わせ → 利用開始までの平均日数（全体）
    AVG(
        start_date - inquiry_date
    ) FILTER (WHERE start_date IS NOT NULL)
        AS avg_days_total,

    -- コンバージョン率
    ROUND(
        COUNT(*) FILTER (WHERE visit_completed_date IS NOT NULL)::numeric /
        NULLIF(COUNT(*), 0) * 100, 1
    ) AS inquiry_to_visit_rate,

    ROUND(
        COUNT(*) FILTER (WHERE status = 'started')::numeric /
        NULLIF(COUNT(*) FILTER (WHERE visit_completed_date IS NOT NULL), 0) * 100, 1
    ) AS visit_to_start_rate

FROM inquiries
WHERE inquiry_date >= CURRENT_DATE - INTERVAL '6 months';
```

### 10.4 紹介元別の成績レポート

どの相談支援事業所からの紹介が利用開始につながりやすいかを分析する。

```sql
-- 紹介元別の成績
SELECT
    referral_source AS "紹介元",
    COUNT(*) AS "問い合わせ数",
    COUNT(*) FILTER (WHERE visit_completed_date IS NOT NULL) AS "見学実施数",
    COUNT(*) FILTER (WHERE status = 'started') AS "利用開始数",
    COUNT(*) FILTER (WHERE status = 'declined') AS "辞退数",
    ROUND(
        COUNT(*) FILTER (WHERE status = 'started')::numeric /
        NULLIF(COUNT(*), 0) * 100, 1
    ) AS "成約率(%)"
FROM inquiries
WHERE referral_type = 'agency'
  AND inquiry_date >= CURRENT_DATE - INTERVAL '1 year'
GROUP BY referral_source
ORDER BY "利用開始数" DESC;
```
