# 07. 個別支援計画

## 1. 概要

### 1.1 個別支援計画とは

個別支援計画は、就労継続支援事業所（A型/B型）において、利用者一人ひとりの支援目標・支援方法・達成基準を定めた計画書である。障害者総合支援法に基づき、サービス管理責任者（以下「サビ管」）が作成し、**6ヶ月ごとの見直し（モニタリング）** が義務付けられている。

行政監査（指導監査・実地指導）において **必ずチェックされる最重要書類** の一つであり、以下の点が確認される：

- 全利用者に対して個別支援計画が作成されているか
- 計画の見直し（モニタリング）が6ヶ月以内に実施されているか
- モニタリング結果に基づき計画が適切に更新されているか
- 利用者本人・家族への説明と同意が得られているか
- アセスメント（初回評価）が適切に実施されているか

### 1.2 現状の課題

| 課題 | 詳細 | リスク |
|------|------|--------|
| 期限管理が属人的 | Excelや紙の台帳で見直し期限を管理しており、サビ管の記憶に依存 | 見直し期限の失念 → 監査指摘 |
| モニタリング記録の散逸 | 面談記録が紙のノートやメモに分散しており、後から参照しづらい | 記録不備 → 監査指摘 |
| 経過把握に時間がかかる | 出席状況・日報・面談記録を個別に確認する必要がある | 支援品質の低下 |
| 計画更新の負担 | 紙で計画書を作り直す工数が大きく、更新が後回しになりがち | 形骸化した計画 |
| 共有不足 | サビ管以外のスタッフが利用者の支援目標を把握していない | チーム支援の機能不全 |

### 1.3 AI秘書による解決

本機能では以下を実現する：

1. **期限の自動管理**: 全利用者の支援計画見直し期限をデータベースで管理し、段階的にリマインド通知を送信する
2. **モニタリング記録のLINE入力**: サビ管がLINEで自然言語で面談内容を送信すると、AIが構造化して記録する
3. **経過の統合表示**: 支援計画の目標に対して、出席率・日報記録・モニタリング記録を横断的に集約して表示する
4. **期限超過の即時検知**: 見直し期限を超過した計画を検知し、社長に毎日通知する

### 1.4 個別支援計画のライフサイクル

```
利用開始
  │
  ▼
[アセスメント] ← 利用開始後30日以内に実施
  │
  ▼
[個別支援計画 作成] ← サビ管が作成、利用者・家族に説明・同意
  │
  ▼
[支援の実施] ← 計画に基づく日々の支援
  │
  ├──→ [日報記録] ── 毎日のフィードバック
  ├──→ [出席管理] ── 出席率の推移
  │
  ▼
[モニタリング] ← 6ヶ月ごと（中間モニタリング + 見直し）
  │
  ├── 目標達成 → 新たな目標設定
  ├── 一部達成 → 目標・方法の調整
  └── 未達成 → 原因分析・計画の見直し
  │
  ▼
[個別支援計画 更新] ← 新しい計画期（第2期、第3期...）
  │
  ▼
  ... (繰り返し)
  │
  ▼
[退所] → 計画のクローズ
```

## 2. ユーザーストーリー

### 2.1 管理者（サビ管）：支援計画の経過確認

**ストーリー**: サビ管として、利用者の支援計画の進捗を素早く把握したい。面談前の準備として、出席率・作業状況・課題を一目で確認したい。

**受け入れ条件**:
- 利用者名を指定すると、現在有効な支援計画の内容が表示される
- 短期目標・長期目標の進捗状況が表示される
- 直近の出席率（週次・月次）が表示される
- 日報で言及されたその利用者に関する記録が時系列で表示される
- 直近のモニタリング記録があれば表示される

**会話例**:
```
管理者: 佐藤さんの支援計画の経過教えて

AI秘書: 佐藤太郎さんの支援計画（第3期）の経過をお伝えします。

■ 計画概要
  期間: 2025/10/01 〜 2026/03/31
  見直し期限: 2026/03/31（残り1日）⚠️
  担当: 田中管理者

■ 目標と進捗
  【短期目標】封入作業のミス率を10%以下にする
    → 直近1ヶ月のミス率: 3%（達成）
  【長期目標】一般就労に向けた基礎体力・集中力の向上
    → 継続的に取り組み中

■ 出席状況
  今月出席率: 72%（目標80%）⚠️
  先月出席率: 85%
  欠席理由: 体調不良3日、通院2日

■ 日報での記録（直近）
  3/25: 「封入作業、正確性が安定してきた」(鈴木スタッフ)
  3/20: 「午後から集中力低下、休憩を多めに取った」(田中管理者)
  3/18: 「新しい作業（ラベル貼り）にも意欲的」(鈴木スタッフ)

■ 前回モニタリング（2025/12/15）
  封入作業のミス率改善を確認（8%→5%）
  出席率は安定（85%前後）
  次の目標として作業種類の拡大を検討

⚠️ 見直し期限が明日です。モニタリング面談の実施をお勧めします。
```

### 2.2 管理者（サビ管）：モニタリング記録の入力

**ストーリー**: サビ管として、利用者との面談後にモニタリング記録をLINEで手軽に入力したい。紙に書いてから転記する手間を省きたい。

**受け入れ条件**:
- 自然言語でモニタリング内容を送信できる
- AIが構造化（進捗・課題・次のアクション）して整形する
- 整形結果を確認後、保存できる
- 出席率は自動計算されて記録に含まれる

**会話例**:
```
管理者: 佐藤さんのモニタリング。封入作業の正確性は改善、ミス率8%→3%。
        出席率が課題、今月72%で目標に届いていない。体調管理が必要。
        次は作業の幅を広げてラベル貼りにも挑戦させたい。

AI秘書: 佐藤太郎さんのモニタリング記録を整理しました。

■ モニタリング日: 2026/03/30
■ 対象計画: 第3期（2025/10/01〜2026/03/31）

【目標の達成状況】
  短期目標「封入作業のミス率を10%以下にする」
  → 達成: ミス率 8% → 3% に改善

【出席状況】
  今月出席率: 72%（目標80%に未達）
  → 課題: 体調管理の支援が必要

【課題・気づき】
  ・出席率が目標を下回っている
  ・体調管理（特に生活リズム）のサポートが必要

【次期に向けたアクション】
  ・作業種類の拡大（ラベル貼り作業への挑戦）
  ・出席率向上のための体調管理支援を計画に盛り込む

この内容で保存してよろしいですか？
```

```
管理者: はい

AI秘書: モニタリング記録を保存しました。
        佐藤太郎さんの支援計画（第3期）の見直し期限は明日（3/31）です。
        新しい計画（第4期）の作成が必要です。作成を開始しますか？
```

### 2.3 社長：今月期限の支援計画一覧

**ストーリー**: 社長として、支援計画の見直し期限が近い利用者を一覧で把握したい。監査前に漏れがないかチェックしたい。

**受け入れ条件**:
- 見直し期限が近い順に一覧表示される
- 期限超過・期限間近・余裕ありの3段階で表示される
- 担当管理者も表示される

**会話例**:
```
社長: 今月期限の支援計画は？

AI秘書: 今月（2026年3月）に見直し期限を迎える支援計画の一覧です。

🔴 期限超過
  なし

🟡 期限間近（7日以内）
  1. 佐藤太郎さん - 期限: 3/31（明日）担当: 田中
     → モニタリング未実施
  2. 山田花子さん - 期限: 4/2（3日後）担当: 田中
     → モニタリング実施済み、計画更新待ち

🟢 今月中（8日〜30日以内）
  3. 鈴木一郎さん - 期限: 4/15（16日後）担当: 佐々木
     → モニタリング未実施
  4. 高橋美咲さん - 期限: 4/25（26日後）担当: 田中
     → モニタリング未実施

合計: 4名（うち要対応: 2名）

佐藤さんのモニタリングが未実施です。至急対応をお勧めします。
```

### 2.4 リマインド通知

**ストーリー**: システムとして、見直し期限が近づいた支援計画を担当者・社長に段階的に通知したい。人間が忘れても、システムが確実にリマインドする。

**受け入れ条件**:
- 30日前: 担当管理者に通知
- 14日前: 担当管理者 + 社長に通知
- 7日前: 担当管理者 + 社長に緊急通知
- 期限超過: 社長に毎日通知（対応完了まで）

**通知例（30日前）**:
```
[リマインド] 支援計画の見直し期限が近づいています

佐藤太郎さんの支援計画（第3期）
見直し期限: 2026/03/31（あと30日）

モニタリング面談の日程調整をお願いします。
```

**通知例（7日前・緊急）**:
```
[緊急] 支援計画の見直し期限まで7日です

佐藤太郎さんの支援計画（第3期）
見直し期限: 2026/03/31（あと7日）

⚠️ モニタリングが未実施です。
至急、面談の実施と計画の更新をお願いします。

※ 期限を過ぎると行政監査で指摘対象となります。
```

**通知例（期限超過）**:
```
[警告] 支援計画の見直し期限を超過しています

佐藤太郎さんの支援計画（第3期）
見直し期限: 2026/03/31（3日超過）

❗ 速やかにモニタリングを実施し、計画を更新してください。
   行政監査で指摘対象となる状態です。

担当: 田中管理者
```

### 2.5 新規利用者のアセスメント記録

**ストーリー**: サビ管として、新規利用者のアセスメント（初回評価）をLINEで記録したい。利用開始後30日以内にアセスメントを完了し、個別支援計画を作成する必要がある。

**会話例**:
```
管理者: 新しく入った木村さんのアセスメント結果を記録したい

AI秘書: 木村健太さん（利用開始日: 2026/03/15）のアセスメント記録を開始します。
        アセスメント期限: 2026/04/14（あと15日）

        以下の項目について教えてください。
        まとめて送っていただいても、一つずつでも大丈夫です。

        1. 本人の希望・意向
        2. 得意なこと・できること
        3. 苦手なこと・課題
        4. 推奨する作業内容
        5. 健康面・配慮事項

管理者: 本人は将来パソコンを使った仕事がしたいと言っている。
        手先は器用で封入作業は上手。集中力は30分程度。
        人と話すのが苦手で声かけに注意が必要。
        まずは封入作業から始めて、慣れたらデータ入力に挑戦。
        てんかんの既往歴あり、服薬中。疲労時に注意。

AI秘書: 木村健太さんのアセスメント記録を整理しました。

■ アセスメント日: 2026/03/30
■ 評価者: 田中管理者

【本人の希望・意向】
  ・将来的にパソコンを使った仕事に就きたい

【能力・得意なこと】
  ・手先が器用（封入作業が上手）

【課題・苦手なこと】
  ・集中力の持続が約30分程度
  ・対人コミュニケーションが苦手（声かけの方法に配慮が必要）

【推奨する作業内容】
  ・第1段階: 封入作業（得意分野から開始）
  ・第2段階: データ入力作業（本人の希望に沿ったステップアップ）

【健康面・配慮事項】
  ・てんかんの既往歴あり（服薬中）
  ・疲労時に発作リスクが高まるため、休憩を適切に確保

この内容で保存してよろしいですか？
保存後、個別支援計画（第1期）の作成に進めます。
```

## 3. 会話フロー（詳細）

### 3.1 正常系: 支援計画の経過確認

```
[トリガー]
  管理者が「〇〇さんの支援計画」「〇〇の計画の経過」等を送信

[処理フロー]
  1. 利用者名の特定（名前の曖昧検索）
  2. 該当利用者の有効な支援計画を取得（status = 'active'）
  3. 以下のデータを並行取得:
     a. support_plans テーブルから目標・計画内容
     b. monitoring_records テーブルから直近のモニタリング
     c. attendance_records テーブルから出席率を集計
     d. daily_reports テーブルから該当利用者に関する記録を抽出
  4. データを統合してレスポンスを構築
  5. 見直し期限が近い場合は警告を付加

[応答構成]
  ├── 計画概要（期間・期・担当・期限）
  ├── 目標と進捗（短期・長期）
  ├── 出席状況（今月・先月・欠席理由）
  ├── 日報での記録（直近3〜5件）
  ├── 前回モニタリング要約
  └── アラート（期限警告等）

[名前の曖昧検索ルール]
  - 「佐藤さん」→ 姓が「佐藤」の利用者を検索
  - 複数該当 → 「佐藤太郎さんと佐藤次郎さんがいます。どちらですか？」
  - 該当なし → 「佐藤さんという利用者が見つかりません。名前を確認してください」
```

### 3.2 正常系: モニタリング記録の入力

```
[トリガー]
  管理者が「〇〇さんのモニタリング」「〇〇のモニタリング記録」等を送信

[処理フロー]
  1. 利用者名の特定
  2. 有効な支援計画の存在確認
  3. 自然言語の入力内容を解析
     - Claude APIで以下を抽出:
       a. 目標ごとの達成状況（達成/一部達成/未達成 + 具体的な数値）
       b. 出席に関する情報
       c. 課題・気づき
       d. 次期に向けたアクション
  4. 出席率を attendance_records から自動算出して補完
  5. 構造化した内容を確認用に表示
  6. 「はい」で保存 / 「修正」で再編集

[AIによる構造化プロンプト]
  以下のモニタリング内容を構造化してください。

  対象利用者: {client_name}
  現在の支援計画:
    短期目標: {short_term_goal}
    長期目標: {long_term_goal}

  管理者の入力:
  {raw_input}

  以下の形式でJSON出力してください:
  {
    "goal_progress": [
      {
        "goal_type": "short_term",
        "status": "achieved|partial|not_achieved",
```

```
[トリガー]
  社長が「支援計画の期限」「今月の支援計画」「計画の一覧」等を送信

[処理フロー]
  1. ロール確認（社長 or 管理者）
  2. support_plans テーブルから status='active' の全計画を取得
  3. review_date との差分を計算
  4. 以下の区分で分類:
     a. 期限超過（review_date < 今日）
     b. 7日以内
     c. 14日以内
     d. 30日以内
     e. それ以降
  5. モニタリング実施状況を付加
  6. 担当管理者を表示

[応答構成]
  ├── 期限超過（赤）: 件数 + 利用者名 + 超過日数 + 担当
  ├── 7日以内（黄）: 件数 + 利用者名 + 残り日数 + 担当 + モニタリング状況
  ├── 30日以内（緑）: 件数 + 利用者名 + 残り日数 + 担当
  └── 合計・要対応数のサマリー
```

### 3.6 異常系: 期限を過ぎてしまった場合

```
[トリガー]
  Cronジョブで期限超過を検知 / 管理者が期限超過の計画にアクセス

[処理フロー]
  1. support_plans.review_date < CURRENT_DATE かつ status = 'active' を検出
  2. status を 'expired' に更新
  3. 社長に通知（毎日繰り返し）
  4. 管理者がモニタリングを入力した場合:
     a. 遅延であることを記録に残す
     b. 「見直し期限を超過していますが、モニタリング記録を保存します」
     c. 新計画作成を強く推奨

[会話例]
管理者: 佐藤さんの支援計画の経過教えて

AI秘書: ⚠️ 佐藤太郎さんの支援計画（第3期）は見直し期限を
        3日超過しています（期限: 2026/03/31）。

        至急モニタリングを実施し、新しい計画を作成してください。
        行政監査で指摘対象となる状態です。

        モニタリング記録の入力を開始しますか？
```

### 3.7 異常系: 利用者が退所した場合の支援計画の扱い

```
[トリガー]
  利用者のステータスが「退所」に変更された場合

[処理フロー]
  1. clients テーブルの status が 'inactive' に変更されたことを検知
  2. 該当利用者の有効な支援計画（status = 'active'）を検索
  3. 支援計画の status を 'closed' に更新
  4. 最終モニタリングの有無を確認
     - 未実施 → 管理者に「退所時の最終モニタリング記録を入力しますか？」
  5. リマインドの停止（該当利用者の通知を無効化）

[会話例]
管理者: 佐藤さんが今月末で退所します

AI秘書: 佐藤太郎さんの退所処理について確認します。

        1. 現在有効な支援計画（第3期）をクローズします
        2. 退所時の最終モニタリング記録を入力しますか？
           ※ 行政上、退所時の記録を残すことが推奨されます
        3. 今後のリマインド通知を停止します

        退所日はいつですか？

管理者: 3月31日です。最終モニタリングも記録したい

AI秘書: 承知しました。退所日を2026/03/31で設定します。
        最終モニタリングの内容を教えてください。
```

### 5.2 support_plans テーブル

個別支援計画の本体。利用者ごとに期（plan_number）を持ち、6ヶ月ごとに新しい計画が作成される。

```sql
CREATE TABLE support_plans (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    -- 利用者との関連
    client_id UUID NOT NULL REFERENCES clients(id) ON DELETE RESTRICT,

    -- 計画の基本情報
    plan_number INTEGER NOT NULL,  -- 第何期（1, 2, 3, ...）

    -- 目標
    goal_short_term TEXT NOT NULL,       -- 短期目標（6ヶ月で達成を目指す具体的な目標）
    goal_long_term TEXT NOT NULL,        -- 長期目標（1〜2年の方向性）

    -- 支援内容（JSONB形式で柔軟に格納）
    support_content JSONB NOT NULL DEFAULT '{}',
    -- 格納例:
    -- {
    --   "tasks": [
    --     {"name": "封入作業", "frequency": "毎日", "goal": "ミス率10%以下"},
    --     {"name": "ラベル貼り", "frequency": "週3回", "goal": "手順を覚える"}
    --   ],
    --   "daily_support": [
    --     "作業開始時に手順を一緒に確認する",
    --     "30分ごとに休憩を促す"
    --   ],
    --   "life_support": [
    --     "生活リズム表の記入を毎朝確認",
    --     "体調不良時は無理をしないよう声かけ"
    --   ],
    --   "considerations": [
    --     "てんかんの既往歴あり、疲労時に注意",
    --     "大きな声での指示は避ける"
    --   ]
    -- }

    -- 期間
    start_date DATE NOT NULL,            -- 計画開始日
    end_date DATE NOT NULL,              -- 計画終了日（通常は開始日 + 6ヶ月）
    review_date DATE NOT NULL,           -- 見直し期限（= end_date と同日が基本）

    -- ステータス
    status VARCHAR(20) NOT NULL DEFAULT 'draft'
        CHECK (status IN ('draft', 'active', 'expired', 'reviewed', 'closed')),
    -- draft: 下書き（作成中）
    -- active: 有効（現在の計画）
    -- expired: 期限超過（見直しが行われていない）
    -- reviewed: 見直し済み（次の計画に移行）
    -- closed: 終了（退所等）

    -- 同意関連
    client_consent BOOLEAN DEFAULT FALSE,       -- 本人同意
    family_consent BOOLEAN DEFAULT FALSE,       -- 家族同意
    consent_date DATE,                          -- 同意取得日

    -- 担当者
    created_by UUID NOT NULL REFERENCES users(id),  -- 作成者（サビ管）
    assigned_to UUID REFERENCES users(id),           -- 担当管理者

    -- タイムスタンプ
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    -- 制約
    UNIQUE(client_id, plan_number)  -- 同一利用者の同一期の計画は1つだけ
);

-- インデックス
CREATE INDEX idx_support_plans_client_id ON support_plans(client_id);
CREATE INDEX idx_support_plans_status ON support_plans(status);
CREATE INDEX idx_support_plans_review_date ON support_plans(review_date);
CREATE INDEX idx_support_plans_assigned_to ON support_plans(assigned_to);
CREATE INDEX idx_support_plans_status_review ON support_plans(status, review_date)
    WHERE status = 'active';

-- 更新日時の自動更新トリガー
CREATE TRIGGER update_support_plans_updated_at
    BEFORE UPDATE ON support_plans
    FOR EACH ROW
    EXECUTE FUNCTION update_updated_at_column();

-- コメント
COMMENT ON TABLE support_plans IS '個別支援計画。利用者ごとに6ヶ月単位で作成される支援の計画書';
COMMENT ON COLUMN support_plans.plan_number IS '計画の期数。第1期=1, 第2期=2, ...';
COMMENT ON COLUMN support_plans.support_content IS '支援内容をJSONBで格納。tasks, daily_support, life_support, considerations等';
COMMENT ON COLUMN support_plans.review_date IS '見直し期限。この日までにモニタリングを実施し計画を更新する必要がある';
COMMENT ON COLUMN support_plans.status IS 'draft=下書き, active=有効, expired=期限超過, reviewed=見直し済, closed=終了';
```

```sql
    -- {
    --   "goal_progress": [
    --     {
    --       "goal_type": "short_term",
    --       "goal_text": "封入作業のミス率を10%以下にする",
```

### 5.6 RLS（Row Level Security）ポリシー

```sql
-- support_plans のRLS
ALTER TABLE support_plans ENABLE ROW LEVEL SECURITY;

-- 社長は全ての計画を閲覧可能
CREATE POLICY support_plans_owner_read ON support_plans
    FOR SELECT
    USING (
        EXISTS (
            SELECT 1 FROM users
            WHERE users.id = auth.uid()
            AND users.role = 'owner'
        )
    );

-- 管理者は自分が担当する計画と自分が作成した計画を閲覧可能
CREATE POLICY support_plans_manager_read ON support_plans
    FOR SELECT
    USING (
        EXISTS (
            SELECT 1 FROM users
            WHERE users.id = auth.uid()
            AND users.role = 'manager'
        )
        AND (assigned_to = auth.uid() OR created_by = auth.uid())
    );

-- 管理者は計画の作成・更新が可能
CREATE POLICY support_plans_manager_write ON support_plans
    FOR ALL
    USING (
        EXISTS (
            SELECT 1 FROM users
            WHERE users.id = auth.uid()
            AND users.role IN ('owner', 'manager')
        )
    );
```

## 6. API/処理フロー

### 6.1 支援計画経過の統合表示

利用者名を指定すると、複数テーブルから情報を横断集約して経過を表示する。

```
[リクエスト]
  POST /api/support-plans/progress
  {
    "client_name": "佐藤",
    "user_id": "管理者のUUID"
  }

[処理フロー]

  Step 1: 利用者の特定
  ┌──────────────────────────────────────────┐
  │ SELECT id, full_name                      │
  │ FROM clients                              │
  │ WHERE full_name LIKE '%佐藤%'             │
  │   AND status = 'active'                   │
  └──────────────────────────────────────────┘
  → 複数該当: 候補一覧を返却して選択を求める
  → 0件: エラー「該当する利用者が見つかりません」

  Step 2: 有効な支援計画の取得
  ┌──────────────────────────────────────────┐
  │ SELECT *                                  │
  │ FROM support_plans                        │
  │ WHERE client_id = $1                      │
  │   AND status = 'active'                   │
  │ ORDER BY plan_number DESC                 │
  │ LIMIT 1                                   │
  └──────────────────────────────────────────┘

  Step 3: 並行データ取得（Promise.all）
  ┌─────────────────────────────────────────────────────┐
  │ [A] 直近のモニタリング記録                              │
  │ SELECT * FROM monitoring_records                      │
  │ WHERE support_plan_id = $plan_id                      │
  │ ORDER BY monitoring_date DESC LIMIT 1                 │
  │                                                       │
  │ [B] 出席率の集計（今月・先月）                            │
  │ SELECT                                                │
  │   COUNT(*) FILTER (WHERE status='present') AS present,│
  │   COUNT(*) AS total,                                  │
  │   ROUND(                                              │
  │     COUNT(*) FILTER (WHERE status='present')::DECIMAL │
  │     / NULLIF(COUNT(*), 0) * 100, 1                    │
  │   ) AS rate                                           │
  │ FROM attendance_records                                │
  │ WHERE client_id = $client_id                          │
  │   AND date >= DATE_TRUNC('month', CURRENT_DATE)       │
  │                                                       │
  │ [C] 日報での言及（直近5件）                              │
  │ SELECT dr.report_date, dr.content, u.display_name     │
  │ FROM daily_reports dr                                  │
  │ JOIN users u ON dr.reported_by = u.id                 │
```

```sql
-- 利用者の支援計画経過を統合取得するファンクション
CREATE OR REPLACE FUNCTION get_support_plan_progress(
    p_client_id UUID
)
RETURNS JSONB
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
    v_result JSONB;
    v_plan RECORD;
    v_monitoring RECORD;
    v_attendance_current RECORD;
    v_attendance_previous RECORD;
    v_reports JSONB;
    v_days_until_review INTEGER;
BEGIN
    -- 有効な支援計画を取得
    SELECT * INTO v_plan
    FROM support_plans
    WHERE client_id = p_client_id
      AND status IN ('active', 'expired')
    ORDER BY plan_number DESC
    LIMIT 1;

    IF v_plan IS NULL THEN
        RETURN jsonb_build_object(
            'error', '有効な支援計画が見つかりません'
        );
    END IF;

    -- 見直し期限までの日数
    v_days_until_review := v_plan.review_date - CURRENT_DATE;

    -- 直近のモニタリング記録
    SELECT * INTO v_monitoring
    FROM monitoring_records
    WHERE support_plan_id = v_plan.id
    ORDER BY monitoring_date DESC
    LIMIT 1;

    -- 今月の出席率
    SELECT
        COUNT(*) FILTER (WHERE status = 'present') AS present_days,
        COUNT(*) AS total_days,
        ROUND(
            COUNT(*) FILTER (WHERE status = 'present')::DECIMAL
            / NULLIF(COUNT(*), 0) * 100, 1
        ) AS rate
    INTO v_attendance_current
    FROM attendance_records
    WHERE client_id = p_client_id
      AND date >= DATE_TRUNC('month', CURRENT_DATE)
      AND date <= CURRENT_DATE;

    -- 先月の出席率
    SELECT
        ROUND(
            COUNT(*) FILTER (WHERE status = 'present')::DECIMAL
            / NULLIF(COUNT(*), 0) * 100, 1
        ) AS rate
    INTO v_attendance_previous
    FROM attendance_records
    WHERE client_id = p_client_id
      AND date >= DATE_TRUNC('month', CURRENT_DATE) - INTERVAL '1 month'
      AND date < DATE_TRUNC('month', CURRENT_DATE);

    -- 日報での言及（直近5件）
    SELECT COALESCE(jsonb_agg(row_to_json(r)), '[]'::JSONB)
    INTO v_reports
    FROM (
        SELECT dr.report_date, dr.content, u.display_name AS reporter
        FROM daily_reports dr
        JOIN users u ON dr.reported_by = u.id
        WHERE dr.client_id = p_client_id
           OR dr.mentioned_clients @> ARRAY[p_client_id]
        ORDER BY dr.report_date DESC
        LIMIT 5
    ) r;

    -- 結果を構築
    v_result := jsonb_build_object(
        'plan', jsonb_build_object(
            'id', v_plan.id,
            'plan_number', v_plan.plan_number,
            'goal_short_term', v_plan.goal_short_term,
            'goal_long_term', v_plan.goal_long_term,
            'support_content', v_plan.support_content,
            'start_date', v_plan.start_date,
            'end_date', v_plan.end_date,
            'review_date', v_plan.review_date,
            'days_until_review', v_days_until_review,
            'status', v_plan.status
        ),
        'attendance', jsonb_build_object(
            'current_month', jsonb_build_object(
                'rate', v_attendance_current.rate,
                'present_days', v_attendance_current.present_days,
                'total_days', v_attendance_current.total_days
            ),
            'previous_month_rate', v_attendance_previous.rate
        ),
        'recent_reports', v_reports,
        'last_monitoring', CASE
            WHEN v_monitoring IS NOT NULL THEN
                jsonb_build_object(
                    'date', v_monitoring.monitoring_date,
                    'progress_notes', v_monitoring.progress_notes,
                    'attendance_rate', v_monitoring.attendance_rate,
                    'issues', v_monitoring.issues,
                    'next_actions', v_monitoring.next_actions
                )
            ELSE NULL
        END,
        'alerts', (
            SELECT COALESCE(jsonb_agg(alert), '[]'::JSONB)
            FROM (
                SELECT jsonb_build_object(
                    'type', CASE
                        WHEN v_days_until_review < 0 THEN 'overdue'
                        WHEN v_days_until_review <= 7 THEN 'urgent'
                        WHEN v_days_until_review <= 14 THEN 'warning'
                        WHEN v_days_until_review <= 30 THEN 'notice'
                        ELSE 'ok'
                    END,
                    'message', CASE
                        WHEN v_days_until_review < 0 THEN
                            '見直し期限を' || ABS(v_days_until_review) || '日超過しています'
                        WHEN v_days_until_review <= 7 THEN
                            '見直し期限まで' || v_days_until_review || '日です（緊急）'
                        WHEN v_days_until_review <= 14 THEN
                            '見直し期限まで' || v_days_until_review || '日です'
                        WHEN v_days_until_review <= 30 THEN
                            '見直し期限まで' || v_days_until_review || '日です'
                        ELSE
                            '見直し期限まで' || v_days_until_review || '日あります'
                    END
                ) AS alert
            ) sub
        )
    );

    RETURN v_result;
END;
$$;
```

### 6.3 期限チェックのCronジョブ

毎朝9:00に実行し、見直し期限が近い計画を抽出して通知を送信する。

```typescript
// cron/check-plan-deadlines.ts
// Vercel Cron: 毎朝 9:00 JST（UTC 0:00）に実行

import { createClient } from '@supabase/supabase-js';

interface PlanDeadline {
  plan_id: string;
  client_name: string;
  plan_number: number;
  review_date: string;
  days_remaining: number;
  assigned_to: string;
  assigned_to_line_id: string;
  has_monitoring: boolean;
}

export async function checkPlanDeadlines() {
  const supabase = createClient(
    process.env.SUPABASE_URL!,
    process.env.SUPABASE_SERVICE_ROLE_KEY!
  );

  // 期限が近い or 超過している有効な計画を取得
  const { data: plans, error } = await supabase.rpc(
    'get_plans_near_deadline'
  );

  if (error) {
    console.error('期限チェックエラー:', error);
    return;
  }

  for (const plan of plans as PlanDeadline[]) {
    const { days_remaining } = plan;

    // 30日前: 担当管理者に通知
    if (days_remaining === 30) {
      await sendNotification({
        plan,
        type: 'reminder_30d',
        recipients: [plan.assigned_to],
        urgency: 'normal',
      });
    }

    // 14日前: 担当管理者 + 社長に通知
    if (days_remaining === 14) {
      const ownerIds = await getOwnerUserIds(supabase);
      await sendNotification({
        plan,
        type: 'reminder_14d',
        recipients: [plan.assigned_to, ...ownerIds],
        urgency: 'high',
      });
    }

    // 7日前: 緊急通知
    if (days_remaining === 7) {
      const ownerIds = await getOwnerUserIds(supabase);
      await sendNotification({
        plan,
        type: 'reminder_7d',
        recipients: [plan.assigned_to, ...ownerIds],
        urgency: 'urgent',
      });
    }

    // 期限超過: 社長に毎日通知
    if (days_remaining < 0) {
      // 計画ステータスを expired に更新
      await supabase
        .from('support_plans')
        .update({ status: 'expired' })
        .eq('id', plan.plan_id)
        .eq('status', 'active');

      const ownerIds = await getOwnerUserIds(supabase);
      await sendNotification({
        plan,
        type: 'overdue',
        recipients: ownerIds,
        urgency: 'critical',
      });
    }
  }

  // アセスメント期限のチェック（利用開始後30日以内）
  await checkAssessmentDeadlines(supabase);
}
```

```sql
-- 期限が近い計画を取得するDB関数
CREATE OR REPLACE FUNCTION get_plans_near_deadline()
RETURNS TABLE (
    plan_id UUID,
    client_name TEXT,
    plan_number INTEGER,
    review_date TEXT,
    days_remaining INTEGER,
    assigned_to UUID,
    assigned_to_line_id TEXT,
    has_monitoring BOOLEAN
)
LANGUAGE sql
SECURITY DEFINER
AS $$
    SELECT
        sp.id AS plan_id,
        c.full_name AS client_name,
        sp.plan_number,
        sp.review_date::TEXT,
        (sp.review_date - CURRENT_DATE)::INTEGER AS days_remaining,
        sp.assigned_to,
        u.line_user_id AS assigned_to_line_id,
        EXISTS (
            SELECT 1 FROM monitoring_records mr
            WHERE mr.support_plan_id = sp.id
              AND mr.monitoring_date >= sp.start_date
        ) AS has_monitoring
    FROM support_plans sp
    JOIN clients c ON sp.client_id = c.id
    JOIN users u ON sp.assigned_to = u.id
    WHERE sp.status IN ('active', 'expired')
      AND (sp.review_date - CURRENT_DATE) <= 30
    ORDER BY sp.review_date ASC;
$$;

-- アセスメント期限チェック用の関数
CREATE OR REPLACE FUNCTION get_assessment_deadlines()
RETURNS TABLE (
    client_id UUID,
    client_name TEXT,
    start_date DATE,
    assessment_deadline DATE,
    days_remaining INTEGER,
    has_assessment BOOLEAN
)
LANGUAGE sql
SECURITY DEFINER
AS $$
    SELECT
        c.id AS client_id,
        c.full_name AS client_name,
        c.start_date,
        (c.start_date + INTERVAL '30 days')::DATE AS assessment_deadline,
        ((c.start_date + INTERVAL '30 days')::DATE - CURRENT_DATE)::INTEGER AS days_remaining,
        EXISTS (
            SELECT 1 FROM assessment_records ar
            WHERE ar.client_id = c.id
              AND ar.assessment_type = 'initial'
        ) AS has_assessment
    FROM clients c
    WHERE c.status = 'active'
      AND c.start_date >= CURRENT_DATE - INTERVAL '60 days'
      AND NOT EXISTS (
          SELECT 1 FROM assessment_records ar
          WHERE ar.client_id = c.id
            AND ar.assessment_type = 'initial'
      )
    ORDER BY c.start_date ASC;
$$;
```

### 8.1 管理者が変更になった場合（担当引き継ぎ）

**シナリオ**: サビ管が退職・異動し、担当利用者の支援計画を新しい管理者に引き継ぐ。

**処理**:
```
[トリガー]
  社長が「田中さんの担当を佐々木さんに変更して」等を送信

[処理フロー]
  1. 変更対象の確認
     - 旧担当者（田中）が assigned_to の全 support_plans を取得
     - 対象利用者の一覧を表示
  2. 一括変更 or 個別変更の選択
     - 「全員を佐々木さんに変更しますか？それとも個別に指定しますか？」
  3. 変更実行
     UPDATE support_plans
     SET assigned_to = $new_manager_id,
         updated_at = NOW()
     WHERE assigned_to = $old_manager_id
       AND status = 'active';
  4. 新担当者に通知
     - 引き継いだ利用者の一覧と直近の見直し期限を通知
  5. 変更履歴の記録
```

**会話例**:
```
社長: 田中さんが来月退職するので、担当の利用者を佐々木さんに引き継いで

AI秘書: 田中管理者が担当している利用者は以下の4名です。

  1. 佐藤太郎さん（第3期・期限: 3/31）
  2. 山田花子さん（第2期・期限: 4/15）
  3. 高橋美咲さん（第4期・期限: 5/20）
  4. 中村健一さん（第1期・期限: 6/01）

全員を佐々木管理者に引き継ぎますか？
それとも個別に担当を指定しますか？

社長: 全員佐々木さんで

AI秘書: 4名の支援計画の担当を佐々木管理者に変更しました。
        佐々木管理者に引き継ぎ内容を通知しました。

        ⚠️ 佐藤太郎さんの見直し期限が明日（3/31）です。
        佐々木管理者に至急対応を依頼しますか？
```

### 8.2 利用者がA型とB型を移行した場合

**シナリオ**: 利用者がA型からB型（またはその逆）に移行する場合、支援計画の扱いをどうするか。

**処理**:
```
[ルール]
  - A型/B型の移行は事業所内での異動扱いとする
  - 現在の支援計画をクローズし、新しい計画（第1期）として再作成するのが原則
  - ただし、連続性のため前計画の目標・経過は参照可能にする

[処理フロー]
  1. 利用者の事業種別を変更（clients.service_type = 'A' → 'B'）
  2. 現在の支援計画を status = 'closed' に変更（close_reason = 'type_transfer'）
  3. 新しい支援計画を第1期として作成
     - 前計画の情報を引き継ぎオプションとして表示
  4. アセスメントの更新を促す（環境変化に伴う再評価）

[会話例]
管理者: 山田さんがB型からA型に移ります

AI秘書: 山田花子さんのA型への移行処理を行います。

        1. 現在の支援計画（B型 第2期）をクローズします
        2. A型として新しい支援計画（第1期）を作成します
        3. アセスメントの更新を推奨します（A型の作業内容に合わせた再評価）

        B型での計画内容を新計画の参考として引き継ぎますか？

管理者: はい、引き継いで

AI秘書: B型での計画内容を引き継ぎました。
        新しい計画案を作成するため、A型での作業内容や目標について教えてください。
```

### 8.3 長期入院等で計画が一時停止する場合

**シナリオ**: 利用者が長期入院・長期欠席で支援が中断する場合、計画の期限管理をどうするか。

**処理**:
```
[ルール]
  - 30日以上の連続欠席が見込まれる場合、支援計画を一時停止可能
  - 一時停止中は見直し期限のカウントダウンを停止する
  - 復帰時に期限を再計算して計画を再開する

[データモデル拡張]
  support_plans テーブルに以下を追加:
    suspended_at TIMESTAMPTZ      -- 一時停止日
    resumed_at TIMESTAMPTZ        -- 再開日
    suspension_reason TEXT         -- 一時停止理由

[期限再計算ロジック]
  停止期間 = resumed_at - suspended_at
  新しい review_date = 元の review_date + 停止期間

[会話例]
管理者: 鈴木さんが入院することになった。1ヶ月くらい

AI秘書: 鈴木一郎さんの支援計画を一時停止しますか？

        現在の支援計画（第2期）
        見直し期限: 2026/04/15（あと16日）

        一時停止すると、見直し期限のカウントダウンが停止します。
        復帰時に期限が再計算されます。

        一時停止しますか？

管理者: はい。入院理由は腰の手術です

AI秘書: 鈴木一郎さんの支援計画を一時停止しました。
        一時停止日: 2026/03/30
        理由: 腰の手術による入院

        退院・復帰時に「鈴木さんが復帰した」と教えてください。
        見直し期限を自動で再計算します。
```

### 8.4 複数の目標がある場合の進捗管理

**シナリオ**: 一つの支援計画に短期目標が複数ある場合、それぞれの進捗を個別に管理する。

**処理**:
```
[データモデル]
  support_content JSONB の tasks 配列で複数目標を管理
  monitoring_records.progress_notes の goal_progress 配列で個別に進捗を記録

[表示例]
管理者: 高橋さんの支援計画の経過教えて

AI秘書: 高橋美咲さんの支援計画（第4期）の経過をお伝えします。

■ 短期目標の進捗
  目標1「封入作業を1時間集中して行える」
    → 現状: 45分程度（一部達成）
  目標2「ラベル貼りの手順を覚える」
    → 現状: 手順書を見ながら実施可能（達成間近）
  目標3「出席率80%以上を維持する」
    → 今月: 82%（達成）

■ 長期目標
  「対人コミュニケーションの改善（簡単な挨拶・報告ができる）」
    → 朝の挨拶は定着。作業報告は声かけが必要
```

### 8.5 その他のエッジケース

| ケース | 対応 |
|--------|------|
| 利用者名が重複する場合 | フルネーム + 利用者IDで確認を求める |
| 下書き状態の計画があるのに新規作成を依頼された場合 | 「下書きの計画があります。こちらを完成させますか？」 |
| 同一計画に対して複数回モニタリングが記録された場合 | 全て保存（中間モニタリングとして記録） |
| 6ヶ月未満で計画を見直したい場合 | 許可する（本人の状況変化に柔軟に対応） |
| 過去日付でモニタリングを記録したい場合 | 日付を指定して記録可能（「3/25に実施したモニタリングを記録したい」） |
| ネットワークエラーで保存に失敗した場合 | 入力内容をセッションに保持し、「保存に失敗しました。再度保存しますか？」 |
| Claude APIの応答が不適切だった場合 | 「整形結果が正しくない場合は修正を指示してください」とフォールバック |

### 9.4 行政監査における重要度

個別支援計画は行政監査（実地指導）で **最も厳しくチェックされる書類** の一つである。以下の指摘を受けると **報酬の返還** に発展する可能性がある：

| 指摘事項 | リスクレベル | 本機能での対策 |
|---------|------------|---------------|
| 計画の見直しが6ヶ月以内に行われていない | 最重要 | 段階的リマインド + 期限超過の毎日通知 |
| モニタリング記録が残っていない | 最重要 | LINEでの手軽な入力 + raw_input の保存 |
| 全利用者に計画が作成されていない | 最重要 | 新規利用者のアセスメント期限管理 |
| 本人・家族の同意が取れていない | 重要 | consent フラグの管理 + 未同意の警告 |
| 計画と実際の支援内容が乖離している | 重要 | 日報・出席データとの統合表示で乖離を検知 |

## 10. 将来の拡張案

### 10.1 支援計画書のPDF自動生成（P2）

行政提出用フォーマットに準拠した計画書PDFを自動生成する。

**概要**:
- support_plans + assessment_records + monitoring_records のデータからPDFを生成
- 自治体ごとに異なるフォーマットに対応（テンプレートエンジン）
- 署名欄・同意欄は印刷後に手書きで記入

**技術案**:
```
[候補]
  - @react-pdf/renderer: React コンポーネントでPDFを定義
  - Puppeteer + HTML テンプレート: HTMLからPDF変換
  - jsPDF: 軽量なPDF生成ライブラリ

[フォーマット項目]
  ├── 事業所情報（名称・事業所番号）
  ├── 利用者情報（氏名・生年月日・障害種別・障害等級）
  ├── 計画期間・作成日
  ├── アセスメント結果の要約
  ├── 長期目標・短期目標
  ├── 具体的な支援内容
  ├── モニタリング結果
  ├── 本人の意向
  └── 署名欄（利用者・家族・サビ管・管理者）
```

### 10.2 相談支援事業所との情報共有（P3）

相談支援専門員（ケアマネに相当する役割）にモニタリング結果を共有する機能。

**概要**:
- サービス等利用計画（相談支援事業所が作成）と個別支援計画の連携
- モニタリング結果のサマリーをPDF or テキストで共有
- 将来的にはAPIでの情報連携も検討

### 10.3 AIによる目標達成度の自動評価（P3）

日報・出席データからAIが目標の達成度を自動推定し、モニタリング面談の準備資料を生成する。

**概要**:
```
[入力データ]
  - 支援計画の目標テキスト
  - 日報での該当利用者に関する記録（直近6ヶ月分）
  - 出席率の推移データ
  - 過去のモニタリング記録

[AI処理]
  Claude APIに以下を入力:
  「この利用者の目標に対して、日報記録と出席データから達成度を評価してください」

[出力]
  - 目標ごとの達成度（5段階 or パーセンテージ）
  - 根拠となる日報記録の引用
  - 課題の自動抽出
  - 次期の目標案の提案
```

**注意点**: AIの評価はあくまで参考情報であり、最終判断はサビ管が行う。AIの出力に「これはAIによる参考評価です」と明記する。

### 10.4 家族向け経過報告書の生成（P3）

面談時に家族へ共有するための経過報告書を自動生成する。

**概要**:
- 専門用語を平易な表現に置き換えた報告書
- 本人の頑張りや成長を前面に出した構成
- 課題は「今後のサポートの方向性」として肯定的に表現

### 10.5 データ分析ダッシュボード（P3）

事業所全体の支援計画のステータスを可視化するダッシュボード。

**表示項目案**:
- 全利用者の計画ステータス分布（active / expired / reviewed）
- 期限超過率の推移グラフ
- 管理者ごとの担当利用者数と期限遵守率
- モニタリング実施率の月次推移
- 目標達成率の全体平均
