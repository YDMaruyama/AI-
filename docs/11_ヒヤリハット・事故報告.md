# ヒヤリハット・事故報告機能 詳細設計書

## 1. 概要

### 1.1 機能の目的

本機能は、就労継続支援事業所（A型/B型）における**ヒヤリハット**（事故に至らなかったが危険を感じた事例）および**事故**の報告・記録・管理を行うものである。

福祉施設における安全管理は最重要課題の一つであり、厚生労働省の「障害者福祉施設等における事故防止に関する指針」に基づき、事故報告書の作成と行政（市区町村・都道府県）への報告義務が課せられている。特に重大事故（利用者の死亡、重篤な負傷、長期入院等）が発生した場合は、速やかに市区町村へ報告しなければならない。

ヒヤリハット報告は法的義務ではないが、**ハインリッヒの法則**（1件の重大事故の背後には29件の軽微な事故があり、その背後には300件のヒヤリハットが存在する）に基づき、ヒヤリハットの段階で危険を察知・対策することが重大事故の予防に直結する。

### 1.2 現状の課題

| 課題 | 詳細 | 影響 |
|------|------|------|
| **紙ベースの報告書** | 所定の用紙に手書きで記入しており、書くのが面倒で後回しになる | 報告が翌日以降になり、記憶が曖昧になる。詳細が欠落する |
| **軽微な事象が報告されない** | 「大したことなかった」と判断してヒヤリハットが報告されない | 潜在的な危険が見過ごされ、同種の事故が繰り返される |
| **報告のハードルが高い** | 報告書のフォーマットが複雑で、5W1Hを漏れなく書くのが難しい | スタッフが報告を避ける傾向が生まれる |
| **社長への伝達遅延** | 紙の報告書が社長の手元に届くまでにタイムラグがある | 重大事故でも社長の状況把握が遅れ、対応が後手に回る |
| **再発防止策の管理不在** | 再発防止策が口頭で決まるだけで、実行管理されない | 同種の事故が繰り返される |
| **統計・傾向分析ができない** | 紙の報告書では集計・分析が困難 | どの作業・時間帯・場所で事故が多いかが把握できない |
| **行政報告の判断遅れ** | 報告義務の有無をスタッフが判断できない | 行政報告が遅れ、指導対象となるリスクがある |

### 1.3 AI秘書による解決

AI秘書は以下のアプローチでこれらの課題を解決する：

1. **対話形式の報告**: LINEでAIと会話するだけで報告が完了する。AIが5W1Hを順番にヒアリングするため、報告漏れがなくなる
2. **報告のハードルを極限まで下げる**: 「ヒヤリハット報告」と送るだけで開始。自由文で話すだけでAIが構造化する
3. **即時通知**: 報告が完了した瞬間に社長へ通知が届く。重大事故は特別アラートで即時通知する
4. **重大度の自動判定**: 怪我の有無・病院搬送の有無から自動で重大度を判定し、行政報告の要否も案内する
5. **再発防止策の管理**: 再発防止策をタスクとして自動生成し、実行状況を追跡する
6. **統計・傾向の可視化**: 月次サマリーで事故の傾向を把握し、予防策に活かす

### 1.4 ハインリッヒの法則とヒヤリハット報告の重要性

```
                    /\
                   /  \        重大事故: 1件
                  /    \       （死亡・重篤な負傷）
                 /──────\
                /        \     軽微な事故: 29件
               /          \    （軽い怪我・通院等）
              /────────────\
             /              \   ヒヤリハット: 300件
            /                \  （危険を感じた・ぶつかりそうになった等）
           /──────────────────\
          /                    \ 不安全行動・不安全状態: 無数
         /                      \（整理整頓不足・ルール不徹底等）
        /────────────────────────\
```

ヒヤリハットを報告・分析することで、重大事故を未然に防ぐことができる。本機能はこの考え方に基づき、**ヒヤリハットの報告を最も重視**する設計としている。

## 2. ユーザーストーリー

### 2.1 スタッフのユーザーストーリー

#### US-11-01: ヒヤリハット報告（スタッフ）

```
役割: スタッフ（支援員・職業指導員）
目的: 現場で危険を感じた事象をすぐに報告したい
理由: 紙の報告書は面倒で後回しになるが、LINEなら隙間時間に報告できる
```

**受け入れ条件:**
- LINEで「ヒヤリハット報告」と送信すると、AIが対話形式で5W1Hをヒアリングする
- When（いつ）、Who（誰が）、Where（どこで）、What（何が起きた）、Why（原因）、How（どう対応したか）を順に質問される
- 回答が曖昧な場合はAIが追加質問する
- 回答内容から報告書フォーマットに自動整形される
- 整形結果を確認後、保存・送信される
- 保存と同時に社長へ即時通知される
- 報告完了までの所要時間が2分以内である

**会話例:**
```
スタッフ: ヒヤリハット報告

AI: ヒヤリハット報告を受け付けます。
    順番にお聞きしますので、わかる範囲でお答えください。

    まず、いつ起きましたか？（日付と時間）

スタッフ: さっき、10時半くらい

AI: 本日（3月28日）10時30分頃ですね。
    次に、関わった利用者さんはいますか？お名前を教えてください。

スタッフ: 山田太郎さん

AI: 山田太郎さんですね。
    どこで起きましたか？（作業室、食堂、トイレ、廊下、屋外など）

スタッフ: 作業室

AI: 作業室ですね。
    何が起きましたか？できるだけ詳しく教えてください。

スタッフ: カッターで紙を切る作業中に、刃が滑って指に当たりそうになった

AI: カッター作業中に刃が滑り、指に接触しそうになったのですね。
    怪我はありましたか？

スタッフ: いや、ギリギリ当たらなかった

AI: 怪我はなしですね。よかったです。
    原因として思い当たることはありますか？
    （道具の不具合、注意力の低下、手順の問題など）

スタッフ: カッターの刃が古くて切れにくくなってて、力を入れすぎた

AI: カッターの刃の劣化が原因で、力を入れすぎたのですね。
    その場でどう対応しましたか？

スタッフ: 作業を一旦止めて、新しい刃に交換した

AI: ありがとうございます。以下の内容で報告書を作成しました。

    ━━━━━━━━━━━━━━━━━━━━
    【ヒヤリハット報告書】
    ━━━━━━━━━━━━━━━━━━━━
    ■ 種別: ヒヤリハット
    ■ 発生日時: 2026年3月28日 10:30頃
    ■ 発生場所: 作業室
    ■ 関係利用者: 山田太郎
    ■ 怪我の有無: なし

    ■ 事象の概要:
    カッターで紙を切る作業中、カッターの
    刃が劣化して切れにくくなっていたため、
    力を入れすぎて刃が滑り、指に接触しそう
    になった。幸い怪我には至らなかった。

    ■ 推定原因:
    カッターの刃の劣化（交換時期を超過）に
    より切れ味が低下。通常より力を入れて
    使用したことで刃が滑った。

    ■ 即時対応:
    作業を一旦中止し、カッターの刃を新品に
    交換した。

    ■ 報告者: 佐藤花子（スタッフ）
    ━━━━━━━━━━━━━━━━━━━━

    この内容でよろしいですか？
    修正があればお知らせください。

スタッフ: OK

AI: 報告を保存しました。社長に通知済みです。
    報告ありがとうございました。

    同様の事故を防ぐため、カッターの刃の
    定期交換をルール化することをお勧めします。
```

#### US-11-02: 事故報告（スタッフ）

```
役割: スタッフ
目的: 利用者が怪我をした等の事故を迅速に報告したい
理由: 事故発生後は対応に追われるが、記録は早めに残す必要がある
```

**受け入れ条件:**
- 事故報告はヒヤリハットより詳細な情報を収集する
- 怪我の程度（軽傷・中程度・重傷）を記録できる
- 病院搬送の有無を記録できる
- 家族への連絡状況を記録できる
- 重大事故の場合は行政報告の要否が自動判定・案内される
- 社長への即時通知に対応状況が含まれる

**会話例:**
```
スタッフ: 事故報告

AI: 事故報告を受け付けます。落ち着いて、順番にお答えください。

    まず、いつ起きましたか？

スタッフ: 今日の14時頃

AI: 本日（3月28日）14時頃ですね。
    関わった利用者さんのお名前を教えてください。

スタッフ: 鈴木一郎さん

AI: 鈴木一郎さんですね。
    どこで起きましたか？

スタッフ: 食堂

AI: 食堂ですね。
    何が起きましたか？できるだけ詳しく教えてください。

スタッフ: 昼食後の片付け中に椅子に足を引っかけて転倒した

AI: 椅子に足を引っかけて転倒されたのですね。
    怪我はありますか？ある場合、どの部位にどの程度の怪我ですか？

スタッフ: 右膝を打って擦り傷。あと右手首をひねったみたいで少し腫れてる

AI: 右膝の擦り傷と、右手首の捻挫の疑いですね。
    病院への搬送はしましたか？または受診の予定はありますか？

スタッフ: これから近くの整形外科に連れて行く予定

AI: 受診予定とのこと、承知しました。
    ご家族への連絡はされましたか？

スタッフ: まだしてない

AI: ご家族への連絡がまだとのことですね。
    受診前にご家族へ一報入れることをお勧めします。

    その場でどのような対応をしましたか？

スタッフ: 擦り傷は消毒して絆創膏を貼った。手首はアイシングして
    安静にしている。他のスタッフが付き添っている。

AI: ありがとうございます。以下の内容で報告書を作成しました。

    ━━━━━━━━━━━━━━━━━━━━
    【事故報告書】
    ━━━━━━━━━━━━━━━━━━━━
    ■ 種別: 事故（軽微）
    ■ 重大度: minor_accident
    ■ 発生日時: 2026年3月28日 14:00頃
    ■ 発生場所: 食堂
    ■ 関係利用者: 鈴木一郎

    ■ 事故の概要:
    昼食後の片付け中、椅子に足を引っかけて
    転倒。右膝に擦り傷、右手首を捻挫した疑い。

    ■ 怪我の程度: 軽傷
      - 右膝: 擦り傷（消毒・絆創膏処置済み）
      - 右手首: 捻挫の疑い（腫れあり）

    ■ 即時対応:
      - 擦り傷の消毒・絆創膏処置
      - 右手首のアイシング・安静
      - スタッフが付き添い中

    ■ 病院受診: 予定あり（近隣整形外科）
    ■ 家族連絡: 未了
    ■ 行政報告: 不要（軽傷のため）

    ■ 報告者: 佐藤花子（スタッフ）
    ━━━━━━━━━━━━━━━━━━━━

    この内容でよろしいですか？

スタッフ: OK

AI: 報告を保存しました。社長に即時通知しました。

    【リマインド】
    ・ご家族への連絡をお願いします
    ・受診結果がわかったら「事故報告 更新」と
      送ってください。記録を追記します。
```

### 2.2 管理者・社長のユーザーストーリー

#### US-11-03: 事故・ヒヤリハット一覧確認（社長）

```
役割: 社長
目的: 今月の事故・ヒヤリハットの全体像を把握したい
理由: 安全管理の責任者として、発生状況と対応状況を常に把握する必要がある
```

**受け入れ条件:**
- 「今月のヒヤリハット」「事故一覧」等で期間指定の一覧が取得できる
- 種類別（ヒヤリハット・軽微な事故・重大事故）で集計される
- 利用者別の発生件数が確認できる
- 再発防止策の実施状況が確認できる
- 未対応（再発防止策未記入）の報告がハイライトされる

**会話例:**
```
社長: 今月のヒヤリハット

AI: 3月の事故・ヒヤリハット報告です。

    【3月 サマリー】
    ・ヒヤリハット: 5件
    ・軽微な事故: 2件
    ・重大事故: 0件
    ━━━━━━━━━━━━━━━━━━━━

    【ヒヤリハット 5件】
    1. 3/5 作業室 - カッター刃滑り（山田太郎）
       → 再発防止: カッター刃の定期交換ルール策定済
    2. 3/10 廊下 - 段差でつまずき（佐々木次郎）
       → 再発防止: 段差に注意テープ貼付済
    3. 3/15 作業室 - はさみ落下（田中三郎）
       → 再発防止: 未記入 ⚠
    4. 3/20 食堂 - 熱い味噌汁こぼれかけ（高橋四郎）
       → 再発防止: 未記入 ⚠
    5. 3/28 作業室 - カッター刃滑り（山田太郎）
       → 再発防止: 未記入 ⚠

    【軽微な事故 2件】
    1. 3/12 トイレ - 転倒・打撲（伊藤五郎）
       → 再発防止: 滑り止めマット設置済
       → 受診: あり（異常なし）
    2. 3/28 食堂 - 転倒・擦り傷+捻挫疑い（鈴木一郎）
       → 再発防止: 未記入 ⚠
       → 受診: 予定あり

    【注意】
    ・再発防止策が未記入のものが4件あります
    ・カッター関連のヒヤリハットが2件発生して
      います（3/5, 3/28）。カッター使用時の
      安全ルール見直しをお勧めします。

    詳細を見たい報告の番号を教えてください。
```

#### US-11-04: 再発防止策の記録（社長）

```
役割: 社長
目的: 事故・ヒヤリハットに対する再発防止策を記録し、実行を管理したい
理由: 再発防止策を決めても、記録・実行管理しないと形骸化する
```

**受け入れ条件:**
- 特定の報告に対して再発防止策を記録できる
- 再発防止策がタスクとして自動生成される
- タスクの担当者・期限を設定できる
- 実施状況を追跡できる

**会話例:**
```
社長: 3月28日のカッターのヒヤリハット、再発防止策を入れたい

AI: 3月28日の報告（カッター刃滑り/山田太郎さん）ですね。
    なお、3月5日にも同種のヒヤリハットがありました。
    合わせて対策を考えるのが良いかと思います。

    再発防止策をお聞かせください。

社長: カッター使用時はスタッフが隣に付くようにする。
    あと刃は毎週月曜に全部交換する。

AI: 以下の2つの再発防止策を記録しました。

    【再発防止策】
    1. カッター使用時はスタッフが隣で見守る
    2. カッターの刃を毎週月曜日に全数交換する

    タスクを自動生成しました：
    ・「カッター使用時の見守りルールを全スタッフに周知」
      → 担当: 佐藤花子 / 期限: 4月1日
    ・「カッター刃の週次交換ルールを掲示・運用開始」
      → 担当: 佐藤花子 / 期限: 4月1日

    3月5日の同種報告にも同じ再発防止策を適用
    しますか？

社長: はい

AI: 3月5日の報告にも適用しました。
    両報告のステータスを「対応済み」に更新しました。
```

## 4. 機能要件

### 4.1 必須機能（MVP）

| ID | 機能 | 説明 | 優先度 |
|----|------|------|--------|
| F-11-01 | ヒヤリハット報告 | 対話形式で5W1Hを収集し、報告書を自動整形・保存する | P0 |
| F-11-02 | 事故報告 | ヒヤリハットに加え、怪我の程度・病院搬送・家族連絡の情報を収集・記録する | P0 |
| F-11-03 | 重大度自動判定 | 怪我の有無・程度・病院搬送から incident_type を自動判定する | P0 |
| F-11-04 | 即時通知（社長） | 報告完了時に社長へLINEで即時通知する。重大事故は緊急通知とする | P0 |
| F-11-05 | 一覧表示 | 期間指定で事故・ヒヤリハットの一覧を種別・利用者別に表示する | P0 |
| F-11-06 | 再発防止策記録 | 報告に対して再発防止策を追記し、タスクとして自動生成する | P0 |
| F-11-07 | 報告更新 | 既存の報告に追加情報（受診結果等）を追記する | P0 |
| F-11-08 | 行政報告要否判定 | 重大事故の場合に行政報告の必要性を自動判定・案内する | P0 |

### 4.2 任意機能（将来実装）

| ID | 機能 | 説明 | 優先度 |
|----|------|------|--------|
| F-11-09 | 事故報告書PDF生成 | 行政提出用フォーマットで事故報告書のPDFを自動生成する | P1 |
| F-11-10 | 行政報告書フォーマット出力 | 市区町村指定の報告書フォーマットに合わせた出力 | P1 |
| F-11-11 | 統計・傾向分析 | 事故種類別・時間帯別・場所別・利用者別の統計・グラフ | P2 |
| F-11-12 | 重複報告の統合 | 同一事故を複数人が報告した場合の自動検出・統合 | P2 |
| F-11-13 | 遅延報告フラグ | 報告が発生日から2日以上遅れた場合に遅延フラグを付与 | P2 |
| F-11-14 | 安全研修リマインド | 同種事故が複数回発生した場合に安全研修の実施を提案 | P3 |

### 4.3 非機能要件

| 項目 | 要件 |
|------|------|
| 応答速度 | 報告の保存から社長への通知送信まで5秒以内 |
| 可用性 | 事故報告はシステム障害時でもテキストメッセージとして受信し、復旧後に登録する |
| データ保持 | 事故報告データは最低5年間保持する（行政監査対応） |
| セキュリティ | 利用者の個人情報を含むため、アクセス権限を厳密に管理する |
| 監査対応 | 報告の作成・変更履歴を全て記録する（変更不可） |

### 6.2 重大度自動判定ロジック

```typescript
// ============================================================
// 重大度判定ロジック
// ============================================================

interface SeverityJudgment {
  incident_type: 'near_miss' | 'minor_accident' | 'major_accident';
  admin_report_required: boolean;
  alert_level: 'normal' | 'urgent' | 'critical';
  reason: string;
}

function judgeSeverity(input: IncidentReportInput): SeverityJudgment {
  // ルール1: 怪我なし → ヒヤリハット
  if (input.injury_level === 'none') {
    return {
      incident_type: 'near_miss',
      admin_report_required: false,
      alert_level: 'normal',
      reason: '怪我なし。ヒヤリハットとして記録。',
    };
  }

  // ルール2: 重傷（骨折・大量出血・意識障害等）→ 重大事故
  if (input.injury_level === 'severe') {
    return {
      incident_type: 'major_accident',
      admin_report_required: true,
      alert_level: 'critical',
      reason: '重傷のため重大事故。行政報告義務あり。',
    };
  }

  // ルール3: 救急搬送あり → 重大事故
  if (input.hospital_visit && isEmergencyTransport(input.hospital_detail)) {
    return {
      incident_type: 'major_accident',
      admin_report_required: true,
      alert_level: 'critical',
      reason: '救急搬送のため重大事故。行政報告義務あり。',
    };
  }

  // ルール4: 中程度の怪我（通院が必要な程度）→ 軽微な事故
  if (input.injury_level === 'moderate') {
    return {
      incident_type: 'minor_accident',
      admin_report_required: false,
      alert_level: 'urgent',
      reason: '中程度の怪我。軽微な事故として記録。',
    };
  }

  // ルール5: 軽傷だが病院受診あり → 軽微な事故
  if (input.injury_level === 'minor' && input.hospital_visit) {
    return {
      incident_type: 'minor_accident',
      admin_report_required: false,
      alert_level: 'urgent',
      reason: '軽傷だが受診あり。軽微な事故として記録。',
    };
  }

  // ルール6: 軽傷で病院受診なし → 軽微な事故（最低レベル）
  if (input.injury_level === 'minor') {
    return {
      incident_type: 'minor_accident',
      admin_report_required: false,
      alert_level: 'normal',
      reason: '軽傷。軽微な事故として記録。',
    };
  }

  // フォールバック
  return {
    incident_type: 'near_miss',
    admin_report_required: false,
    alert_level: 'normal',
    reason: '判定不能。ヒヤリハットとして記録。',
  };
}

// 救急搬送の判定（キーワードベース）
function isEmergencyTransport(detail?: string): boolean {
  if (!detail) return false;
  const emergencyKeywords = ['救急', '119', '搬送', '緊急', 'ER', '救急車'];
  return emergencyKeywords.some(keyword => detail.includes(keyword));
}
```

```typescript
interface AdminReportJudgment {
  required: boolean;
  reason: string;

function judgeAdminReport(
  incident_type: string,
  injury_level: string,
  hospital_visit: boolean,
  description: string
): AdminReportJudgment {

  // 重大事故は原則報告義務あり
  if (incident_type === 'major_accident') {
    return {
      required: true,
      reason: '重大事故（重傷または救急搬送）のため、行政への報告義務があります。',

  const recommendCases = [
    { condition: injury_level === 'moderate' && hospital_visit, reason: '中程度の怪我で通院あり' },

    { condition: description.includes('虐待') || description.includes('暴力'), reason: '虐待・暴力の疑い' },
  ];

  for (const c of recommendCases) {
    if (c.condition) {
      return {
        required: true, // 実質義務と同等
        reason: `${c.reason}のため、行政への報告を強く推奨します。`,
        deadline: '事故発生後、可能な限り速やかに。',

        format_hint: '市区町村指定の事故報告書フォーマットを使用してください。',
      };
    }
  }

  return {
    required: false,
    reason: '行政への報告義務はありません。ただし、事業所内での記録・管理は必要です。',
    deadline: '-',
    report_to: '-',
```

```typescript
function formatIncidentReport(input: IncidentReportInput, judgment: SeverityJudgment): string {

  if (input.injury_detail) {
    report += `\n■ 怪我の詳細:\n${input.injury_detail}\n`;
  }

  report += `\n■ 事象の概要:\n${input.description}\n`;

  if (input.cause) {
    report += `\n■ 推定原因:\n${input.cause}\n`;
  }

  if (input.immediate_action) {
    report += `\n■ 即時対応:\n${input.immediate_action}\n`;
  }

  // 事故の場合の追加情報
  if (judgment.incident_type !== 'near_miss') {
    report += `\n■ 病院受診: ${input.hospital_visit ? 'あり' : 'なし'}`;
    if (input.hospital_detail) {
      report += `（${input.hospital_detail}）`;
    }
    report += '\n';

    report += `■ 家族連絡: ${input.family_notified ? '連絡済み' : '未了'}`;
    if (input.family_notified_to) {
      report += `（${input.family_notified_to}）`;
    }
    report += '\n';

    report += `■ 行政報告: ${judgment.admin_report_required ? '必要' : '不要'}`;
    if (judgment.admin_report_required) {
      report += '（※速やかに報告してください）';
```

### 6.6 統計集計クエリ

```sql
-- ============================================================
-- 月次サマリー（種類別集計）
-- ============================================================
SELECT
    incident_type,
    COUNT(*) AS count,
    COUNT(*) FILTER (WHERE injury_level != 'none') AS with_injury,
    COUNT(*) FILTER (WHERE hospital_visit = TRUE) AS with_hospital,
    COUNT(*) FILTER (WHERE admin_report_required = TRUE) AS with_admin_report,
    COUNT(*) FILTER (WHERE prevention_measures = '[]'::jsonb) AS no_prevention,
    COUNT(*) FILTER (WHERE status = 'closed') AS closed_count
FROM incident_reports
WHERE incident_date >= DATE_TRUNC('month', CURRENT_DATE)
  AND incident_date < DATE_TRUNC('month', CURRENT_DATE) + INTERVAL '1 month'
GROUP BY incident_type
ORDER BY
    CASE incident_type
        WHEN 'major_accident' THEN 1
        WHEN 'minor_accident' THEN 2
        WHEN 'near_miss' THEN 3
    END;

SELECT
    COALESCE(client_name_snapshot, 'スタッフ・その他') AS client_name,

    COUNT(*) FILTER (WHERE incident_type = 'near_miss') AS near_miss_count,
    COUNT(*) FILTER (WHERE incident_type = 'minor_accident') AS minor_count,
    COUNT(*) FILTER (WHERE incident_type = 'major_accident') AS major_count
FROM incident_reports
WHERE incident_date >= DATE_TRUNC('month', CURRENT_DATE)
  AND incident_date < DATE_TRUNC('month', CURRENT_DATE) + INTERVAL '1 month'
GROUP BY client_id, client_name_snapshot
ORDER BY total_count DESC;

    location,

    COUNT(*) FILTER (WHERE incident_type != 'near_miss') AS accident_count

WHERE incident_date >= DATE_TRUNC('month', CURRENT_DATE)
  AND incident_date < DATE_TRUNC('month', CURRENT_DATE) + INTERVAL '1 month'

-- ============================================================
SELECT
    CASE
        WHEN incident_time IS NULL THEN '不明'

        WHEN EXTRACT(HOUR FROM incident_time) BETWEEN 12 AND 13 THEN '昼（12-14時）'
        WHEN EXTRACT(HOUR FROM incident_time) BETWEEN 14 AND 16 THEN '午後（14-17時）'
        ELSE 'その他'
    END AS time_slot,
    COUNT(*) AS total_count,
    COUNT(*) FILTER (WHERE incident_type != 'near_miss') AS accident_count
FROM incident_reports
GROUP BY time_slot
ORDER BY total_count DESC;

    id,

    a.id AS report_id,

    AND ABS(EXTRACT(EPOCH FROM (a.created_at - b.created_at))) < 7776000  -- 90日以内
```

```typescript
async function createPreventionTasks(
  reportId: number,

    // 1. タスクテーブルにINSERT（04_タスク自動生成のテーブルを使用）

      title: `【再発防止】${m.measure}`,

      source_type: 'incident_report',
      source_id: reportId,
      assigned_to: assigneeId,
      due_date: deadline,
      priority: 'high',
      status: 'pending',
    });

    // 2. incident_reports の prevention_measures を更新
    await db.query(`
      UPDATE incident_reports
      SET prevention_measures = prevention_measures || $1::jsonb
      WHERE id = $2
    `, [
      JSON.stringify([{
        id: measures.indexOf(m) + 1,
        measure: m.measure,
        category: m.category,
        task_id: task.id,
        task_status: 'pending',
        assigned_to: assigneeId,
        deadline: deadline,
        completed_at: null,
        created_at: new Date().toISOString(),
      }]),
      reportId,
    ]);

    // 3. 変更履歴を記録
    await db.incident_report_updates.insert({
      incident_report_id: reportId,
      update_type: 'prevention_added',
      field_changed: 'prevention_measures',
      old_value: null,
      new_value: m.measure,
      note: `再発防止策を追加。タスクID: ${task.id}`,
      updated_by: getCurrentUserId(),

  await db.incident_reports.update(reportId, {
    status: 'reviewed',
```

```typescript
async function detectDuplicate(newReport: IncidentReportInput): Promise<number | null> {
  // 同日・同場所・同利用者の報告を検索
  const candidates = await db.query(`
    SELECT id, incident_date, incident_time, location,
           client_name_snapshot, description, reported_by
    FROM incident_reports
    WHERE incident_date = $1
      AND location = $2
      AND (client_id = $3 OR ($3 IS NULL AND client_id IS NULL))
      AND created_at > NOW() - INTERVAL '24 hours'
    ORDER BY created_at DESC
  `, [newReport.incident_date, newReport.location, newReport.client_id]);

  if (candidates.length === 0) return null;

  // 時間が近い（前後1時間以内）報告があれば重複の可能性
  for (const candidate of candidates) {
    if (isTimeSimilar(newReport.incident_time, candidate.incident_time, 60)) {
      return candidate.id;
    }
  }

  return null;
}
```

### 9.4 行政対応における重要性

本機能は行政監査・指導において特に重要度が高い。以下の理由から、開発優先度を高く設定する。

1. **法的義務**: 重大事故の報告義務を怠ると行政指導の対象となる
2. **実地指導**: 行政の実地指導時に事故報告書の記録が求められる。紙の報告書が散逸していると指導対象となるリスクがある
3. **監査証跡**: デジタル化により全ての報告・変更が記録され、監査に耐えうる記録が残る
4. **報告の迅速性**: 重大事故の発生から行政報告までのリードタイムを短縮できる

### 10.3 安全研修リマインド（P3）

同種事故が一定回数以上発生した場合に、安全研修の実施を提案する。

```typescript
// 同種事故の閾値チェック（月次バッチ）
async function checkTrainingNeeds(): Promise<void> {
  // 直近3ヶ月で同じ場所・同じ種類の事故が3件以上
  const patterns = await db.query(`
    SELECT location,
           COUNT(*) as count,
           array_agg(DISTINCT client_name_snapshot) as clients
    FROM incident_reports
    WHERE incident_date >= CURRENT_DATE - INTERVAL '90 days'
    GROUP BY location
    HAVING COUNT(*) >= 3
    ORDER BY count DESC
  `);

  for (const pattern of patterns) {
    await sendLineMessage(OWNER_USER_ID,
      `【安全研修の提案】\n` +
      `${pattern.location}で直近3ヶ月に${pattern.count}件の事故・ヒヤリハットが発生しています。\n` +
      `安全研修の実施をお勧めします。\n` +
      `研修タスクを生成しますか？`
    );
  }
}
```

## 付録A: 用語集

| 用語 | 説明 |
|------|------|
| ヒヤリハット | 事故に至らなかったが、危険を感じた事例。「ヒヤリ」としたり「ハッ」とした経験から命名 |
| インシデント | ヒヤリハットと事故を総称する用語 |
| ハインリッヒの法則 | 1件の重大事故の背後に29件の軽微な事故と300件のヒヤリハットが存在するという経験則 |
| near_miss | ヒヤリハット（怪我なし） |
| minor_accident | 軽微な事故（軽傷、通院程度） |
| major_accident | 重大事故（重傷、入院、救急搬送） |
| 行政報告 | 重大事故発生時に市区町村へ提出する義務のある報告 |
| 再発防止策 | 同種事故の再発を防ぐために講じる対策 |
| 5W1H | When, Who, Where, What, Why, How の頭文字。報告の基本要素 |

## 付録B: injury_level 判定ガイドライン

| レベル | 値 | 具体例 | 対応目安 |
|--------|-----|--------|----------|
| なし | `none` | 怪我なし（ヒヤリハット） | 記録のみ |
| 軽傷 | `minor` | 擦り傷、小さな切り傷、軽い打撲 | 事業所内で処置可能 |
| 中程度 | `moderate` | 捻挫、深い切り傷、広範囲の打撲、軽度のやけど | 通院が必要 |
| 重傷 | `severe` | 骨折、大量出血、意識障害、重度のやけど、頭部打撲 | 救急搬送・入院 |

## 付録C: 行政報告が必要となる事故の例

以下に該当する場合は原則として行政（市区町村）への報告が必要：

1. **利用者の死亡**
2. **重篤な負傷**（骨折、入院を要する怪我）
3. **食中毒・感染症の集団発生**
4. **利用者の行方不明・離設**
5. **職員による虐待の疑い**
6. **その他、市区町村が報告を求める事案**

※具体的な報告基準は市区町村によって異なるため、所管の障がい福祉課に確認すること。
