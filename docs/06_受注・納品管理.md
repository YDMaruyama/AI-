# 06. 受注・納品管理

## 1. 概要

### 1.1 この機能が解決する課題

就労継続支援事業所（A型/B型）では、外部企業から受注した作業案件（封入作業、清掃業務、データ入力、軽作業、検品作業等）が事業収入の柱となる。現状、以下の課題が存在する：

- **案件状況の把握が困難**: どの案件がどこまで進んでいるか、社長やスタッフが正確に把握できていない。口頭確認や紙のメモに依存しており、情報が属人化している
- **納期遅れの事前検知ができない**: 納期に間に合わないペースで作業が進んでいても、直前まで気づかないことがある。納期遅延は取引先との信頼関係を損ない、事業所の評判に直結する
- **工賃計算の根拠データがない**: 利用者の作業量が正確に記録されていないため、工賃計算が曖昧になりがち。A型事業所では最低賃金保障が必要であり、作業量に基づく公正な工賃支払いのために正確なデータが不可欠
- **受注判断の材料がない**: 新しい案件を受注できるか（現在の作業キャパシティに余裕があるか）を判断するための情報が即座に取得できない
- **月次契約案件の管理が煩雑**: 清掃業務やデータ入力業務など、毎月繰り返しの案件が多いが、そのたびに手動で案件を登録・管理している

### 1.2 提供する価値

- **社長がいつでも全案件を把握できる**: 「今の案件」とLINEで送るだけで、全案件の進捗・納期・ステータスが一覧で返ってくる
- **納期遅れを事前に検知し自動アラート**: 作業ペースと残り日数から「このままだと間に合わない」を自動判定し、社長・担当スタッフに警告を送る
- **進捗報告が簡単**: スタッフは「封入作業 今日150個」と送るだけで進捗が記録される。日報との連動も可能
- **工賃計算の基礎データとなる**: 利用者別・案件別の作業量が正確に記録され、09_売上・工賃集計の入力データとして活用される
- **受注判断を即座にサポート**: 現在の作業キャパシティと進行中案件の状況から、新規受注の可否判断を支援する

### 1.3 対象となる作業種類

| 作業カテゴリ | 具体例 | 数量単位 | 特徴 |
|------------|--------|---------|------|
| 封入作業 | DM封入、チラシ折り、ラベル貼り | 個、枚、セット | 数量が明確、単価が決まりやすい |
| 清掃業務 | ビル清掃、公園清掃、施設清掃 | 時間、回 | 月次契約が多い、作業時間ベース |
| データ入力 | 名簿入力、アンケート集計、PDF変換 | 件、ページ | 正確性が求められる |
| 軽作業 | 部品組立、検品、仕分け | 個、箱、ロット | 大量・反復作業が多い |
| 映像制作関連 | テロップ入力、素材整理、簡易編集 | 本、カット | さくら映像研の本業関連 |
| その他 | 農作業、リサイクル分別、清掃用品補充 | kg、袋、回 | 事業所ごとに多様 |

## 2. ユーザーストーリー

### 2.1 スタッフ：進捗報告

```
スタッフとして、
「封入作業 今日150個完了」とLINEで送るだけで
その日の作業実績が記録され、
残りの作業量や納期に間に合うかも自動で教えてほしい。

受け入れ条件：
- 自然言語で作業名と数量を送信するだけで記録される
- 進捗率（%）と残り数量が返答に含まれる
- 納期に間に合わないペースの場合、警告メッセージが表示される
- 日報作成時に、その日の作業実績が自動反映される
```

### 2.2 スタッフ：当日の自分の作業確認

```
スタッフとして、
「今日の作業」と送ったら
自分が担当している進行中の案件と、今日やるべき目安量が知りたい。

受け入れ条件：
- 自分が担当する進行中案件の一覧が返る
- 各案件の残り数量と納期が表示される
- 納期までの推奨日産ペースが表示される
```

### 2.3 管理者：案件の進捗と利用者配置

```
管理者として、
各案件の進捗率・残り作業量を把握し、
利用者の配置（誰をどの作業に割り当てるか）の判断に活用したい。

受け入れ条件：
- 「案件進捗」で全案件の進捗率・残数量・納期を一覧表示
- 進捗が遅れている案件がハイライトされる
- 各案件の担当スタッフ名が表示される
```

### 2.4 社長：全案件の把握

```
社長として、
「今抱えてる仕事一覧」とLINEで送るだけで
全ての進行中案件のステータス・進捗率・納期・金額が一覧で見たい。

受け入れ条件：
- 進行中の全案件が一覧で返る
- 各案件の進捗率がバーまたはパーセントで表示される
- 納期順にソートされる
- 合計受注金額が表示される
```

### 2.5 社長：個別案件の確認

```
社長として、
「○○株式会社の案件どう？」と送ったら
その取引先の案件の詳細状況がすぐ分かりたい。

受け入れ条件：
- 取引先名で案件を検索できる
- 進捗率、残り数量、納期までの日数、日別作業推移が表示される
- 間に合うかどうかのAI判定が含まれる
```

### 2.6 社長：新規案件の登録

```
社長として、
「○○株式会社から封入2000個、納期4/5、単価@15円」と送るだけで
案件が登録されるようにしたい。

受け入れ条件：
- 自然言語から取引先名・作業内容・数量・納期・単価を自動抽出
- 抽出結果を確認画面で表示し、社長が「OK」で確定
- 確定後、関連スタッフに通知が飛ぶ
```

### 2.7 社長：納期アラートの自動受信

```
社長として、
特に何もしなくても
納期が近い案件や進捗が遅れている案件のアラートを
自動的に受け取りたい。

受け入れ条件：
- 納期3日前に自動通知（進捗80%未満の場合は警告付き）
- 作業ペースが予定を下回った場合に自動通知
- 案件完了時に売上額付きで自動通知
```

## 4. 機能要件

### 4.1 必須機能（MVP）

| # | 機能 | 説明 | 優先度 |
|---|------|------|--------|
| F-1 | 案件登録 | 自然言語から案件情報を抽出して登録する。取引先名、案件名、数量、単価、納期を含む | P0 |
| F-2 | 進捗報告 | スタッフがLINEで作業量を報告し、累計・残り・進捗率を自動計算する | P0 |
| F-3 | 案件一覧表示 | 進行中の全案件をステータス・進捗率・納期付きで一覧表示する | P0 |
| F-4 | 個別案件詳細 | 取引先名や案件名で検索し、詳細情報・作業推移を表示する | P0 |
| F-5 | 納期管理 | 残り日数×平均ペースから納期予測を行い、遅延リスクを検知する | P0 |
| F-6 | 完了処理 | 案件を完了にし、売上額を確定。社長に通知を送る | P0 |
| F-7 | 納期アラート | 定期的に全案件の納期リスクを評価し、自動通知する | P0 |
| F-8 | 月次契約管理 | 毎月繰り返しの案件を自動作成・管理する | P1 |

### 4.2 任意機能（拡張）

| # | 機能 | 説明 | 優先度 |
|---|------|------|--------|
| F-9 | 売上自動計算 | 数量×単価から売上を自動計算。月次集計にも対応 | P1 |
| F-10 | 利用者別作業量記録 | 利用者ごとの作業量を記録。工賃計算の根拠データとなる | P1 |
| F-11 | 取引先管理 | 取引先の連絡先・過去案件履歴を管理する | P2 |
| F-12 | 案件テンプレート | よくある案件をテンプレートとして保存し、登録を簡素化 | P2 |
| F-13 | 受注可否判断支援 | 現在のキャパシティから新規受注の可否を提案する | P2 |
| F-14 | 日報連動 | 日報作成時に当日の作業実績を自動で取り込む | P1 |

AIが社長のメッセージから以下の情報を抽出する：

```

抽出結果：

  "client_company": "山田工業",
  "title": "封入作業",

  "unit_price": 15,
  "deadline": "2026-04-05",
  "order_type": "one_time"
}
```

抽出ルール：
- 取引先名：「○○から」「○○の」で判定。既存取引先との名寄せを行う
- 作業内容：「封入」「清掃」「データ入力」「ラベル」等のキーワードで判定
- 数量：数字+単位（個、件、枚、セット、箱、時間等）で判定
- 単価：「@○○円」「単価○○円」「1個○○円」等で判定
- 納期：日付表現（4/5、4月5日、来週金曜、今月末 等）をパース
- 種別：「毎月」「月次」「定期」があれば recurring、なければ one_time

### 5.5 SQL: ビュー -- 案件進捗サマリー

```sql
-- =====================================================
-- 案件進捗サマリービュー
-- 案件ごとの累計作業量・進捗率・平均ペースを集計
-- =====================================================
CREATE OR REPLACE VIEW order_progress_summary AS
SELECT
    o.id AS order_id,
    o.client_company,
    o.title,
    o.quantity_total,
    o.quantity_unit,
    o.unit_price,
    o.total_amount,
    o.deadline,
    o.status,
    o.assigned_staff_ids,
    o.order_type,

    -- 累計作業量
    COALESCE(SUM(p.quantity_done), 0) AS quantity_completed,

    -- 残り作業量
    CASE
        WHEN o.quantity_total IS NOT NULL
        THEN o.quantity_total - COALESCE(SUM(p.quantity_done), 0)
        ELSE NULL
    END AS quantity_remaining,

    -- 進捗率（%）
    CASE
        WHEN o.quantity_total IS NOT NULL AND o.quantity_total > 0
        THEN ROUND(
            COALESCE(SUM(p.quantity_done), 0)::NUMERIC / o.quantity_total * 100,
            1
        )
        ELSE NULL
    END AS progress_percent,

    -- 作業日数
    COUNT(DISTINCT p.reported_date) AS working_days,

    -- 平均日産ペース
    CASE
        WHEN COUNT(DISTINCT p.reported_date) > 0
        THEN ROUND(
            COALESCE(SUM(p.quantity_done), 0)::NUMERIC / COUNT(DISTINCT p.reported_date),
            1
        )
        ELSE 0
    END AS avg_daily_pace,

    -- 納期までの残り日数（営業日ではなくカレンダー日数）
    CASE
        WHEN o.deadline IS NOT NULL
        THEN (o.deadline - CURRENT_DATE)
        ELSE NULL
    END AS days_until_deadline,

    -- 最終報告日
    MAX(p.reported_date) AS last_reported_date

FROM orders o
LEFT JOIN order_progress p ON o.id = p.order_id
GROUP BY o.id;
```

### 6.4 進捗報告 -- バックエンド処理（TypeScript）

```typescript
// /api/orders/report-progress.ts

interface ProgressReport {
  order_id: string;
  reported_by: string;
  quantity_done: number;
  note?: string;
  reported_date?: string;    // デフォルトは当日
}

interface ProgressAnalysis {
  quantity_completed: number;     // 累計完了数
  quantity_remaining: number;     // 残り数量
  progress_percent: number;       // 進捗率（%）
  working_days: number;           // 稼働日数
  avg_daily_pace: number;         // 平均日産ペース
  days_until_deadline: number;    // 納期までの残り日数
  required_daily_pace: number;    // 必要な日産ペース
  estimated_completion: string;   // 完了予測日
  status: 'on_track' | 'slightly_behind' | 'at_risk'; // 状態判定
}

async function reportProgress(report: ProgressReport): Promise<ProgressAnalysis> {
  // 1. 進捗の記録
  const { error: insertError } = await supabase
    .from('order_progress')
    .insert({
      order_id: report.order_id,
      reported_by: report.reported_by,
      quantity_done: report.quantity_done,
      note: report.note || null,
      reported_date: report.reported_date || new Date().toISOString().split('T')[0],
    });

  if (insertError) throw insertError;

  // 2. 累計・分析の計算
  const analysis = await analyzeOrderProgress(report.order_id);

  // 3. 遅延アラートの判定
  if (analysis.status === 'at_risk') {
    await sendDeadlineAlert(report.order_id, analysis);
  }

  return analysis;
}

async function analyzeOrderProgress(orderId: string): Promise<ProgressAnalysis> {
  // 案件情報の取得
  const { data: order } = await supabase
    .from('orders')
    .select('*')
    .eq('id', orderId)
    .single();

  // 進捗集計
  const { data: progressRecords } = await supabase
    .from('order_progress')
    .select('quantity_done, reported_date')
    .eq('order_id', orderId);

  const quantityCompleted = progressRecords?.reduce(
    (sum, r) => sum + r.quantity_done, 0
  ) ?? 0;

  const workingDays = new Set(
    progressRecords?.map(r => r.reported_date)
  ).size;

  const quantityRemaining = (order.quantity_total ?? 0) - quantityCompleted;
  const progressPercent = order.quantity_total
    ? Math.round((quantityCompleted / order.quantity_total) * 1000) / 10
    : 0;

  const avgDailyPace = workingDays > 0
    ? Math.round(quantityCompleted / workingDays)
    : 0;

  // 納期までの日数計算
  const today = new Date();
  const deadline = order.deadline ? new Date(order.deadline) : null;
  const daysUntilDeadline = deadline
    ? Math.ceil((deadline.getTime() - today.getTime()) / (1000 * 60 * 60 * 24))
    : null;

  // 必要ペースの計算
  const requiredDailyPace = (daysUntilDeadline && daysUntilDeadline > 0)
    ? Math.ceil(quantityRemaining / daysUntilDeadline)
    : null;

  // 完了予測日の計算
  const estimatedDays = avgDailyPace > 0
    ? Math.ceil(quantityRemaining / avgDailyPace)
    : null;
  const estimatedCompletion = estimatedDays
    ? new Date(today.getTime() + estimatedDays * 24 * 60 * 60 * 1000)
        .toISOString().split('T')[0]
    : '予測不可';

  // 状態判定
  let status: 'on_track' | 'slightly_behind' | 'at_risk' = 'on_track';
  if (requiredDailyPace && avgDailyPace > 0) {
    if (avgDailyPace < requiredDailyPace * 0.7) {
      status = 'at_risk';
    } else if (avgDailyPace < requiredDailyPace) {
      status = 'slightly_behind';
    }
  }

  return {
    quantity_completed: quantityCompleted,
    quantity_remaining: quantityRemaining,
    progress_percent: progressPercent,
    working_days: workingDays,
    avg_daily_pace: avgDailyPace,
    days_until_deadline: daysUntilDeadline ?? 0,
    required_daily_pace: requiredDailyPace ?? 0,
    estimated_completion: estimatedCompletion,
    status,
  };
}
```

### 6.5 納期アラート判定ロジック

毎日18:00にCronで実行する定期処理：

```typescript
// /api/cron/deadline-check.ts

interface AlertResult {
  order_id: string;
  alert_type: 'deadline_approaching' | 'pace_behind' | 'critical';
  message: string;
}

async function checkDeadlines(): Promise<AlertResult[]> {
  const alerts: AlertResult[] = [];

  // 1. 進行中の全案件の進捗サマリーを取得
  const { data: orders } = await supabase
    .from('order_progress_summary')
    .select('*')
    .eq('status', 'active')
    .not('deadline', 'is', null);

  if (!orders) return alerts;

  for (const order of orders) {
    const daysLeft = order.days_until_deadline;
    const progressPct = order.progress_percent ?? 0;
    const avgPace = order.avg_daily_pace ?? 0;
    const remaining = order.quantity_remaining ?? 0;
    const requiredPace = daysLeft > 0 ? Math.ceil(remaining / daysLeft) : Infinity;

    // ルール1: 納期3日前 + 進捗80%未満 → 警告
    if (daysLeft <= 3 && daysLeft > 0 && progressPct < 80) {
      alerts.push({
        order_id: order.order_id,
        alert_type: 'critical',
        message: `⚠️ 【緊急】${order.title}（${order.client_company}）\n` +
          `納期まで${daysLeft}日ですが、進捗は${progressPct}%です。\n` +
          `残り${remaining}${order.quantity_unit}、` +
          `必要ペース: ${requiredPace}${order.quantity_unit}/日`,
      });
    }
    // ルール2: 納期3日前 + 進捗80%以上 → リマインド
    else if (daysLeft <= 3 && daysLeft > 0 && progressPct >= 80) {
      alerts.push({
        order_id: order.order_id,
        alert_type: 'deadline_approaching',
        message: `📋 ${order.title}（${order.client_company}）\n` +
          `納期まで${daysLeft}日、進捗${progressPct}%。順調です。`,
      });
    }
    // ルール3: 平均ペースが必要ペースの70%未満 → 遅延アラート
    else if (daysLeft > 3 && avgPace > 0 && avgPace < requiredPace * 0.7) {
      alerts.push({
        order_id: order.order_id,
        alert_type: 'pace_behind',
        message: `⚠️ ${order.title}（${order.client_company}）\n` +
          `ペースが遅れています。\n` +
          `平均: ${avgPace}${order.quantity_unit}/日 → ` +
          `必要: ${requiredPace}${order.quantity_unit}/日\n` +
          `このままだと納期に間に合いません。`,
      });
    }
    // ルール4: 納期を過ぎている + 未完了
    else if (daysLeft < 0) {
      alerts.push({
        order_id: order.order_id,
        alert_type: 'critical',
        message: `🚨 【納期超過】${order.title}（${order.client_company}）\n` +
          `納期を${Math.abs(daysLeft)}日過ぎています。\n` +
          `進捗: ${progressPct}%（残り${remaining}${order.quantity_unit}）`,
      });
    }
  }

  // アラートを送信
  for (const alert of alerts) {
    await sendAlertToOwner(alert);
    if (alert.alert_type === 'critical' || alert.alert_type === 'pace_behind') {
      await sendAlertToAssignedStaff(alert);
    }
  }

  return alerts;
}
```

### 6.6 案件一覧取得クエリ

```sql
-- 進行中の案件一覧（社長向け）
SELECT
    o.id,
    o.client_company,
    o.title,
    o.order_type,
    o.quantity_total,
    o.quantity_unit,
    o.total_amount,
    o.deadline,
    o.status,
    o.assigned_staff_ids,

    -- 進捗集計
    COALESCE(SUM(p.quantity_done), 0) AS quantity_completed,
    CASE
        WHEN o.quantity_total > 0
        THEN ROUND(COALESCE(SUM(p.quantity_done), 0)::NUMERIC / o.quantity_total * 100, 1)
        ELSE NULL
    END AS progress_percent,

    -- 納期までの日数
    (o.deadline - CURRENT_DATE) AS days_until_deadline

FROM orders o
LEFT JOIN order_progress p ON o.id = p.order_id
WHERE o.status = 'active'
GROUP BY o.id
ORDER BY o.deadline ASC NULLS LAST;
```

### 6.7 月次案件の自動作成処理

```typescript
// /api/cron/recurring-orders.ts
// 毎月1日の朝に実行

async function createRecurringOrders() {
  const today = new Date();
  const currentMonth = today.toISOString().slice(0, 7); // "2026-04"

  // recurring_configを持つ親案件を取得
  const { data: templates } = await supabase
    .from('orders')
    .select('*')
    .eq('order_type', 'recurring')
    .is('parent_order_id', null)       // 親案件（テンプレート）のみ
    .eq('status', 'active');

  if (!templates) return;

  for (const template of templates) {
    // 今月分が既に作成されていないか確認
    const { data: existing } = await supabase
      .from('orders')
      .select('id')
      .eq('parent_order_id', template.id)
      .gte('created_at', `${currentMonth}-01`)
      .lt('created_at', `${currentMonth}-32`);

    if (existing && existing.length > 0) continue;

    // 今月分の案件を作成
    const lastDay = new Date(today.getFullYear(), today.getMonth() + 1, 0);

    await supabase.from('orders').insert({
      client_company: template.client_company,
      client_company_id: template.client_company_id,
      title: `${template.title}（${today.getMonth() + 1}月分）`,
      description: template.description,
      order_type: 'recurring',
      quantity_total: template.quantity_total,
      quantity_unit: template.quantity_unit,
      unit_price: template.unit_price,
      total_amount: template.total_amount,
      deadline: lastDay.toISOString().split('T')[0],
      status: 'active',
      assigned_staff_ids: template.assigned_staff_ids,
      created_by: template.created_by,
      parent_order_id: template.id,
    });

    // 社長に通知
    await notifyOwner(
      `📋 月次案件が自動作成されました\n` +
      `【${template.title}】${template.client_company}\n` +
      `${today.getMonth() + 1}月分 / 金額: ${template.total_amount?.toLocaleString()}円`
    );
  }
}
```
