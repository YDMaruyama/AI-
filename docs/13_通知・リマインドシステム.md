## 1. 概要

### 1.1 システムの位置づけ

通知・リマインドシステムは、AI秘書システム全体の「神経系」に相当する横断的な基盤機能である。日報管理、タスク管理、出欠管理、支援計画、事故報告、行政書類管理、受注管理、シフト管理など、すべての機能モジュールから発生する通知イベントを一元的に受け取り、適切なタイミング・適切な相手・適切なチャネルで配信する統合通知ハブとして機能する。

### 1.2 通知の種類

本システムが扱う通知は、配信タイミングにより以下の4種類に分類される。

| 種類 | 説明 | 配信タイミング | 例 |
|------|------|---------------|-----|
| **即時通知（Immediate）** | 発生した瞬間に送信すべき緊急性の高い通知 | イベント発生から30秒以内 | 事故報告、重大アラート、連続欠席通知 |
| **定時通知（Scheduled）** | 決まった時刻に送信するバッチ通知 | 毎日8:30/17:00/18:00等 | 朝のタスク一覧、日報リマインド、日次サマリー |
| **リマインド（Reminder）** | 期限や予定に先立って送信する事前通知 | 期限の1日前/当日等 | タスク期限リマインド、支援計画更新リマインド |
| **アラート（Alert）** | 異常検知や閾値超過時に送信する警告通知 | 条件合致時に即時 | 期限超過タスク、出席率低下、LINEブロック検知 |

### 1.3 通知先（ロール別受信者）

| ロール | 説明 | 受信する通知の範囲 |
|--------|------|-------------------|
| **社長（owner）** | 事業所の最終責任者 | 全体サマリー、重大アラート、承認系通知、月次レポート |
| **管理者（admin）** | サービス管理責任者 | 担当範囲の通知、支援計画関連、出欠異常 |
| **スタッフ（staff）** | 現場支援員 | 自分のタスク、日報リマインド、担当利用者関連のみ |

### 1.4 通知チャネル

現時点での通知チャネルは **LINE Push Message（Messaging API）** のみとする。

- LINE Messaging API の Push Message を使用
- 1対1のメッセージとして各ユーザーのLINE IDに直接送信
- Flex Message を活用したリッチな通知フォーマット対応
- 01_システム基盤で実装済みのLINE連携基盤を利用

### 1.5 通知過多問題への対策

通知が多すぎるとユーザーが無視するようになり、本当に重要な通知を見逃すリスクが生じる。本システムでは以下の対策を講じる。

| 対策 | 詳細 |
|------|------|
| **優先度による分類** | critical/high/medium/low の4段階で通知を分類し、critical/highのみ即時送信 |
| **バッチ集約** | medium/lowの通知は朝・夕の定時配信にまとめて1メッセージで送信 |
| **通知設定** | ユーザーごとに通知タイプ別のON/OFF・配信モード（即時/バッチ/無効）を設定可能 |
| **Quiet Hours** | 深夜・早朝の通知を抑制し、翌朝のバッチに回す |
| **レート制限** | 同一ユーザーへの1時間あたりの通知上限を設定（デフォルト10通） |
| **重複排除** | 同一source_type + source_id + user_idの通知が短時間に複数発生した場合、最新のみ送信 |

## 2. ユーザーストーリー

### 2.1 社長のユーザーストーリー

#### US-NOTIF-001: 重要通知の即時受信

```
ユーザー: 社長
目的:     事故報告や重大な異常を即座に把握したい

受け入れ条件:
- 事故・ヒヤリハット報告が登録された時点で、30秒以内にLINE通知が届く
- 通知には事故の概要（日時、場所、関係者、概要）が含まれる
- 行政報告が必要な場合、その旨が明記される
- 利用者の連続欠席3日目が検知された場合も即時通知が届く
- 新規ユーザー登録リクエストが来た場合も即時通知が届く
```

#### US-NOTIF-002: 日次サマリーの受信

```
ユーザー: 社長
目的:     1日の事業所の状況を夕方にまとめて把握したい

受け入れ条件:
- 毎日18:00に日次サマリーが届く
- サマリーには以下が含まれる：
  - 日報の提出状況（提出済/未提出の人数と名前）
  - タスクの完了状況（完了数/残数、期限超過の有無）
  - 本日の利用者出席状況（出席数/予定数、欠席者名）
  - 新規の問い合わせや申請の有無
  - 特記事項（事故報告、異常検知等）
- サマリーの内容はFlexMessageで見やすく整形されている
```

#### US-NOTIF-003: 通知頻度の調整

```
ユーザー: 社長
目的:     通知が多すぎる場合に頻度を調整したい

受け入れ条件:
- LINEで「通知設定」と送ると、現在の通知設定一覧が表示される
- 通知タイプごとにON/OFF、配信モード（即時/まとめ/無効）を切り替えられる
- Quiet Hoursの開始・終了時刻を設定できる
- 設定変更は即座に反映される
```

#### US-NOTIF-004: 朝のブリーフィング受信

```
ユーザー: 社長
目的:     出勤時に今日の概要を把握したい

受け入れ条件:
- 毎日8:30に朝のブリーフィングが届く
- 以下の情報が含まれる：
  - 今日期限のタスク一覧
  - 期限超過中のタスク
  - 今日の利用者予定人数
  - 今日のスタッフシフト
  - 今週の重要な行政書類期限（該当がある場合）
```

### 2.2 管理者のユーザーストーリー

#### US-NOTIF-010: 担当範囲の通知のみ受信

```
ユーザー: 管理者
目的:     自分の担当に関する通知のみ受け取りたい

受け入れ条件:
- 自分が担当する利用者に関する出欠異常のみ通知される
- 自分が担当する支援計画の期限リマインドのみ通知される
- 自分にアサインされたタスクの通知のみ届く
- 他の管理者の担当範囲の通知は届かない
```

#### US-NOTIF-011: 朝のタスクリマインド

```
ユーザー: 管理者
目的:     朝の出勤時に今日のタスクを確認したい

受け入れ条件:
- 毎日8:30に今日のタスク一覧が届く
- タスクは優先度順にソートされている
- 期限超過タスクがある場合は赤字（またはアイコン）で強調される
- 今日の担当利用者の予定が含まれる
```

#### US-NOTIF-012: 支援計画期限の事前通知

```
ユーザー: 管理者
目的:     支援計画の更新期限を忘れないようにしたい

受け入れ条件:
- 支援計画の更新期限7日前に「1週間前リマインド」が届く
- 更新期限当日に「本日期限リマインド」が届く
- 期限超過した場合は翌日から毎日「期限超過アラート」が届く（社長にも同時通知）
```

### 2.3 スタッフのユーザーストーリー

#### US-NOTIF-020: 日報リマインド

```
ユーザー: スタッフ
目的:     日報の提出を忘れないようにしたい

受け入れ条件:
- 毎日17:00に日報未提出の場合のみリマインドが届く
- 既に提出済みの場合はリマインドが届かない
- リマインドから日報提出フローに直接遷移できる（「日報を書く」のクイックリプライ付き）
```

#### US-NOTIF-021: タスクリマインド

```
ユーザー: スタッフ
目的:     自分に割り当てられたタスクの期限を把握したい

受け入れ条件:
- 毎日8:30に今日期限のタスクが届く
- 期限超過タスクがある場合は別途アラートが届く
- 新しいタスクがアサインされた場合は即時通知が届く
```

### 3.6 異常系：通知送信に失敗した場合

```
[システム] LINE Push Message送信 → HTTP 500エラー
[システム] notifications.retry_count = 1 に更新
[システム] notifications.status = 'queued' のまま
[システム] 5分後のリトライスケジュールを設定

--- 5分後（1回目リトライ）---
[システム] LINE Push Message送信 → HTTP 500エラー
[システム] notifications.retry_count = 2 に更新
[システム] 30分後のリトライスケジュールを設定

--- 30分後（2回目リトライ）---
[システム] LINE Push Message送信 → HTTP 500エラー
[システム] notifications.retry_count = 3 に更新
[システム] 2時間後のリトライスケジュールを設定

--- 2時間後（3回目リトライ = 最終）---
[システム] LINE Push Message送信 → HTTP 500エラー
[システム] notifications.retry_count = 4 に更新
[システム] notifications.status = 'failed' に更新
[システム] notification_error_logにエラー詳細を記録
[システム] 管理者（システム運用担当）にアラートメール送信

※ failed になった通知は管理画面から手動再送可能
```

### 3.7 異常系：大量の通知が同時発生した場合


## 4. 機能要件

### 4.1 必須機能（MVP）

#### FR-NOTIF-001: 通知キュー管理

| 項目 | 内容 |
|------|------|
| 概要 | すべての通知はまずキューに投入され、配信タイミングに応じて処理される |
| 詳細 | 各機能モジュールから通知リクエストを受け取り、notificationsテーブルにstatus='queued'で挿入する。即時通知の場合はscheduled_at=NULLとし、送信ワーカーが即座にピックアップする。定時通知の場合はscheduled_atに送信予定時刻を設定する |
| 受け入れ条件 | 通知が漏れなくキューに投入されること。キューの状態遷移（queued→sent/failed）が正しく管理されること |

#### FR-NOTIF-002: 優先度制御

| 項目 | 内容 |
|------|------|
| 概要 | 通知をcritical/high/medium/lowの4段階で分類し、優先度に応じた配信制御を行う |
| 詳細 | **critical**: 即時送信。Quiet Hours中でも送信する。事故報告の行政報告要など。**high**: 即時送信。ただしQuiet Hours中は翌朝の最初のバッチで送信。期限超過タスク、出欠異常など。**medium**: バッチ送信。朝8:30/夕17:00/夕18:00の定時配信にまとめる。日報提出通知、タスク完了通知など。**low**: バッチ送信。省略可能。ユーザーがOFFにしやすい通知 |
| 受け入れ条件 | 各優先度の通知が規定のタイミングで正しく配信されること |

#### FR-NOTIF-003: 定時バッチ送信

| 項目 | 内容 |
|------|------|
| 概要 | Cronジョブにより定時にバッチ処理を実行し、キュー内の通知をユーザーごとに集約して送信する |
| 詳細 | 以下のCronスケジュールで実行する（詳細は7章参照） |

| Cronスケジュール | 処理内容 |
|-----------------|---------|
| `30 8 * * *` | 朝のブリーフィング/タスクリマインド送信 |
| `0 17 * * *` | 日報未提出リマインド送信 |
| `0 18 * * *` | 日次サマリー生成・送信 |
| `0 9 * * 1` | 週次通知（シフト、支援計画期限、先週サマリー） |
| `0 9 1 * *` | 月次通知（行政書類期限、月次レポート） |

#### FR-NOTIF-004: 即時送信

| 項目 | 内容 |
|------|------|
| 概要 | priority=critical/highかつscheduled_at=NULLの通知を即座にLINE Push Messageで送信する |
| 詳細 | 送信ワーカーがnotificationsテーブルをポーリング（5秒間隔）またはイベントトリガーで検知し、30秒以内に送信を完了する。送信後はstatus='sent'、sent_at=NOW()に更新する |
| 受け入れ条件 | 通知作成から30秒以内にLINE Push Messageが送信されること |

#### FR-NOTIF-005: 送信履歴管理

| 項目 | 内容 |
|------|------|
| 概要 | すべての通知の送信結果を記録し、監査・デバッグに使用可能にする |
| 詳細 | notificationsテーブルにstatus, sent_at, retry_countを記録する。失敗した通知はerror_messageカラム（または別テーブル）にエラー詳細を保存する |
| 受け入れ条件 | 過去30日分の送信履歴が参照可能であること。送信失敗の原因が特定可能であること |

#### FR-NOTIF-006: 通知設定管理

| 項目 | 内容 |
|------|------|
| 概要 | ユーザーごとに通知タイプ別の配信設定を管理する |
| 詳細 | notification_settingsテーブルで通知タイプごとのON/OFF、配信モード（immediate/batched/disabled）、Quiet Hoursを管理する。LINEの会話フローから設定変更が可能。critical通知はOFFにできない |
| 受け入れ条件 | ユーザーが通知設定を変更でき、変更が次回の通知から反映されること |

### 5.2 テーブル定義

#### notifications テーブル

```sql
CREATE TABLE notifications (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    -- 送信先ユーザー
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,

    -- 通知タイプ（notification_type_masterの値）
    type VARCHAR(50) NOT NULL,

    -- 優先度: critical, high, medium, low
    priority VARCHAR(10) NOT NULL DEFAULT 'medium'
        CHECK (priority IN ('critical', 'high', 'medium', 'low')),

    -- 通知の内容
    title VARCHAR(200) NOT NULL,
    body TEXT NOT NULL,

    -- 通知の発生元（どの機能のどのレコードから発生したか）
    -- source_type: daily_report, task, incident, admin_document,
    --             attendance, support_plan, shift, order, user_management
    source_type VARCHAR(50) NOT NULL,
    source_id UUID,  -- 発生元のレコードID（該当なしの場合はNULL）

    -- 送信状態
    -- queued: キュー待ち
    -- sent: 送信完了
    -- failed: 送信失敗（リトライ上限到達）
    -- read: 既読（将来実装）
    -- cancelled: キャンセル（集約により不要になった場合等）
    status VARCHAR(20) NOT NULL DEFAULT 'queued'
        CHECK (status IN ('queued', 'sent', 'failed', 'read', 'cancelled')),

    -- 送信予定時刻（NULLの場合は即時送信）
    scheduled_at TIMESTAMP WITH TIME ZONE,

    -- 実際の送信時刻
    sent_at TIMESTAMP WITH TIME ZONE,

    -- 既読時刻（将来実装）
    read_at TIMESTAMP WITH TIME ZONE,

    -- リトライ回数（最大3回）
    retry_count INTEGER NOT NULL DEFAULT 0,

    -- 最後のエラーメッセージ（送信失敗時）
    error_message TEXT,

    -- メタデータ（追加情報をJSONで保持）
    -- 例: {"flex_message": true, "template_id": "daily_summary"}
    metadata JSONB DEFAULT '{}',

    -- 集約グループID（バッチ集約時に同一グループの通知を束ねる）
    batch_group_id UUID,

    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
);

-- インデックス
CREATE INDEX idx_notifications_user_id ON notifications(user_id);
CREATE INDEX idx_notifications_status ON notifications(status);
CREATE INDEX idx_notifications_scheduled_at ON notifications(scheduled_at)
    WHERE status = 'queued';
CREATE INDEX idx_notifications_type ON notifications(type);
CREATE INDEX idx_notifications_source ON notifications(source_type, source_id);
CREATE INDEX idx_notifications_created_at ON notifications(created_at);
CREATE INDEX idx_notifications_priority_status
    ON notifications(priority, status)
    WHERE status = 'queued';
CREATE INDEX idx_notifications_batch_group
    ON notifications(batch_group_id)
    WHERE batch_group_id IS NOT NULL;

-- 更新時刻の自動更新トリガー
CREATE OR REPLACE FUNCTION update_notifications_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_notifications_updated_at
    BEFORE UPDATE ON notifications
    FOR EACH ROW
    EXECUTE FUNCTION update_notifications_updated_at();
```

#### notification_settings テーブル

```sql
CREATE TABLE notification_settings (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    -- 対象ユーザー
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,

    -- 通知タイプ（notification_type_masterの値）
    notification_type VARCHAR(50) NOT NULL,

    -- 有効/無効
    is_enabled BOOLEAN NOT NULL DEFAULT TRUE,

    -- 配信モード
    -- immediate: 発生次第即時送信
    -- batched: 定時バッチにまとめて送信
    -- disabled: 送信しない（is_enabled=falseと同義だが明示的）
    delivery_mode VARCHAR(20) NOT NULL DEFAULT 'immediate'
        CHECK (delivery_mode IN ('immediate', 'batched', 'disabled')),

    -- Quiet Hours（この時間帯は即時通知を抑制し、バッチに回す）
    -- ※ criticalは除く
    quiet_hours_start TIME,  -- 例: '22:00:00'
    quiet_hours_end TIME,    -- 例: '07:00:00'

    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),

    -- ユーザー×通知タイプの組み合わせは一意
    UNIQUE(user_id, notification_type)
);

-- インデックス
CREATE INDEX idx_notification_settings_user_id
    ON notification_settings(user_id);
CREATE INDEX idx_notification_settings_type
    ON notification_settings(notification_type);

-- 更新時刻の自動更新トリガー
CREATE TRIGGER trigger_notification_settings_updated_at
    BEFORE UPDATE ON notification_settings
    FOR EACH ROW
    EXECUTE FUNCTION update_notifications_updated_at();
```

```sql
    failed_count INTEGER NOT NULL DEFAULT 0,

    -- エラーメッセージ（バッチ全体の失敗時）
    error_message TEXT,
```

```typescript
// Edge Function: /api/notifications/create
// 他の機能モジュールから呼び出される内部API

interface CreateNotificationRequest {
  userId: string;        // 送信先ユーザーID
  type: NotificationType;
  title: string;
  body: string;
  sourceType: string;    // 発生元テーブル名
  sourceId?: string;     // 発生元レコードID
  priority?: NotificationPriority;  // 省略時はタイプのデフォルト値
  scheduledAt?: string;  // ISO8601形式、省略時は即時
  metadata?: Record<string, unknown>;
}

interface CreateNotificationResponse {
  notificationId: string;
  status: 'queued' | 'cancelled';  // 設定でdisabledの場合cancelled
  scheduledAt: string | null;
}

async function createNotification(
  req: CreateNotificationRequest
): Promise<CreateNotificationResponse> {

  // 1. 通知タイプのデフォルト設定を取得
  const typeDefault = NOTIFICATION_TYPE_DEFAULTS[req.type];
  const priority = req.priority || typeDefault.priority;

  // 2. ユーザーの通知設定を取得
  const settings = await getNotificationSettings(req.userId, req.type);

  // 3. 無効設定チェック
  if (settings && !settings.is_enabled) {
    // critical通知はOFFにできない
    if (priority !== 'critical') {
      const notification = await insertNotification({
        ...req,
        priority,
        status: 'cancelled',
      });
      return {
        notificationId: notification.id,
        status: 'cancelled',
        scheduledAt: null,
      };
    }
  }

  // 4. 配信タイミングの決定
  let scheduledAt = req.scheduledAt || null;

  if (!scheduledAt && settings?.delivery_mode === 'batched') {
    scheduledAt = getNextBatchTime(); // 次回バッチ時刻
  }

  // 5. Quiet Hours判定
  if (!scheduledAt && priority !== 'critical') {
    const isQuietHours = checkQuietHours(settings);
    if (isQuietHours) {
      scheduledAt = getNextMorningTime(settings.quiet_hours_end);
    }
  }

  // 6. レート制限チェック
  if (!scheduledAt) {
    const recentCount = await getRecentNotificationCount(req.userId, 60); // 過去60分
    if (recentCount >= 10 && priority !== 'critical') {
      scheduledAt = getOneHourLater();
    }
  }

  // 7. 重複チェック
  const isDuplicate = await checkDuplicate(req.userId, req.type, req.sourceType, req.sourceId);
  if (isDuplicate && priority !== 'critical') {
    // 既存の通知を最新の内容で更新
    await updateExistingNotification(isDuplicate.id, { title: req.title, body: req.body });
    return {
      notificationId: isDuplicate.id,
      status: 'queued',
      scheduledAt: isDuplicate.scheduled_at,
    };
  }

  // 8. 通知レコード作成
  const notification = await insertNotification({
    userId: req.userId,
    type: req.type,
    priority,
    title: req.title,
    body: req.body,
    sourceType: req.sourceType,
    sourceId: req.sourceId,
    status: 'queued',
    scheduledAt,
    metadata: req.metadata || {},
  });

  // 9. 即時送信が必要な場合はワーカーに通知
  if (!scheduledAt) {
    await triggerImmediateSend(notification.id);
  }

  return {
    notificationId: notification.id,
    status: 'queued',
    scheduledAt,
  };
}
```

```typescript
// 即時送信対象の通知をピックアップして送信する

async function sendImmediateNotifications(): Promise<void> {
  // 1. 即時送信対象を取得（scheduled_at IS NULL AND status = 'queued'）
  const notifications = await supabase
    .from('notifications')
    .select('*, users!inner(line_user_id, display_name)')
    .eq('status', 'queued')
    .is('scheduled_at', null)
    .order('priority', { ascending: true }) // critical → high → medium → low
    .order('created_at', { ascending: true })
    .limit(50); // 1回あたり最大50件処理

  for (const notification of notifications.data) {
    try {
      // 2. LINE Push Message送信
      const lineUserId = notification.users.line_user_id;

      if (!lineUserId) {
        throw new Error('LINE user ID not found');
      }

      const message = buildLineMessage(notification);
      await sendLinePushMessage(lineUserId, message);

      // 3. 送信成功 → status更新
      await supabase
        .from('notifications')
        .update({
          status: 'sent',
          sent_at: new Date().toISOString(),
        })
        .eq('id', notification.id);

    } catch (error) {
      // 4. 送信失敗 → リトライスケジュール
      await handleSendFailure(notification, error);
    }
  }
}

// LINE Push Message送信
async function sendLinePushMessage(
  lineUserId: string,
  messages: LineMessage[]

    method: 'POST',

    }),
  });

    throw new Error(`LINE API error: ${response.status} ${errorBody}`);
  }
}

// メッセージ構築
function buildLineMessage(notification: Notification): LineMessage[] {
  // 文字数チェック（LINE Push Messageの上限: 5000文字/メッセージ）
  const body = notification.body;

  if (body.length <= 5000) {
    return [{
      type: 'text',
      text: `${getPriorityEmoji(notification.priority)} ${notification.title}\n\n${body}`,
    }];
  }

  // 5000文字超の場合は分割
  return splitMessage(notification.title, body, 5000);
}

function getPriorityEmoji(priority: string): string {
  switch (priority) {
    case 'critical': return '🚨';
    case 'high': return '⚠️';
    case 'medium': return '📋';
    case 'low': return '📌';
    default: return '📋';
  }
}

// メッセージ分割（5000文字制限対応）
function splitMessage(title: string, body: string, maxLength: number): LineMessage[] {
  const messages: LineMessage[] = [];
  const prefix = `${title}\n\n`;
  const effectiveMax = maxLength - prefix.length - 20; // マージン

  let remaining = body;
  let partNumber = 1;
  const totalParts = Math.ceil(body.length / effectiveMax);

  while (remaining.length > 0) {
    const chunk = remaining.substring(0, effectiveMax);
    remaining = remaining.substring(effectiveMax);

    const partLabel = totalParts > 1 ? `(${partNumber}/${totalParts})\n` : '';
    messages.push({
      type: 'text',
      text: `${partNumber === 1 ? prefix : ''}${partLabel}${chunk}`,
    });
    partNumber++;
  }

  // LINEは1回のPushで最大5メッセージまで
  if (messages.length > 5) {
    // 5メッセージに収まるよう再分割（実運用上ほぼ発生しない）
    return messages.slice(0, 5);
  }

  return messages;
}
```

### 6.5 リトライ処理

```typescript
// Edge Function: /api/notifications/retry-processor
// 送信失敗した通知のリトライ処理

const RETRY_INTERVALS = [
  5 * 60 * 1000,       // 1回目: 5分後
  30 * 60 * 1000,      // 2回目: 30分後
  2 * 60 * 60 * 1000,  // 3回目: 2時間後
];
const MAX_RETRY_COUNT = 3;

async function handleSendFailure(
  notification: Notification,
  error: Error
): Promise<void> {
  const newRetryCount = notification.retry_count + 1;

  if (newRetryCount > MAX_RETRY_COUNT) {
    // リトライ上限到達 → failed
    await supabase
      .from('notifications')
      .update({
        status: 'failed',
        retry_count: newRetryCount,
        error_message: error.message,
      })
      .eq('id', notification.id);

    // 送信失敗をログに記録
    console.error(`Notification ${notification.id} permanently failed:`, error.message);

    // システム管理者に通知（別経路で）
    await alertSystemAdmin({
      type: 'notification_send_failure',
      notificationId: notification.id,
      userId: notification.user_id,
      error: error.message,
      retryCount: newRetryCount,
    });

    return;
  }

  // リトライスケジュール設定
  const retryDelay = RETRY_INTERVALS[newRetryCount - 1];
  const retryAt = new Date(Date.now() + retryDelay);

  await supabase
    .from('notifications')
    .update({
      retry_count: newRetryCount,
      scheduled_at: retryAt.toISOString(),
      error_message: error.message,
    })
    .eq('id', notification.id);

  console.warn(
    `Notification ${notification.id} retry ${newRetryCount}/${MAX_RETRY_COUNT} ` +
    `scheduled at ${retryAt.toISOString()}`
  );
}

// リトライ対象の通知を処理するCronジョブ（1分間隔）
async function processRetryQueue(): Promise<void> {
  const now = new Date().toISOString();

  const retryTargets = await supabase
    .from('notifications')
    .select('*, users!inner(line_user_id, display_name)')
    .eq('status', 'queued')
    .not('scheduled_at', 'is', null)
    .lte('scheduled_at', now)
    .gt('retry_count', 0)
    .lte('retry_count', MAX_RETRY_COUNT)
    .order('priority', { ascending: true })
    .limit(20);

  for (const notification of retryTargets.data || []) {
    try {
      const message = buildLineMessage(notification);
      await sendLinePushMessage(notification.users.line_user_id, message);

      await supabase
        .from('notifications')
        .update({
          status: 'sent',
          sent_at: now,
          scheduled_at: null,
        })
        .eq('id', notification.id);

    } catch (error) {
      await handleSendFailure(notification, error);
    }
  }
}
```

```typescript
// レート制限: 同一ユーザーへの1時間あたり最大通知数
const RATE_LIMIT_PER_HOUR = 10;
```

### 8.3 深夜・早朝の通知抑制（Quiet Hours）

```typescript
// Quiet Hours判定
function isQuietHours(settings: NotificationSettings): boolean {
  if (!settings.quiet_hours_start || !settings.quiet_hours_end) {
    return false;
  }

  const now = new Date();
  const currentTime = now.getHours() * 60 + now.getMinutes(); // 分換算
  const startTime = parseTimeToMinutes(settings.quiet_hours_start); // "22:00" → 1320
  const endTime = parseTimeToMinutes(settings.quiet_hours_end);     // "07:00" → 420

  // 日をまたぐケース（例: 22:00〜07:00）
  if (startTime > endTime) {
    return currentTime >= startTime || currentTime < endTime;
  }

  // 同日のケース（例: 13:00〜15:00 ※あまり使わない）
  return currentTime >= startTime && currentTime < endTime;
}

function parseTimeToMinutes(timeStr: string): number {
  const [hours, minutes] = timeStr.split(':').map(Number);
  return hours * 60 + minutes;
}

// Quiet Hours中の通知の扱い
async function handleQuietHoursNotification(
  notification: CreateNotificationRequest,
  settings: NotificationSettings
): Promise<string> {
  // critical通知はQuiet Hours無視
  if (notification.priority === 'critical') {
    return null; // scheduled_at = null（即時送信）
  }

  // Quiet Hours終了時刻を次回の送信時刻に設定
  const endTime = settings.quiet_hours_end; // "07:00"
  const [endHour, endMinute] = endTime.split(':').map(Number);

  const scheduledAt = new Date();
  scheduledAt.setHours(endHour, endMinute, 0, 0);

  // 既にQuiet Hours終了時刻を過ぎている場合は翌日に設定
  if (scheduledAt <= new Date()) {
    scheduledAt.setDate(scheduledAt.getDate() + 1);
  }

  return scheduledAt.toISOString();
}
```

### 8.5 通知テキストが長すぎる場合の分割

```typescript
// LINEテキストメッセージの上限: 5,000文字
// Push APIの1リクエストあたり最大メッセージ数: 5

const MAX_TEXT_LENGTH = 5000;
const MAX_MESSAGES_PER_PUSH = 5;
const MAX_TOTAL_TEXT = MAX_TEXT_LENGTH * MAX_MESSAGES_PER_PUSH; // 25,000文字

function splitLongNotification(
  title: string,
  body: string
): LineMessage[] {
  const fullText = `${title}\n\n${body}`;

  // 25,000文字以内に収まるようトランケート
  const truncatedText = fullText.length > MAX_TOTAL_TEXT
    ? fullText.substring(0, MAX_TOTAL_TEXT - 50) + '\n\n...（以下省略）'
    : fullText;

  // 5,000文字ごとに分割
  const messages: LineMessage[] = [];
  let remaining = truncatedText;
  let part = 1;

  while (remaining.length > 0) {
    // 改行位置で分割（文章の途中で切れないように）
    let splitPos = MAX_TEXT_LENGTH;
    if (remaining.length > MAX_TEXT_LENGTH) {
      const lastNewline = remaining.lastIndexOf('\n', MAX_TEXT_LENGTH);
      if (lastNewline > MAX_TEXT_LENGTH * 0.8) {
        splitPos = lastNewline;
      }
    }

    const chunk = remaining.substring(0, splitPos);
    remaining = remaining.substring(splitPos);

    messages.push({
      type: 'text',
      text: chunk,
    });

    part++;
    if (messages.length >= MAX_MESSAGES_PER_PUSH) break;
  }

  return messages;
}
```

### 10.4 AIによる通知重要度の動的調整

```
優先度: 低（実験的）
概要:   通知の内容をAIが分析し、重要度を動的に判定する
実装案:
  - Claude API を利用して通知本文を分析
  - 以下の要素から重要度スコアを算出:
    - キーワード分析（「緊急」「至急」「事故」等）
    - 過去の類似通知に対するユーザーの反応パターン
    - 時間的な緊急性（期限までの残り時間）
    - 影響範囲（関係する利用者数、金額等）
  - デフォルト優先度をAI判定結果で上書き可能にする
  - ただし、critical通知の降格は不可
```

### 10.5 ユーザーの反応パターンに基づく通知最適化

```
優先度: 低（実験的）
概要:   ユーザーの通知への反応パターンを学習し、最適な配信タイミング・頻度を自動調整
実装案:
  - 以下のデータを収集:
    - 通知の既読時刻（送信からの経過時間）
    - 通知後のアクション実行率（日報リマインド → 日報提出）
    - 通知のスルー率（特定タイプの通知を連続で無視）
  - 分析結果に基づく最適化:
    - よく見る時間帯に配信時刻を自動調整
    - 無視されがちな通知の集約度を上げる
    - 反応が良い通知フォーマット（テキスト vs Flex Message）の自動選択
  - オプトアウト: ユーザーが「自動最適化を使わない」を選択可能
```

## 付録

### A. 用語集

| 用語 | 説明 |
|------|------|
| Push Message | LINE Messaging APIの機能。Bot側から能動的にメッセージを送信する方式 |
| Flex Message | LINE Messaging APIのリッチメッセージ形式。レイアウトをJSON（CSS Flexbox風）で定義できる |
| Quiet Hours | 通知を抑制する時間帯。この間はcritical以外の即時通知がバッチに回される |
| バッチ集約 | 複数の通知をまとめて1メッセージで送信する仕組み |
| レート制限 | 同一ユーザーへの単位時間あたりの通知送信数を制限する仕組み |
| リトライ | 送信失敗時に一定時間後に再送信を試みる仕組み |
| 通知キュー | 送信待ちの通知を一時的に保持するバッファ（notificationsテーブルのqueued状態のレコード） |

### B. 環境変数

| 変数名 | 説明 | 例 |
|--------|------|-----|
| `LINE_CHANNEL_ACCESS_TOKEN` | LINE Messaging APIのチャネルアクセストークン | `xxxxx...` |
| `LINE_MONTHLY_PUSH_LIMIT` | LINE Push Messageの月間送信上限 | `1000` |
| `NOTIFICATION_RATE_LIMIT_PER_HOUR` | 1ユーザーあたりの1時間の通知上限 | `10` |
| `NOTIFICATION_MAX_RETRY_COUNT` | 送信失敗時の最大リトライ回数 | `3` |
| `NOTIFICATION_RETRY_INTERVALS` | リトライ間隔（ミリ秒、カンマ区切り） | `300000,1800000,7200000` |
| `SYSTEM_ADMIN_USER_ID` | システムアラートの送信先ユーザーID | `uuid` |
| `TZ` | タイムゾーン | `Asia/Tokyo` |
