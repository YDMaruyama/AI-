# 14. DB設計（統合データベース設計書）

## 1. 概要

### 1.1 目的
AI秘書システム全体のデータベース設計を統合的に定義する。各機能ドキュメント（03〜13）で個別に定義されたテーブルをここで一覧化し、テーブル間のリレーション、ER図、RLS（行レベルセキュリティ）ポリシー、インデックス戦略、マイグレーション計画を記述する。

### 1.2 技術基盤
- **DBMS**: PostgreSQL 15（Supabase提供）
- **ORM/クライアント**: Supabase JavaScript SDK（@supabase/supabase-js）
- **認証**: LINE User ID ベース（Supabase Auth は使用しない。独自のusersテーブルで管理）
- **RLS**: Supabaseのサービスロールキーでバイパス（サーバーサイドのみ）

## 4. 全テーブル CREATE TABLE 文

### 4.1 共通関数・トリガー

```sql
-- updated_at を自動更新するトリガー関数
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 各テーブルにトリガーを適用するマクロ
-- 使用例: SELECT create_updated_at_trigger('users');
CREATE OR REPLACE FUNCTION create_updated_at_trigger(table_name TEXT)
RETURNS VOID AS $$
BEGIN
    EXECUTE format(
        'CREATE TRIGGER trigger_updated_at
         BEFORE UPDATE ON %I
         FOR EACH ROW
         EXECUTE FUNCTION update_updated_at()',
        table_name
    );
END;
$$ LANGUAGE plpgsql;
```

### 4.2 ユーザー管理

```sql
-- ========================================
-- users: スタッフ・管理者・社長
-- ========================================
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    line_user_id TEXT UNIQUE NOT NULL,
    line_display_name TEXT,
    display_name TEXT NOT NULL,
    role TEXT NOT NULL DEFAULT 'pending'
        CHECK (role IN ('owner', 'manager', 'staff', 'pending', 'rejected')),
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    last_message_at TIMESTAMPTZ,
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE UNIQUE INDEX idx_users_line_user_id ON users(line_user_id);
CREATE INDEX idx_users_role ON users(role);
CREATE INDEX idx_users_is_active ON users(is_active);
SELECT create_updated_at_trigger('users');
```

## 6. RLS（行レベルセキュリティ）方針

### 6.1 方針
本システムではVercel Edge FunctionsからSupabaseにアクセスする際、**Service Role Key**を使用する。そのためRLSはバイパスされるが、将来的にクライアントサイドアクセス（LINE LIFF等）を導入する場合に備え、RLSポリシーを定義しておく。

### 6.2 基本ポリシー

```sql
-- 全テーブルでRLSを有効化
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
ALTER TABLE daily_reports ENABLE ROW LEVEL SECURITY;
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;
-- ... 他テーブルも同様

-- owner は全テーブルの全行にアクセス可能
CREATE POLICY "owner_full_access" ON daily_reports
    FOR ALL
    USING (
        EXISTS (
            SELECT 1 FROM users
            WHERE users.id = auth.uid()
            AND users.role = 'owner'
        )
    );

-- staff は自分の日報のみアクセス可能
CREATE POLICY "staff_own_reports" ON daily_reports
    FOR ALL
    USING (user_id = auth.uid())
    WITH CHECK (user_id = auth.uid());

-- manager は全日報を閲覧可能
CREATE POLICY "manager_read_all_reports" ON daily_reports
    FOR SELECT
    USING (
        EXISTS (
            SELECT 1 FROM users
            WHERE users.id = auth.uid()
            AND users.role IN ('manager', 'owner')
        )
    );
```

## 7. インデックス戦略

### 7.1 方針
- **主キー**: 全テーブルUUID（gen_random_uuid()）
- **外部キー**: 参照先のid列に自動インデックス
- **検索頻度の高いカラム**: 明示的にインデックス作成
- **複合インデックス**: 頻出するWHERE + ORDER BY パターン
- **部分インデックス**: status等の特定値に絞ったインデックス

### 7.2 パフォーマンス考慮事項

| テーブル | 想定レコード数（1年後） | 対策 |
|---------|----------------------|------|
| conversation_messages | 〜50,000件 | user_id + created_at の複合インデックス。古いデータはアーカイブ検討 |
| attendance_records | 〜5,000件 | client_id + date の複合ユニーク制約 |
| notifications | 〜20,000件 | status = 'queued' の部分インデックス。送信済みは定期アーカイブ |
| daily_reports | 〜1,500件 | user_id + report_date の複合ユニーク制約 |
| その他 | 〜1,000件以下 | 特別な対策不要 |
